# 套账隔离统一 · 00 概念与边界

> 这是"让所有运营数据按套账隔离"项目的北极星。动任何相关模块前先读这页。
> 状态:**图纸已审 · Zihao 2026-06-08(remote-control)拍板放手按 PO-0→8 施工 · 授权 push · 独占中心文件(src/home 收尾窗口排其后)**。
> 触发:Zihao 2026-06-07 拍板"多主体做账 = Pearnly 核心形态,套账 = 所有运营数据的硬边界" = Pearnly 2.0 核心地基。
> 进度:PO-0 地基施工中(core/workspace_context.py 已落 + 单测 8/8)。

## 一句话概念

**"套账"(workspace_client · 你在为哪家公司做账 / 哪个经营主体)= 所有运营数据的唯一硬边界。租户(tenant)只负责三件事:登录用户、计费/额度、谁能进哪些套账。除此之外,没有任何业务数据该挂在"租户"层。**

这是多主体会计(multi-entity accounting)的行业标准模型(Xero Organisation / QuickBooks Company / NetSuite Subsidiary):进了 A 公司的账,只看得到 A 公司的一切;切换是显式"换公司",换完列表里不可能出现别家数据;员工权限按"能进哪几家公司"分配。

## 现状:套账隔离做了一半

| 模块 | 现隔离维度 | 套账隔离? |
|---|---|---|
| 库存 inventory_stock/transactions/warehouses | tenant + workspace_client_id | ✅ 对 |
| 收银台/收银员/小票 pos_* | tenant + workspace_client_id | ✅ 对 |
| 套账开票资料 workspace_clients / seller_profile | 本就按套账 | ✅ 对 |
| 商品 products / product_units | **只有 tenant_id** | ❌ 缺 |
| 销项发票 sales_documents / lines / 连号 / 税票 | **只有 tenant_id**(套账列只存不滤) | ❌ 缺 |
| 进项识别记录 ocr_history | tenant + 买方 client_id | ⚠️ 列已有,写入对,**读不滤** |
| 对账 bank_recon_* / vat_recon / recon_jobs | user_id + 可选买方 client_id | ❌ 缺 |

**根因**:Pearnly 最初隔离单位是"租户 + 买方客户(client_id)",顶栏切换器切的是买方;"套账(workspace_client)"是 2026-05-25 后期才加的概念,只在新模块(库存/POS)从建表起就贯彻,旧模块(商品/进项/对账/销项)只补了个归属列、读取过滤仍停在旧维度。详见排查记录 [[workspace-isolation-audit]] 与 STATE。

## ⚠️ 两个"客户"必须分清(全项目第一原则)

| 字段 | 含义 | 角色 |
|---|---|---|
| `workspace_client_id` | **套账主体**(为哪家公司做账) | **隔离硬边界** · 顶栏"套账切换器" |
| `client_id` | **发票买方 / 对方客户**(开给谁 / 谁开来的) | 套账**内部**的一个普通筛选维度 · "买方切换器" |

现状把这俩混在一个"顶栏客户切换器"里。目标:**两层**——顶层切套账(换公司,整界面换上下文),套账内再按买方/供应商筛。

## 命名定调(Zihao 2026-06-08 拍板 · 全程统一)

**三个概念严格分清,绝不混层:**

| 层 | 概念 | 职责 | 是隔离边界? |
|---|---|---|---|
| 1 | **租户 tenant** | 账号 / 计费 / 谁能进哪些套账 | 否(更粗的授权层) |
| 2 | **套账 workspace_client** | 一个**法人公司 / 经营主体**(一套账本) | **✅ 唯一硬边界** |
| 3 | **买方 client_id** | 套账**内部**的筛选维度(开给谁/谁开来) | 否(套账内一个字段) |

- **UI 显示词按业态可配,底层永远是同一个 `workspace_client`**:会计事务所 → 「公司 / 客户公司」;商户 → 「我的公司」。一个概念,显示词可配。
- **门店 = 套账【内部】的 location**(连锁:一套账多门店),**绝不做成多套账**。本期**不实现**门店维度,但数据模型**留好口子**(别把"门店"和"套账"混成一层 → 将来加门店列时不返工)。
- **集团多套账合并报表 = 留后续**,本期不做(但因每套账已是干净边界,将来合并是"向上聚合",不返工)。

## 目标数据模型

```
Tenant 租户(= 付费账号 / 事务所)
  │  只放:用户、计费/额度、订阅套餐、"谁能进哪些套账"的授权
  │
  └── Workspace 套账(workspace_client · 一家被做账的公司 / 一个经营主体)  ★运营数据唯一硬边界★
        ├── 商品 products / product_units
        ├── 库存 inventory                  ✅ 已对
        ├── 收银 POS                        ✅ 已对
        ├── 销项 sales(含连号/税票/合规)    ❌ 待回填(连号必须按主体,见下)
        ├── 进项识别 ocr_history             ❌ 待回填(读侧)
        ├── 对账 recon                      ❌ 待回填(面最大)
        ├── 开票资料 seller_profile          ✅ 已对
        └── 买方/供应商 clients(套账内部维度,不是隔离层)
```

## 一个内核覆盖两类用户(不做两套架构)

- **会计事务所**:一个租户 N 个套账,频繁切 → 多主体。
- **零售/餐厅/药房商户**:一家公司 = 租户下**恰好 1 个套账**(注册时自动建),永不切;多门店是套账**内部**的 location,不是多套账。

→ 商户 = "套账数=1"的退化情形。同一模型两类用户全覆盖。

## 关键设计决策(本项目铁律)

1. **运营数据零容忍挂租户层**:每张业务表带 `workspace_client_id NOT NULL`(回填后),每句查询按它过滤——待遇等同现在的 `tenant_id`。
2. **"当前套账"是请求上下文**,服务端解析(从 token/header)+ 校验归属,注入每个接口、贯穿每句 SQL。**运营接口缺套账 = fail closed**(报错,绝不静默退回"看全部")。杀死"前端可选传的 query 参数"反模式——识别记录串数据就是这么来的。
3. **切套账 = 整 App 换上下文**,不是某列表加筛选框。
4. **隔离在应用层强制 + 机械闸兜底**,不指望 Postgres RLS(prod 角色 BYPASSRLS,RLS 形同虚设,见 [[pos-rls-bypass-app-layer-isolation]])。推广 POS 已有的 `test_pos_inventory_sql_isolation` 范式到全部运营表。
5. **连号/税票按主体**:泰国 RD 合规——每个经营主体(VAT 登记号)跑各自的发票连号。`document_number_sequences` 现按 tenant,多主体下会让两家公司共用一条连号 = 合规错误。**必须改按 workspace_client_id 计号**(03-backfill 详述,施工时先核现状)。
6. **商品归套账不归租户**:互不相关的公司 SKU 不该互串;连锁共享商品 = 例外,走显式"模板复制/导入",不靠租户层堆。
7. **统一术语**:隔离列全用既有名 `workspace_client_id`(不新造 `workspace_id` 免更乱);文案统一"套账";`workspace_clients` 表名暂不改(改名高 churn 低收益,留术语表备注)。
8. **老数据/老用户零破坏**:每租户建一个"默认套账",孤儿运营数据全回填进去;现有用户切换前感知不到差异。

## 范围

**做**:把"商品/进项识别/对账/销项"四块运营数据从"租户级"补成"套账级";请求上下文套账化;顶栏套账切换器(与买方切换器分层);默认套账回填迁移;隔离机械闸。

**不做 / 留后续**:`workspace_clients` 表重命名;多套账合并报表(集团视图);套账级独立计费;RLS 真启用(架构上仍以应用层为准)。

## 阶段顺序(风险 低→高 · 详见 04)

地基/上下文 → 默认套账回填(纯数据) → 顶栏套账切换器 → 商品 → 进项识别读侧 → 库存批次补缺 → **对账(面最大)** → **销项(最敏感,连号合规,最后)** → 机械闸切硬门 + 逐模块跨套账 E2E。
