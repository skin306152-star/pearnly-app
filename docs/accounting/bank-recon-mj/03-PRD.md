# 银行对账 + 手工凭证 · 第三阶段:PRD(2026-06-11)

> UI 定稿 = 桌面 `Pearnly_银行对账+手工凭证_UI预览/`(01 银行对账 · 02 手工凭证,拍板后入库挂视觉照搬闸)。
> 竞品依据 = 02-竞品矩阵;复用边界 = 01-现状审计。

## 1. 目标用户与痛点

- **SME 老板/内勤会计**:账本是引擎自动生成的,但"账和银行对不对得上"没人知道;月底拿网银流水对 Excel,1 小时起步还容易漏。计提工资/折旧这类引擎记不到的账,现在没有入口,账本永远缺一块。
- **事务所会计**:替客户做账,银行对账是交付前的必做校验;手工调整分录是月结日常。没有这两件,Pearnly 的账本只能算"半成品账"。

## 2. 核心价值(一句话)

**银行对账 = 账本可信的最后一公里**(对完账=账面和银行一分不差,老板敢信、报税敢用);**手工凭证 = 账本完整性的补丁口**(引擎够不到的计提/摊销/调整有正规入口)。两者补完,做账模块才构成可交付客户的完整闭环——而且我们的第一方数据占比(销项/POS/进项全在系统里)让自动匹配率天然高于竞品(竞品匹配的是手录数据,我们匹配的是自己生成的凭证)。

## 3. 使用场景(主)

1. 月底,老板会计从网银下载 KBANK 月对账单(xlsx),拖进银行对账页 → 系统自动对上 40+ 笔 → 高置信区一键收下 → 剩 4-6 笔逐笔确认/新建 → 差额归零,完成态提示去结账。
2. 一笔 ฿32,100 入账是客户付 3 张发票 → 组合匹配勾 3 张 → 生成收款凭证核销。
3. 每月 ฿35 银行手续费:第 1 次手动确认,第 2 次系统带建议,之后自动入账(出现在"自动对上 N 笔")。
4. 月末计提工资:做账主屏 →「+ 手工凭证」→ 从模板"月末计提工资"起 → 改金额 → Alt+= 补平 → 过账,30 秒。

## 4. 主流程 / 异常流程

**银行对账主流**:登记银行账户(一次性)→ 导入对账单 → 解析落流水池 → 自动匹配评分 → 高置信区批量收/低置信区逐笔(确认匹配 | 组合匹配 | 新建交易[收入/支出/内部转账] | 排除)→ 差额=0 → 完成态。
**异常流**:文件解析失败(人话提示+不扣费+重传)/ 同文件重复导入(sha256 拦+提示)/ 流水日期落已结期间(行操作禁用+提示)/ 匹配后发现错(撤销→流水回 unmatched、凭证 unpost)/ 组合差额非零(禁确认+显示差多少)/ 候选单据已被别的流水匹配(409 人话)。
**手工凭证主流**:新建(空白或模板)→ 填头+借贷行 → 实时配平 → 存草稿(进待审)或直接过账。
**异常流**:借贷不平(过账禁用+一键补平)/ 日期落已结期间(拦)/ 已过账要改(只能红冲或复制再开)/ 无过账权限(只见存草稿)。

## 6. 数据状态机

- 银行流水行:`unmatched → matched`(关联凭证)/ `excluded`(非记账项,可还原);全程可逆。
- 凭证(复用既有):`pending_review(=草稿) → posted / auto_posted → void`;手工凭证 method='manual';银行对账生成凭证 source_type='bank_line'。
- 对账完成判定:期间内 unmatched=0 且 差额=0(余额闭环,不设"对账期间锁"实体)。

## 8. 边界情况

多套账隔离(流水/账户/模板全挂 workspace_client_id,fail-closed)· 同名同额多候选(全列出让人选,不自动猜——核对表哲学)· POS 日汇总入账(按日聚合小计作候选)· 跨期流水(导 6 月单含 5 月 31 日行:已结→禁,未结→正常)· 金额 0 行(银行冲正,允许排除)· 多银行账户(账户维度切换,互不串)· 模板被删后历史凭证不受影响。

## 10. 价值数据指标(上线后看板)

1. 自动+建议命中率 = (高置信+有建议行)/总行数,目标 ≥70%(第一方数据优势的直接证明)。
2. 对账中位耗时(导入→差额归零),目标 ≤5 分钟/50 笔。
3. 开通做账的套账中,当月完成过 ≥1 次对账的占比(渗透率)。
4. 手工凭证模板复用率 = 从模板起的凭证占手工凭证比。
5. 撤销率(质量信号:过高=自动匹配在出错)。
