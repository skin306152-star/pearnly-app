# 功能开发五阶段流程(L/S 分级 · 2026-06-11 Zihao 拍板)

> 治"聊得美好,做出来小作坊"的根:质感在细节,细节必须在开工前钉死成文档和可验收清单。
> 任何新功能先分级,再按级走流程。**五件套未经 Zihao 确认,一行业务代码不许写。**

## 分级

| 级 | 适用 | 流程 |
|---|---|---|
| **L** | 新屏/新子模块/核心域改动(对账、计费、报税、工作台级) | 全五阶段,产出五件套文档 |
| **S** | 既有屏加功能、小改动、纯视觉 | 三件套轻量版(对标拆解 1-2 家 + 价值规格 + 稿),并入既有图纸 |

判级拿不准 → 按 L。整顿期铁律(0 新功能/高敏先报方案)优先级高于本流程。

## 五阶段(L 档)

1. **现状审计**:盘相关页面/接口/表/权限码/公共组件;列可复用清单;点名重复建设和技术债风险点。**审计没完不许写码。**
2. **竞品调研**:≥5 家真实成熟产品(必含泰国本地 + 国际主流),输出功能矩阵:入口/角色/流程/状态/权限/批量/自动化/通知/审计日志/套餐限制/优劣/来源 URL。**查不到的标【未验证】,禁止推测。**
3. **产品定义(PRD)**:目标用户/痛点/核心价值/场景/主流程/异常流程/页面与状态/权限矩阵/边界/本期范围/不做清单/扩展方向/**价值数据指标**。
4. **技术方案**:状态流转/库变更/API/前端组件/权限校验/并发幂等/操作日志/通知/性能/安全/测试/迁移/回滚/对整顿计划影响。**必须优先复用现有系统,禁止复制相似逻辑。**
5. **验收标准**:逐条可验清单,必含:正常/异常/空态/载入/错误/无权限/重复点击/断网/并发/移动端/4语/性能/安全/日志/套餐额度/自动化测试。

## 施工纪律

编码按小阶段;每阶段完成后:① 列改动文件 ② 说明设计理由 ③ 跑相关测试 ④ 对照验收清单 ⑤ 派 5 个独立 agent 按角色审查(产品经理/资深架构师/UX/QA/安全),修完审查问题才进下一阶段。
最终目标不是"能跑",是:用户能理解 · 流程完整 · 异常可处理 · 权限可靠 · 数据可追溯 · 体验一致 · 可维护 · 可扩展 · 可正式交付客户。

## 放哪(文档与代码位置规则)

- 五件套文档:`docs/<域>/<feature-slug>/01~05-*.md` + `00-KICKOFF.md`(施工窗口文案)。
- 桌面预览稿:`桌面/Pearnly_<功能>_UI预览/`,拍板后稿入库挂视觉照搬闸。
- 代码:后端 `routes/<域>_*.py` + `services/<域>/`;前端 `src/home/<域前缀>-*.ts`;导航位置必须在 KICKOFF 里写死(组名+子项+顺序),不许施工窗口自行决定。

## 存量功能怎么办(体检排队制)

不推倒重做。每个存量模块排队做一次 **L 档"体检版"**(只跑阶段 1+2+5:审计现状 + 竞品矩阵 + 按验收清单打分),产出差距清单 → Zihao 按价值排期补强施工。排队顺序由 Zihao 定,默认按使用频率:进项 → 销项 → 做账/报税 → POS → 对账中心 → 知识库。一个窗口体检一个模块,不与新功能窗口混。
