## 三、PRD(本期范围)

**目标用户**:事务所主管(给记账员配权)、SME 老板(给店长/会计配权)。**痛点**:6 预设角色不够贴
(合伙人原话诉求);老板想藏成本价/工资;日志查不了导不出;席位形同虚设。
**本期做(P0→P1 顺序)**:
1. **G1 席位 enforce**:邀请数+活跃成员 ≥ seats_max → 422 `team.seat_limit` + 前端满员升级提示(文案接波3定价,先指"联系我们")。
2. **G2 安全日志完善**:类型/操作者/时间范围筛选 + 真分页 + CSV 导出(同筛选参数);保留策略沿 operation_logs。
3. **G3 自定义角色(三步向导)**:选基底(从 6 预设复制)→ 按域分组勾权限(62 码,组级全选/展开逐码)→ 命名+确认。预设角色只读不可改;自定义角色可编辑/停用;删除时有成员在用 → 拦(先转移)。
4. **G4 敏感字段遮蔽(第一刀)**:角色加两个开关「可见成本/进价」「可见工资数据」(默认预设角色按现状全开,自定义可关);本期实施成本价遮蔽(进项/库存/POS 报表查询层列过滤),工资留位(工资模块未建)。
**不做**:G5 多维作用域(等泰方 use case,UI 留扩展位)/ 多角色叠加 / G6 实时推送 / G7 不占席角色(波3)。
**指标**:自定义角色创建租户占比;日志导出使用次数;席位满→升级咨询转化;邀请→接受转化率。

## 四、技术方案(复用优先 · 新增面极小)

- **自定义角色零 resolver 改动**:resolver 本就读 roles.permissions JSONB;新增行 `roles(tenant_id=租户, key='custom:<slug>', permissions=[勾选码])` 即生效。⚠️ 唯一风险点:`_seed_roles` 每启动按 registry 刷新 JSONB——**必须确认/收紧为只刷 `tenant_id IS NULL` 的预设行**(施工第一步写守门测试钉死)。
- 端点(并入 console_team_routes):`GET/POST/PATCH/DELETE /api/team/roles`(perm `team.role.manage`,新码?——不新增,复用 `team.member.manage` 级;owner/admin);DELETE 前查 memberships 引用计数。
- **席位 enforce**:`create_invitation` 入口 count(active memberships)+count(pending invitations) ≥ plan.seats_max → 422;错误码 `team.seat_limit`;cashier/不占席类型留参数位。
- **日志**:`GET /api/team/security-events` 加 `?type=&actor=&from=&to=` + 游标分页;`GET .../export` 返 CSV(UTF-8 BOM,Excel 直开),上限 5000 行/次。
- **敏感字段遮蔽**:roles.permissions 增加两个虚拟码 `field.cost.view` / `field.payroll.view`(进 registry 62→64,预设角色按现状含 cost);读路径在 purchase/inventory/POS 报表的序列化层按码过滤列(返回 null+前端显 --)。
- 前端:/console 改双 tab(成员/角色)+ 日志筛选导出 + 向导弹窗;复用 console.js 既有 can()/api 协议;静态 console 系文件(不碰 src/home)。
- 测试:矩阵单测扩 custom 角色;真库 E2E:建角色→邀人→登录验门→改角色即时生效→删角色被拦;席位满 422;CSV 列齐;custom 角色启动重启不被 seed 覆盖。
- 回滚:全部增量端点+新列零迁移(roles 表已有 tenant_id);revert 即回。
