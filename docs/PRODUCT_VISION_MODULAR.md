# Pearnly · 北极星:模块化平台(防跑偏 · 所有窗口必读)

> Zihao 2026-06-06 拍板。产品在从"会计事务所助手"长成"可随业态伸缩的经营平台"。变大不可怕,**散**才可怕。本页钉死身份 + 模块化原则,任何窗口加功能前先对一遍,**别又做成"什么都做一点"的四不像**。

## 一句话身份(North Star)
**Pearnly = 面向泰国会计事务所与中小商户的 AI 财务自动化平台。**
- **灵魂(不可替代)= AI/OCR 自动化 + LINE 原生 + 泰语多语本地化 + 事务所"一人代多家"多账套。**
- 开票 / 商品 / 库存 / POS / 进项…… 都是**模块(features)**,不是身份。
- **不是"又一个 ERP"**。不靠"比 Odoo/FlowAccount 更全"赢——靠它们没做好的那块:智能自动化 + 泰国本地 + 代记账场景。

## 三层架构(所有功能按这个归位)
```
① 底座(永远在·共享)   主数据(商品/客户/供应商/账套)· 账号权限 · AI 录入引擎 · 多语言
② 模块(按租户开关)    开票(销项)· 进项/费用 · 库存 · POS · 应收 · 对账 · 知识库
③ 出口                发票 / 收银 / 报税 …… 都接在"选好品 + 录好账"之后
```
- **商品 = 主数据(①)**,不属销项也不属进项,谁都用,**只此一份,禁止各模块各建一套**。
- **库存 = 独立域**,被进项(入库)喂、被销项(出库)耗;**卖实物才需要,卖服务不需要**。
- 加任何新功能前先问:它是**底座**、**模块**、还是**出口**?归不进去 = 没想清楚,先别写。

## 模块化六原则
1. **底座常开 + 模块可开关**:`tenant_modules(tenant_id, module_key, enabled, config)`,导航/路由/功能按它显隐。
2. **业态套餐代替逐个拨开关**:给预设(服务型/零售/餐厅/批发/事务所)一键开好,高级再微调。**非技术用户能用起来的关键。**
3. **导航随模块显隐**:开了才显示,没开的不出现或标"可开启 →"。服务型公司不被库存/POS 吓到。
4. **模块松耦合(关了不报错)**:靠事件联动——开票开出→发"卖了X"事件→库存(若开)扣减;没开库存就没人接,**什么都不发生也不崩**。这是能任意开关的技术底线。
5. **关模块 = 隐藏不删数据**:随时可重开,数据不丢,用户敢试。
6. **计费天然按模块**:基础(开票)→ 加库存 → 加 POS;不逼人一上来买全套。(Pearnly 现按额度计费,此条以后再接。)

## 业态预设(示例 · 选一个一键开好)
| 业态 | 默认开的模块 |
|---|---|
| 服务型公司(咨询/律师/设计) | 开票 · 进项 · 客户 |
| 零售 / 小卖部 | 开票 · 商品 · 库存 · POS · 客户 |
| 餐厅 | 开票 · 商品 · 库存(原料) · POS · 客户 |
| 批发贸易(B2B 卖货) | 开票 · 商品 · 库存 · 应收 · 进项 · 客户 |
| 会计事务所(代记账) | 开票 · 进项 · 对账 · 客户 · 知识库 · 多账套 |

## 纪律(防跑偏)
- **先尖后广**:把"事务所助手 + 快速开票"这把尖刀做到别人比不了,用它切市场、证明价值,**再**模块化扩。
- **按阶段来**:Phase1 开票闭环 → Phase2 库存/LINE开单/应收 → Phase3 e-Tax → 独立项目 POS。**别一次全做。**
- **守边界**:开票页不做收银(现金/找零/扫码枪/小票机/实时库存 = POS 模块/项目)。
- 设计草稿:`桌面/Pearnly开票UI预览/onboarding-modules.html`(选业态→自动开模块 + 导航预览)。

> 一句话:**有共享底座、有 AI 灵魂、有清晰主线地往大长 = 平台;没主线乱铺摊 = 四不像。** Pearnly 走前者。
