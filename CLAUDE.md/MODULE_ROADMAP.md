# 1. 仪表盘(Dashboard)

### 业务目的
会计事务所所长 / SME 老板每天上班第一件事是问「昨天到今天,我账上有多少未处理的单?哪个客户卡住了?」目前他们靠**翻群、打开 Excel、问员工** 拼凑这个画面,平均花 15 分钟。仪表盘要把这 15 分钟压到 5 秒。

### 用户故事
- 作为**事务所所长**,我需要一屏看到「今日处理 / 未处理 / 异常 / ERP 推送失败」四个数字,这样我可以**马上知道员工今天工作量与瓶颈**
- 作为**SME 老板**,我需要看到本月各客户的处理进度卡片,这样我可以**月末提前预警哪个客户对账还没做完**
- 作为**会计**,我需要点击任何卡片直接跳到对应工作页,这样我可以**0 思考成本切到要处理的事**

### 当前实现状态
- **仅占位**:`home.html` `<section id="page-dashboard">` 内容只有 `auto-coming-hero` 大图标 + 三条 `cs-dashboard-f1/f2/f3` 文案 + 「预计 v110 上线」
- **无 JS**:`home.js` 中 `loadDashboardPage` 函数不存在
- **无后端 API**:`app.py` 没有 `/api/dashboard/*` 端点
- 已有可复用的数据源:`/api/erp/stats/today`(今日 ERP)、`/api/history`(单据)、`/api/clients`(客户列表)、**`/api/exceptions/stats`(v118.20 新)**

### 参考产品
- **Xero Dashboard**:左 KPI 数字 / 右账单到期 + 应收账款柱状图 · 用户买单是因为「打开就知道公司财务节奏」,不需要点 5 个菜单
- **FlowAccount**:首页「待办事项卡片流」(开发票 / 收款 / 报税倒计时)· 泰国本土用户买单是因为它直接对接 ภ.พ.30 / ภงด 报税进度

### 我们的差异化
Pearnly 的优势是**事务所多客户视图**:
- Xero 是「一家公司一个仪表盘」· FlowAccount 是「一家公司一个待办」
- Pearnly 必须做「**一个事务所同时看 N 个客户的横切卡片**」+ 一行「最异常的客户置顶」(异常数据已有 · 直接接)
- 复用 LINE Bot 通道:仪表盘异常数字旁直接放「推送给老板的 LINE」按钮(老板不开网页也能看到)

### 优先级 + 估时
**P1** · **5-7 天** · 先做 4 个 KPI(其中「待复核异常」直接接 v118.20 stats API)+ 客户卡片网格(复用 `clients-grid` CSS 即可)· 趋势图与 LINE 推送下一迭代

# 2. 异常栏(Exceptions)· ✅ 已上线 v118.21.3

### 业务目的
泰国会计的最大风险是**收到假票 / 漏申报 / 重复入账**。一旦混进 1 张假票,事务所要赔客户全额 + 罚款 + 信誉损失。会计现在的做法是「肉眼看每一张票」,效率低且容易漏。异常栏 = **AI + 税局 API 双重把门 + 集中复核工作台**,把异常单从「淹没在历史记录里」抠出来,让会计 1 个工作流处理完。

### 用户故事
- 作为**会计**,我需要所有「OCR 置信度 < high / 金额对不上 / 重复 / 假税号」的单据集中显示在一页,这样我可以**专注复核 5%、不浪费时间在合格的 95% 上** ✅
- 作为**事务所风控**,我需要看到「这张被拦的原因 chip + 详细差额」,这样我**1 秒就知道为什么被拦** ✅
- 作为**会计师**,我需要复核完后系统记住我的判断,这样**下次同类单据自动放行,我的精力越用越省** ✅
- 作为**月底加班的会计**,我需要勾选 50 张异常一次性放行,而不是 50 次点抽屉 ✅(v118.20.5)
- 作为**事务所风控**,我需要边对照原 PDF 边核字段,而不是新开 tab ✅(v118.20.7)
- 作为**会计**,我改完 OCR 错抽的金额后异常自动消失,不用再手动放行 ✅(v118.21.3)
- 作为**事务所所长**,我需要按客户筛选 + 看自己历史处理过哪些 + 撤销误学的规则 ✅(v118.21.0/.1/.2)

### 用户故事
- 作为**事务所会计**,我需要给每张发票打上客户标签,这样**月底每家客户单独一个 PDF 包发出去**
- 作为**会计**,我需要在客户卡片直接看到本月发票数 + VAT 总额,这样**不用每个客户单独打开 Excel 算**
- 作为**事务所所长**,我需要一键导出「某客户某月所有进项发票 + Excel 汇总表 + ภพ.30 草稿」zip 给客户,这样**客户主动续费时我能秒响应**

### 当前实现状态
**已实现 80%**:
- UI:`home.html` 完整客户网格 + 弹窗(name / short / tax / address / contact / phone / email / color picker / notes)
- JS:完整 CRUD `loadClientsCache / renderClientsGrid / 弹窗保存 / 删除`
- 后端:`/api/clients/*` 完整 CRUD + `assign_client` + `client_export`
- 数据库:`db.py:ensure_clients_table` · `ocr_history.client_id` 字段已加
- **缺**:

# 4. 识别中心(OCR)· 核心 · ⚠️ 速度待优化

### 业务目的
Pearnly 的核心引擎。会计的最大痛点是「**纸质票 + 多语言**」:泰国本土票泰文、中资客户票中文、日资客户票日文 · 三种语言的会计要么各请一个、要么手抄到死。识别中心要做到「**任意语言任意格式 PDF/图片**,30 秒出结果」。

### 用户故事
- 作为**会计**,我需要把 1 个 PDF 里的 5 张发票自动拆开,这样**不用 PS / 拆 PDF 再上传**
- 作为**外勤**,我需要在手机直接拍照上传,这样**不用回办公室再扫一遍**
- 作为**会计**,我需要看到 OCR 结果的置信度,这样**只复核置信度低的几张**
- 作为**事务所**,我需要月底一次传 500 张发票 · **5 分钟内全部识别完** · 不用挂着浏览器等

### 当前实现状态
**已实现 95% · 但速度未达事务所要求**:
- UI:拖拽 + 拍照 + 上传图片 + 多文件 + 结果表格 + 搜索 + 排序
- 后端:Gemini 主 + Vision fallback / 一页多发票拆分 / 30 天文件 hash 去重
- 配额:套餐配额 + 月统计
- 前端并发:`PARALLEL_LIMIT = 6`(v118.20.1.6 提升)
- 后端单 PDF 内:3 页并发跑 Gemini
- 整体上限:6 × 3 = 18 路 · 但 Gemini API rate limit 是真天花板
- **未做**:
  - **PDF 文本预筛**(电子发票直接抽 text · 跳 Gemini · 速度从 8s → 0.3s)
  - **异步任务模式 + 队列页**(用户传完不用挂浏览器 · 后台跑)
  - 「自定义模板」按钮 `#btn-custom-template` 是 `btn-locked`(占位未启用)
  - 没有「对识别结果的快速纠错」(目前只能进单据记录详情页改)
  - 拍照引导 `camera-tip-*` 文案有但没看到实际拍照前的引导浮层

### 速度场景算账(v118.20.3 实测)

| 张数 | 现状耗时 | 目标 | 解决方案 |

| 50 张 | ~2 分钟 | < 30 秒 | PDF 文本预筛(60% 命中) |
| 300 张 | ~13 分钟 | < 3 分钟 | 文本预筛 + 6 并发 |
| 1000 张 | ~45 分钟 | < 10 分钟 | 文本预筛 + 异步队列(关浏览器也跑) |

- **FlowAccount OCR**:仅泰文 · 用户买单是因为开票 + OCR 一站式
- **Vimcar / Datev(德国)**:支持自定义字段映射

### 我们的差异化
**多语言 + 一页多票 + 文件去重**(已建立护城河)
**+ 文本 PDF 快速通道**(下一步) · **+ 异步队列**(事务所规模化必备)

### 真正的核心 · 映射 UI(铁律 47)

#### 主数据 + 默认值 + Fuzzy Match 三层

**第一层 · 主数据缓存**(`erp_master_cache` 表):
- 用户连接 MR.ERP 后 · 后台拉取所有客户 / 商品 / 仓库 / 部门 / 销售员
- 缓存 24 小时 · 用户可手动刷新

**第二层 · 默认值配置**(连接后必弹的 modal):
- 默认仓库:[0000 ▼](从主数据下拉)
- 默认部门:[BOI1 ▼]
- 默认销售员:[กร ทดสอบ ▼]
- 默认客户类型:[现金销售 ▼ / 赊销]
- 默认商品税率:[0% ▼ / 7% / นอกระบบ]

这些是不变的字段 · 一次配置后所有推送都用。

**第三层 · 客户/商品 Fuzzy Match**(`erp_field_mapping` 表):
- OCR 推送时 · 检查 `erp_field_mapping` 是否有这个 OCR 名字的映射
- 没有 → 弹 modal · 后端 fuzzy match 给 top 3 候选 · 用户选 1
- 选完保存 · 下次自动用

## v118.30+ · D 浏览器扩展(后置 · 公测后)

公测后看真实用户反馈 · 决定何时启动 D。

### D 的产品定位

「不愿存密码 · 但要一键推送」的客户 · 装 Chrome 扩展:

```
[Pearnly OCR 完成] → [识别中心点「推 MR.ERP」]
                          ↓
                  [扩展检测到 MR.ERP 已登录]
                          ↓
              [扩展用浏览器现成 PHPSESSID 直接 fetch 推]
```

技术优势:
- ✅ 不存密码(用浏览器现成登录态)
- ✅ MR.ERP 升级时只更新扩展 · 不影响后端
- ✅ 法律最安全(用户主动操作 · 跟手动点击无区别)

技术挑战:
- 用户需安装扩展(教育成本中等)
- 不能后台自动跑(浏览器关了就停)
- Chrome Web Store 审核
