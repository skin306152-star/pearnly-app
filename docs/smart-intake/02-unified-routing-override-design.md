# 统一智能录入 · 方向分流 + 手动改方向 + 事务所客户做账 + LINE 接入(设计稿 · 待 Zihao 拍板)

> 来源:Zihao 2026-06-09 真用发现 —— LINE 拍的进项票只落「识别记录」、没有"手动选模块"的覆盖、事务所给客户做账上传客户的票该怎么分流、怎么不污染别的模块。
> 关系:承 `00-unify-and-followups.md` 的 F12-14(统一入口/待归类/识别记录门控,已上线一半 · 见 [[purchasing-followups-shipped]]),本稿**补全缺口**:① 手动改方向 UI ② LINE 图片接同一分流 ③ 事务所多客户做账分流明确化 ④ 不污染保证。
> 本稿是**设计**,不是施工单。拍板后再出 `03-construction-plan` 动手。**高敏:LINE/OCR/计费主路径(铁律#26)+ 多模块 + UI(先稿后建)。**

---

## 一、问题(Zihao 真用发现)

1. **落错地方**:LINE 拍一张吃饭费用票 → 只进「识别记录」(销项侧的 OCR 日志),没进「进项/待归类」,也没法记账。
2. **没有手动覆盖**:系统判错方向 / 拿不准时,用户只能接受判定或落待归类,**不能主动说"这张去采购"或"去销项"**。
3. **事务所给客户做账**:事务所上传的是**客户公司**的票(进项 + 销项都有),需要按"在为哪个客户做账"分流,且**不能串到别的客户**。
4. **怕污染**:想要"这张去 A 模块、不污染 B 模块"的确定性,而不是系统乱塞。

| 识别记录按业态门控(事务所专用) | 🟡 半 | 门控位已留,但商户的票仍会写进 ocr_history(日志) |
| **手动改方向(上传后确认卡可改)** | 🔴 **缺** | 本稿 §四 |
| **LINE 图片接同一智能分流** | 🔴 **缺** | LINE 图片走老路只写 ocr_history + 仅商户落待归类(且门控严:未选业态/事务所早退) |

## 三、核心模型(钉死,先对齐再设计屏)

### 3.1 两个"客户"维度(必须分清,否则全乱 · 见 [[workspace-isolation-audit]])
- **套账主体 `workspace_client_id`**:在为**哪家公司**做账 = "我是谁" = 硬隔离边界。事务所给客户 A 做账 = 切到 A 的套账。
- **买方/交易对手 `client_id`**:一张票上的对方,是票内字段,不是切换器。

### 3.2 方向 = 谁的税号匹配(已实现的决策表)
| 票面 | 判定 | 落点 |
|---|---|---|
| 卖方税号 == 当前套账主体税号 | 我开的票 → **销项** | 销售发票模块 |
| 买方税号 == 当前套账主体税号 | 我收的票 → **进项** | 采购/进项(有 VAT)或费用 |
| 两边都不匹配但确是发票 | 默认 **进项**(商户拍的多是收到的票) | 采购/进项,置信下调 |
| 银行流水单 | → **对账** | 对账中心 |
| 非发票 / 糊图 / 低置信 / 拿不准 | → **待归类** | 待归类收件箱 |

> 这张表**不变**,本稿只加"人可以改这个判定"。

### 3.3 不污染 = 套账隔离 + 单一落点
- 每张票**只落一个**业务模块的表(进项 `purchase_docs` / 销项 `documents` / 对账 recon),按 `workspace_client_id` 隔离。
- 「识别记录」是 **OCR 识别日志(留底)**,不是业务数据。它存所有识别过的票(含进项),**不等于"塞进销项业务"**。但 Zihao 觉得它叫"识别记录"且在销项组下 → 误以为污染。**设计上澄清:商户视角下识别记录降为"历史日志"(可选/折叠),业务落点永远是分流后的模块。**

## 四、统一智能录入 · 核心流程(新)

```
                ┌── Web「智能录入」口(拍照/选图/上传文件)
   一个入口 ────┼── Web 各模块的快捷"拍票"(进项/销项内,预设方向)
                └── LINE 发图片
                          │
                          ▼  OCR(复用现有管线 · 计费同网页)
                          │
                   judge_direction(用当前套账主体税号)
                          │
            ┌─────────────┼───────────────┬──────────────┐
        高置信+方向明确   低置信/拿不准    非发票/糊图     银行单
            │                │                │             │
            ▼                ▼                ▼             ▼
    ┌──「方向确认卡」──┐   待归类收件箱    坏图四态       对账中心
    │ 判定:进项票      │   (带票图+猜测)  (重拍/手填)
    │ 卖方 Kodtalay    │
    │ ฿2,722 · 你是买方 │   ← 所有入口最终都到这套分流,LINE 也是
    │ [确认入账]        │
    │ [改为销项]        │   ★ 关键新增:确认卡 = 人对判定的控制权
    │ [改为费用]        │
    │ [这是客户XX的票]  │   ← 事务所:切套账主体
    │ [不是票/待归类]   │
    └──────────────────┘
```

### 4.1 方向确认卡(★本稿核心新增)
- **何时出**:OCR 完成、方向判定后。高置信也出(让用户一眼确认),但默认选中判定结果,一键即过(不增负担)。
- **卡内容**(四态完整):
  - 判定结果:`进项票 · 卖方 Kodtalay Seafood · ฿2,722.00 · 开票日 2025-08-24`
  - "你是谁":`记账给 SINCERE(当前套账)` + `切换`
  - 主操作:`[确认 · 进采购录入]`
  - 改方向:`[改为销项]` `[改为费用]` `[这是银行单·去对账]` `[不是票·待归类]`
  - 票图缩略(点开原生看图器,复用本轮 C+D)
- **改方向后**:重新按选中方向建草稿/落点;系统**记住**该供应商/这类票的修正(复用 `review_learned` 思路),下次同类自动按修正判。
- **行为**:确认 → 进对应模块录入屏(预填);改方向 → 切到对应模块草稿。**永远不静默落错。**

### 4.2 "上传前预设方向"(可选 · 高级)
- 入口处提供轻量预设:`[智能判断(默认)]` / `[这批都是进项]` / `[这批都是销项]` / `[给客户XX做账]`。
- 预设后跳过/弱化确认卡(批量上传场景省事),但仍可逐张改。
- 满足 Zihao "想去这个模块、不污染另一个"的确定性诉求。

## 五、事务所给客户做账(关键场景 · firm 底线)

> 事务所上传的是**客户公司**的票(进项+销项混),要按"在为哪个客户做账"分流,且不串客户。

- **机制(已有)**:切到客户 A 的套账主体 → `_my_tax_id` = A 的税号 → judge_direction 按 A 判方向 → 落 A 的进项/销项,套账隔离不串 B。
- **缺的(本稿补)**:
  1. **录入口顶部明确显示并强制确认"在为哪个客户做账"**(`记账给 客户A · 切换`),避免上错客户。未选客户主体 → 拦上传 + 引导先选(对事务所是硬前提)。
  2. 确认卡的 `[这是客户XX的票]` = 快捷切套账主体(不用回右上角)。
  3. **识别记录** 对事务所**保留为主工具**(firm 底线一字不破坏);商户走分流。门控单一来源 `line_expense_gate_open` / business_type。
- **批量**:事务所一次传客户一叠票(进项+销项混)→ 逐张判方向 + 落 A 的对应模块;混入的别家票(税号不匹配 A)→ 待归类提示"这张不是客户A的票"。

## 六、LINE 图片接入(补缺 · 高敏)

- **现状**:LINE 图片 → 只写 ocr_history(识别记录)+ 仅"商户且选了业态且开 expense"才落待归类(门控严,事务所/未选业态早退)→ 多数情况只剩识别记录。
- **改为**:LINE 图片 → 同一 `judge_direction` 分流:
  - 商户:判定后落对应模块的**待归类**(带票图)+ LINE 回**方向确认卡**(LINE Flex Message:`进项票?[确认入账] [改销项] [待归类]`),用户在 LINE 一点即入账。
  - 事务所:维持 firm 底线(识别记录为主),但 LINE 回卡可让其选"给哪个客户做账"。
  - 票图:LINE 落待归类时也存图(复用本轮 C 的 `pdf_storage.save_bytes`),`intake_items.image_url` 不再传 None。
- **铁律#26**:LINE 主路径改动 = 高敏,施工时纯加法 + firm 早退一字不破坏 + 真账号 LINE 闭环复验。

## 七、屏幕清单(过 [[ui-design-completeness-gate]] 六类 · 缺一类返工)

1. **操作面**:统一智能录入卡(拍照/选图/上传 + 预设方向)· 方向确认卡(判定+改方向)· 待归类收件箱(逐张定向)。
2. **老板配置后台**:业态/模块设置(决定门控)· 录入默认行为(是否每张都弹确认卡 / 信任高置信自动入账)· 学习记忆管理(已记住的供应商→方向,可清)。
3. **登录接入流**:LINE 绑定后欢迎语说明"发票即记账"· 首次录入引导选业态/客户主体。
4. **四态边界**:OCR 加载中 / 糊图坏态(重拍·手填)/ 空(还没录入)/ 错(OCR 失败·额度不足)。
5. **角色权限**:owner 全权 · member 可录入但审批/付款受限 · 事务所 firm 专属识别记录路径。
6. **模块联动**:确认/改方向后正确落 销项↔进项↔对账↔做账;待归类→建单→做账引擎接力([[accounting-design-sealed]]);套账切换全局刷新(本轮已收口)。

## 八、后端改动点(施工预览 · 拍板后细化)

- `judge_direction`:不变(决策表稳定)。
- **新**:确认卡的"改方向"= 前端拿 OCR fields + 用户选的方向 → 调对应模块建草稿接口(进项 `/purchase/intake` 已有 · 销项需对应入口 · 对账落 recon)。后端加"按指定方向建草稿"的薄入口(不重判)。
- **新**:学习记忆 `review_learned`(供应商→方向修正),judge_direction 先查记忆再判。
- **改(高敏)**:`line_image_ocr` 接 judge_direction 全分流 + 存图 + LINE 回方向确认卡;门控放宽与否见 §九决策点。
- **改**:识别记录商户视角降为日志(折叠/次要入口),不作主要落点。
- 复用:`pdf_storage.save_bytes`(本轮 C)· `intake_items`(待归类)· 套账隔离。

## 九、待 Zihao 拍板的决策点(设计分叉)

1. **确认卡默认行为**:每张都弹确认卡(稳,稍累)vs 高置信自动入账+仅低置信弹(省事,偶误)?建议:**默认弹、可在设置里关成"高置信自动"**。
2. **LINE 门控**:未选业态的商户,LINE 进项票要不要也进待归类?(放宽 `line_expense_gate_open`)还是坚持"先选业态"?建议:**先引导选业态**(零代码先验),放宽作为后备。
3. **识别记录定位**:商户视角彻底隐藏识别记录(只在分流后模块看票)?还是保留为"全部识别日志"折叠入口?建议:**保留为折叠日志**(可追溯),业务落点走分流。
4. **统一入口 vs 各模块快捷口**:做一个全局"智能录入"大入口,还是各模块保留拍票口(共用分流)?建议:**两者都要**(全局口 + 模块内快捷口,后端同一套分流)。
