# 02 · LINE 采购进项产品正本

> 状态:产品正本 · 2026-06-18。以后讨论 LINE/LIEN 采购进项、票据识别、卡片、纠错、对话闭环,先按本页口径走。
> 本页吸收了 Paypers 逆向、`docs/smart-intake/*` 旧讨论、`docs/line-product-truth/*` 真相包和 2026-06-17/18 真机验收结论。
> 冲突时优先级:本页 > `docs/line-platform/00` > `docs/smart-intake/09/10/15/16/18` > 其它历史设计稿。

## 0. 产品一句话

LINE 采购进项不是“聊天机器人”,而是老板/会计在 LINE 里的**票据收集、识别、确认、纠错和打开复核页**入口。轻操作在聊天里完成,重核对进 LIFF/Web。机器人只说真实发生的事,不猜、不编、不越权。

## 1. 当前阶段

已上线或正在收口:
- 图片/文件进采购 OCR,返回 Flex 卡。
- 四态卡:已入账 / 待确认 / 可能重复 / 需补全。
- loading、quote/reply context、卡片 postback、撤销/确认/丢弃。
- OCR 后处理:总额优先级、VAT/rounding、佛历年、付款方式、低置信明细诚实降级。
- P1E-1:闲聊/能力问答/旧示例文案收口,LLM 不再把 `咖啡65` 之类示例吐给用户。
- P1E-2:纠错 / 澄清 / 执行动作统一收口正在施工。

第一阶段边界:
- 做透采购进项车道。
- 其它模块只做入口或提醒,不在 LINE 里直接写入高风险业务。

## 2. 旧讨论冲突清单

| 旧口径/文档想法 | 当前决策 | 理由 |
|---|---|---|
| `intake_items` / purchase inbox / 待归类收件箱作为长尾安全网 | **退役,不再作为施工方向** | 2026-06-16 已整模块下线。识别完一律建采购草稿进入采购列表,低置信在卡片/详情页诚实提示。 |
| 把 `image-first` 作为 OCR 主路径重写 | **不作为当前计划** | 后续真机与 prod 修复已经把收益落在确定性后处理、触发器、字段优先级和诚实兜底。不要再按 06-13 旧稿重写 OCR 主路径。 |
| Drive/Sheets/Excel 外流一期和 LINE 采购一起做 | **可作为未来模块,不进 P1E/P1F** | 价值存在,但涉及 OAuth、外部存储、异步任务、套账隔离,不是当前 LINE 对话闭环。 |
| 全站对话 AI 助手一次性上线 | **未来平台能力,当前只做采购车道 Brain OS** | 先把一条车道做透。全站工具调用会放大权限、账务和文案风险。 |
| Rich Menu 作为所有模块遥控器第一版 | **保留为后续交互层,不阻塞采购闭环** | 当前用户痛点在发票识别、确认、纠错、引用。菜单不是这轮主矛盾。 |
| 把采购卡 postback 能力泛化到税务、银行对账、会计凭证 | **禁止** | 这些是高风险写入动作。LINE 只做提醒/入口,真正处理进 Web/LIFF。 |
| `channel=line` 代表 Pearnly OA 主动给买方推销项发票 | **错误** | 当前销项 line 发送是分享链接,不是 OA 主动推送。 |
| 报销人/员工报销审批字段 | **不做** | Pearnly 当前定位是老板/会计自记账,不是员工报销审批系统。 |
| 把 OCR 历史当业务票据 | **错误** | 采购业务对象是 `purchase_docs`。OCR/history 是识别日志,不是业务落点。 |

## 3. 旧讨论可采纳部分

| 可采纳点 | 落到 Pearnly 的方式 |
|---|---|
| Paypers 的 quote-reply 机制 | 所有动作尽量引用用户原消息,让用户知道机器人回应哪张票/哪句话。 |
| “引用某张卡再说删除/修改”比自然猜上一笔可靠 | reply/quote purchase record 的优先级高于闲聊、OCR 失败帮助和普通文本记账。 |
| 轻操作在 LINE,重操作进 Web/LIFF | 卡片上确认/撤销/丢弃/保存核心字段;明细复杂、字段多、低置信时打开详情页。 |
| 数据卡比纯文字更有高级感 | 保持 Flex 卡字段清晰、状态诚实、按钮按状态变化。 |
| STP + HITL | 高置信低风险可直入账并可撤销;中低置信建草稿/提示复核。 |
| 默认分类树/字段集参考 Paypers | 可用于分类 seed、doc_type 展示、字段补齐,但不能照抄报销审批线。 |
| 对话要像产品,不是像模型 | LLM 只输出 intent/slots;用户可见文案走 i18n 模板。 |

## 4. Brain OS 口径

Brain OS 不是一个“大模型自由聊天层”,而是分层决策系统:

| 层 | 职责 |
|---|---|
| L0 消息类型 | image/file/text/postback/follow 等确定性分发。 |
| L1 规则 | 绑定码、删除/撤销、金额文本、常见问候、否定词、引用记录等。 |
| L2 域内模型 | 只做意图和字段抽取,输出固定 schema。 |
| L3 会话状态 | pending confirmation、最近明细列表、第 N 笔引用、纠错上下文。 |
| L4 深链 | 复杂编辑、低置信明细、网页复核、设置。 |

硬规则:
- 模型只“听懂”,不“开口”。
- LLM 不算金额、不算税、不判税期、不改锁期、不宣称申报成功。
- 用户可见回复只能来自 `line_i18n`/模板。
- 问句、否定、假设、泛泛闲聊不写账。
- 破坏性动作必须确认或来自卡片 postback。

## 5. 票据识别与卡片口径

核心字段优先级:
- 总额:优先票面 final total / net / total baht / ยอดรวมสุทธิ,禁止被 cash/payment/change 覆盖。
- 付款信息:Cash / QRPayment / PromptPay / credit card 只作为付款方式或支付状态线索,不覆盖费用总额。
- VAT/rounding:票面有明确 subtotal/VAT/rounding 时优先用票面字段;rounding 只能来自 rounding 字段。
- 日期:佛历/泰历确定性归一,不让 LLM 单独决定。
- 卖家/发票号/税号:能读则显示;简式票/非税票不强迫完整税票字段。

明细策略:
- 明细合计与总额一致:显示多行明细,允许正常确认。
- 明细不完整或合计不一致:显示已识别候选或隐藏明细,必须提示需检查,主按钮改为“เปิดเพื่อตรวจสอบ”。
- 0 元 modifier、赠品、`original`、纯标签不进主金额明细,可作为备注。
- 绝不为了凑齐明细而编金额、摊金额或让模型补行。

## 6. 纠错闭环口径

用户 reply/quote 某张采购卡后说:
- 这个识别错了 / 不对 / wrong / incorrect / ผิด / ไม่ถูก

必须进入纠错澄清,不得走“重新拍照/图片不清晰”。

流程:
1. 有引用记录:定位该 `doc_id`。
2. 有具体字段:出确认,不直接写账。
3. 无具体字段:问要改金额、日期、卖家、分类、明细还是付款方式。
4. 多行票改总额:不猜改哪一行,默认引导打开详情页。
5. 能安全改的字段:走 void 原单 + 重建草稿/确认链,保留 `corrected_from` 审计。
6. 已锁期/已申报/已结账:诚实拦截,不破账。

用户没有引用却说“这个识别错了”:
- 提示请回复要修改的那张记录,或输入“查明细”后选择第几笔。

## 7. 回复文案口径

泰语为主,四语完整。文案分三类:
- 状态说明:短、确定、只说真实状态。
- 操作引导:详细明确,温和,告诉用户下一步怎么做。
- 错误/限制:诚实解释原因,给可执行出路。

所有回复尽量带引用:
- 收到图片、识别中、识别失败、确认卡、已保存、已撤销、已丢弃、找不到引用、纠错澄清、闲聊/能力问答。

如果 LINE API 某类消息无法 quote,可降级普通回复,但主路径必须保留 quote context。

## 8. 不做清单

当前采购 LINE 车道不做:
- 聊天内税务申报。
- 聊天内银行对账确认匹配。
- 聊天内创建/批量维护客户、商品、供应商。
- 聊天内配置 ERP、OAuth、模块开关。
- 员工报销审批。
- Drive/Sheets 外流。
- 通用全站 AI 助手。
- Rich Menu 全模块遥控器。

这些可以后续做,但必须单独设计权限、确认、回滚和 Web/LIFF 复核边界。

## 9. 验收基线

每次改 LINE 采购主路径,至少验:
- Amazon 70/140:total、VAT、date、invoice、PromptPay、主明细不回归。
- CP ALL/7-Eleven:不能把 Cash 当 total。
- Seafood 2722:total/subtotal/VAT/rounding 与票面一致。
- Little Betong 431:明细不完整时默认打开核对,不默认确认。
- reply 卡片说“识别错了”:进入纠错澄清,不回“请重新拍照”。
- 无引用说“识别错了”:提示回复具体记录。
- 闲聊/能力问答:不出现硬编码示例,不让 LLM 自由文案直出。
- 所有动作回复尽量带引用。
