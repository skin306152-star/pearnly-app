## 三、被证伪 / 已否决(别再走一遍)

1. **「推送日志成功但 Express 找不到」不是假成功。** 小助手真回执了单号,是账套写错(70EXP vs 69EXP)。
2. **「建套账按税号查重」不用做 —— 已经有了。** `routes/workspace_routes.py` `_create_validated_client` 已有 `workspace.tax_id_duplicate` 闸(422)。库里那三条同税号记录(ws 86/87/88,tax `0105562046201`)全是 `is_active=FALSE` 的停用存量,不参与路由。**这是我在清单里提错的一条,已撤。**
3. **库存的 8 条前置条件没有一条是 Pearnly 造的**(商品档 / 科目组 / 单位 / 仓库 / 入库 / 成本 / 日期顺序 / 数量足够),一条也简化不掉。Pearnly 的问题只是一条都没提前检查。
4. **⛔「缺进货单就自动建进货单」不能做。** 凭空建进货单 = 虚增存货和应付、**虚报进项税**、编造成本(系统不知道实际进价,拿售价倒推毛利变 0)。不是难,是不该 —— 那会是我们自己干的最严重一次静默改数。
   正确形态:**摸底 → 诊断 → 人填真实数据 → 机器写入**。人填的是**期初存货**(数量+单位成本+日期,三格),不是进货单 —— 期初只动存货一头,不碰应付和进项税,风险面小得多;Express 的期初就在商品档的 `ล็อตสินค้ายกมา <F7>` tab。真进货票走 C6 的正规入口。
5. **这家客户不该开库存模式。** 端点指纹 `stcrd_lines: 8, stcrd_lines_moving_stock: 0` —— 历史上一条明细都没动过真实库存。代账场景只有票、没有仓库流水和实物盘点,硬做永续账必然账实不符。

## 七、给客户的结论(已发泰文版)

Sincere Ice 用「ขาย • บริการ」服务模式就够,不要开库存。要开库存必须先满足:商品库存主档 → 科目组 → 单位换算 → 仓库 → **入库(进货单或期初)** → 成本金额 → 入库日期 ≤ 出库日期 → 数量足够。前四条是一次性主数据,后四条**每个商品每个月都要重新成立** —— 这意味着每月必须先录完进货票才能录销项票。这条没有捷径。
