## 三、数据集代表性评估

### 3.1 这个数据集能代表泰国会计真实生产场景吗?

**不能。** 理由:

1. **模板单一**:59 张里 ~30 张是 IV69 系列发票(基本只有 2 张原始,其他都是同一发票的不同 hash 副本 / 扫描复制 / 字符微变);BAKELAB 系列约 10 张也是同一卖家;Bangkok Office Supplies ~13 张是同一卖家不同 invoice no。**真正"不同卖家不同模板"** 大概只有 5-6 种。

2. **扫描件占比偏低**:80% 电子 PDF。真实生产场景里中小客户上传扫描件 / 手机照片比例会高很多(尤其是 LINE bot 入口几乎全是手机照片)。

3. **质量过于干净**:没有看到:
   - 倾斜 / 旋转的拍照
   - 暗光 / 反光 / 阴影
   - 折痕 / 污渍 / 撕角
   - 印章 / 手写补充 / 红色批注
   - 多张发票挤在一张 A4 上

4. **缺关键文档类型**:
   - 没有 receipts / 收据(感热纸,布局完全不同于 invoice)
   - 没有 credit notes / refund notes / 退款单(ใบลดหนี้)
   - 没有 ภ.ง.ด.3 / ภ.ง.ด.53 withholding tax 单独文档
   - 没有 multi-invoice PDF(一张 PDF 含多张不同 invoice_no — 实际场景:用户拍多张发票合成一个 PDF)

5. **特殊 case 没覆盖**:
   - 有 3 张 9707bdff 文件 (`4dfe`, `9f38`, `c49f`) 的 OCR 文本里出现大段 `■■■` 字符 — 说明 pypdf 抽取时**字体不可读**(常见于嵌入字体未授权的 PDF)— 但这种"字体损坏 PDF"在我们的 3 层架构里还没真测过

6. **VAT 报告 + GL 总账(`d256a4d3/`)** 是完全不同的文档类型 — 它们**不该走发票 OCR pipeline**,应该走 [vat_report_parser.py](../../vat_report_parser.py)。这俩混在 storage/pdfs/ 里要么是用户误上传,要么是不同入口共用了同一存储目录 — **需要 OCR 会话后续与用户确认是不是分流问题**。

### 3.2 缺失场景清单

按"是否触发新架构 layer 3 兜底"分类:

#### A. 触发率高的硬场景(layer 3 重点验证目标)

| 场景 | 触发原因 | 现有覆盖 | 建议样本数 |
|---|---|---|---|
| 手机拍照(倾斜/光照不均) | Vision 关键区域 conf 大幅低于 0.85 | **0 张** | 10-15 张 |
| 热敏纸收据 | 文字密度低 + 行距特殊 + 字体细 | **0 张** | 5-10 张 |
| 多发票合并 PDF(单 PDF 多 invoice_no) | invoice_grouper 输出 N 张,每张走全套 | **0 张** | 3-5 张 |
| 印章/手写覆盖关键字段 | conf 极低 + 字段缺失 | **0 张** | 5 张 |
| 折痕/污渍/低光照扫描 | conf 不稳 + 边缘字符识别错 | **0 张** | 5-10 张 |

#### B. 触发率中等(普通场景)

| 场景 | 现有覆盖 | 建议样本数 |
|---|---|---|
| 300 DPI 高质量扫描 | 3 张 | 已足够(再加 2-3 张增样本即可)|
| ~88 DPI 低质量扫描 | 9 张 | 已足够 |
| 多页 invoice(delivery note + receipt 套) | 11 张 | 已足够 |
| 双语(泰 + 英) | ~10 张 | 已足够 |
| 含 WHT(หัก ณ ที่จ่าย) | **0 张明确** | 5 张 |

#### C. 其他文档类型(非发票,本架构不该处理)

| 场景 | 现有覆盖 | 处置建议 |
|---|---|---|
| Credit note / 退款 | 0 | 收 3-5 张作为 `is_not_invoice` 控制集 |
| 报价单 / Quotation | 0 | 同上 |
| VAT 报告(รายงานภาษีขาย) | 1 | 应分流到 [vat_report_parser.py](../../vat_report_parser.py),不走 OCR pipeline。layer 2 输出 `is_not_invoice=True` 即可 |
| GL 总账(รายงานสมุดแยกประเภท) | 1 | 同上,走 [bank_recon_v2.py](../../bank_recon_v2.py) 的 GL 分支,不走 OCR pipeline |
| 银行对账单 | 0 | 同上,走 [bank_recon_v2.py](../../bank_recon_v2.py) |

### 3.3 准确率基线所需样本数(粗估)

要建立**可信的准确率基线**(置信区间在 ±5% 内):

- **每个独立场景**:最少 20-30 张样本,15 张才勉强够
- **总计**:50-100 张(覆盖各类场景的代表性子集)
- **当前可用**:实际 5-6 个独立场景 × 平均 5 张 / 场景 ≈ **真有效样本 25-30 张**

**结论**:**当前 59 张数据,去重并扣除非发票后真正可用的不超过 25 张**;远不足以建立生产级准确率基线。

### 用户额外需要提供的(本仓库没有)

## 六、扫描方法学(可复核)

本次分析跑了一次 Python 脚本(`/tmp/analyze_pdfs.py` 服务器上,**不进 git**),对每个 PDF 计算:
- `size_bytes` / `size_kb` — 文件大小
- `page_count` — pypdf 读出
- `avg_chars_per_page` — pypdf extract_text 后字符数 / 页数
- `is_electronic` — `avg_chars_per_page >= 200`
- `estimated_dpi` — PyMuPDF 找首页最大嵌入图像,用 `image_pixels / (page_points / 72)` 估算
- `first_page_text_sample` — 首页前 120 字符(用 ` | ` 替换换行)

输出 JSON 大约 39 KB,内含 59 条记录。**脚本临时性,不入仓库,本文档结论已落地**。
