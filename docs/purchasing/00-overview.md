# 商户采购(进项)· 00 概念与边界(B 路线 Phase 1)

> Pearnly 2.0 闭环第一块 = 把"花出去的钱"补上。动前先读这页 + `docs/product-vision/00`(2.0 定位)。
> 状态:图纸阶段(纯设计 · UI 稿先行待 Zihao 拍板)。前置地基:套账隔离(`docs/workspace-isolation`)。

## 一句话

**商户采购 = 记录"这家店花出去的钱"(进货 + 费用),拍照 / LINE 一句话自动入账,抵进项税,联动库存与报税。** 给**商户自己用**——求简单、自动、不用懂会计。

## 跟"事务所识别中心"区别(别混)

- **商户采购(本模块)**:我这家店进货、付供应商、记我的费用、抵我的进项税。商户栈。
- **事务所识别中心**:批量识别一堆客户公司的票 → 对账 → 喂 ERP。事务所栈,不是本模块。
- 详见 `docs/product-vision/00` §五-bis「两个进项分清」。

## 两个录入入口(都靠 AI,不让人手填)

1. **正式进项发票**(有 VAT,要抵扣):**拍照 / 上传 → OCR + AI 抽全字段**(供应商/税号/金额/VAT/品项)→ 确认 → 入账。税号/VAT 必须抽全,否则抵不了税。
2. **小额费用**(常无正式票):**LINE 一句话「清洁剂 50」/ 拍小票 → AI 自动归类**(交通/办公/水电…)→ 记一笔。

## 智能点(打破石板 ERP · 2.0 灵魂)

- 拍照即录,AI 抽全字段;**新供应商自动建、见过的自动匹配**。
- AI 自动推荐费用科目(打车→交通费),人只确认例外。
- LINE 一句话记账。
- **重复票自动拦**(防同一张票重复抵扣)。
- **月底进项税自动汇总**(联报税:销项税 − 进项税 = 应交)。

## 六类完整性(本模块设计必须全覆盖 · 见 product-vision §8)

| 类 | 本模块对应 |
|---|---|
| ① 操作面 | 拍照识别确认入账 · 进项/费用列表 · LINE 一句话记费用 |
| ② 老板配置后台 | 供应商管理 · 费用科目/分类 · 采购设置(默认 VAT/库存联动/付款) |
| ③ 登录/接入 | 复用主程序登录 · LINE bot 接入(一句话)· 角色(谁能录/审付款) |
| ④ 四态+边界 | 坏图/识别失败/无票费用/**重复票**/外币/未付 |
| ⑤ 角色权限 | owner/会计 录+审付款;员工录但不能审;供应商改 = owner |
| ⑥ 模块联动 | 进货→**库存入库** · 进项 VAT→**报税** · 供应商=**底座主数据** · 付款→**应付/银行对账** |

## 模块联动(seam 全列 · 设计层就接好 · 别只画屏漏联动)

采购"入账"那一下会牵动好几处,每条 seam 在图纸层定清,施工不返工:

| 联动 | 怎么接 | 状态 |
|---|---|---|
| 进货 → **商品库** | 进货品项先匹配现有 SKU(条码/名),配不上 → 一键建新商品(底座主数据) | ⚠️ **要补**:屏2 品项加"匹配/建商品"动作(否则入库对不上 SKU) |
| 进货 → **库存入库** | 确认入账 +「同时进库存」→ 按匹配到的 SKU 写 inventory 入库(批次/效期/成本) | ✅ 屏2 开关 + 设置 |
| 进项 VAT → **报税** | 入账即进"本月可抵进项税";月底 销项税 − 进项税 = 应交(ภ.พ.30) | ✅ 屏1 KPI(Phase3 出申报) |
| 付款 → **应付账款** | 未付 = 欠供应商;账期到期提醒;按供应商看欠款(应付 aging) | ⚠️ **要补**:现仅"付款状态"标签;完整应付 aging 归应收应付模块/单独屏 |
| 付款 → **银行对账** | 付款记录跟银行流水对上(确认真付了) | ⚠️ **seam 留**:接现有银行对账 |
| 入账 → **自动做账(Phase 2)** | 每笔入账 = 生成进项凭证(借 费用/库存+进项税,贷 应付/现金)喂做账引擎 | ⚠️ **seam 留**:采购入账留 hook,Phase 2 接 |
| 供应商 | = 底座主数据,按套账隔离 | ✅ 屏4 |

**三处采购阶段就要接好/留好,别漏**:① 进货品项↔商品库(否则入库对不上)② 应付账款 + 付款对账 ③ 入账→做账凭证 hook。

## 三个接入(都要 · Zihao 2026-06-08 确认)

- **手机端优先**:拍票、LINE 记费用、看花费,主场景在手机。所有屏**手机优先适配**(铁律:无 App 只有手机网页端,默认适配手机)。拍照本就是手机动作,桌面版次要。
- **接入 OCR**:进项票/小票识别**复用现有 OCR 引擎**,但要**扩 schema 认进项票/费用小票**(现 OCR 只识销项 `tax_invoice/receipt`,要加进项侧票种 + 抽供应商/进项 VAT)。
- **接入 LINE**:LINE 一句话记费用 + 拍小票走 **LINE bot**(复用现有 LINE 接入 @pearnly)。无票=记费用不抵税;拍正式票=抽 VAT 进可抵进项税。
- **LINE/上传 = 通用智能入口(别做死成纯事务所路径)**:现有 LINE 收图→OCR→识别中心是**单一路径无分流**(`services/ocr/line_image_ocr.py`)。2.0 在它上面加**智能分流**(详见 `docs/product-vision/00` §三-bis):AI 判"是什么(进项票/采购单/小票/一句话)+ 哪家公司(套账)+ 去哪(商户采购/事务所识别中心)"。采购 Phase 1 负责接出"商户采购"这个去处 + 激活文字记费用;套账分流由 workspace-isolation(PO-4 ocr_history)一并处理。**别删现有 LINE→OCR,它是这条管道的种子。**

## 套账隔离(硬要求)

进项单据 / 供应商 / 费用全部按 `workspace_client_id` 隔离(切公司只看本家)。供应商归套账不归租户。建在 workspace-isolation 地基之上。

## UI 设计稿(桌面 `Pearnly_采购_UI预览/` · 先预览拍板)

1. 采购/进项主屏(列表 + 花费总览 + 付款状态 + 四态)
2. 拍照识别确认(★核心:票图 vs AI 抽出的字段 左右对照,确认/改 → 入账)

4. 供应商管理(配置后台)
5. 采购设置(费用科目 + 默认 VAT + 库存联动 + 付款)
