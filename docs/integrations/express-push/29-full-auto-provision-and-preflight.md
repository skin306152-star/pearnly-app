## 0. 一句话目标 + Owner 三项拍板

**目标**:用户拿到一张发票,Pearnly 自动判断它在 Express 里**够不够条件落账**;够 → 高置信自动推;不够 → 说清楚缺什么、能自己补的自己补(建供应商/补科目)、补完自动重推;拿不准的才交人工。让用户**看得见问题、且大多数时候不用动手**。

**Owner 2026-06-23 三项拍板(本页据此设计)**:
1. **自建边界 = 全自建**:供应商(APMAS)和科目(GLACC)都可由 Pearnly 自动建,走置信度门 —— 高置信自动建、低置信转人工问。
2. **本次交付 = 设计文档**(本页),拍板后再分阶段施工。
3. **自治级别出厂默认 = 标准**:高置信自动、低置信问。

> 对齐行业:这就是 AP 自动化里的 **touchless / straight-through processing(STP)**——高置信自动过账、低置信转人工、只把真异常(<10–20%)交人,2026 标杆自动率 80%+。我们的"置信度托管 + 异常兜底"与该范式一致,方向是对的。

## 2. 核心新增 A:前置条件体检(Preflight)

### 2.1 心智模型
把现在"推送到第某道闸才失败"的隐式流程,**显式化成一张体检清单**。每张票推送前(或进异常后)能回答一句话:"这票为什么推不了 / 已经就绪。"

### 2.2 体检项(逐项 = 现有闸的显式化,不新增判定逻辑)

| # | 体检项 | 数据来源 | 红/黄/绿判定 | 失败对应现有码 |
|---|---|---|---|---|
| 1 | 账套已选且可写 | endpoint.config + 心跳 reported_account_sets.writable | 选了且 writable=绿 | `account_set_not_allowed` |
| 2 | 主体已绑定 | history.workspace_client_id 非 NULL | 非空=绿 | (前置)`direction_unknown` 根因 |
| 3 | 自己税号 13 位 | workspace_client.tax_id | clean 后 13 位=绿 | `direction_unknown` |
| 4 | 方向可判且启用 | direction.resolve + config.directions | 判得出且启用=绿 | `direction_unknown` / `direction_not_enabled` |
| 5 | 对手方在主档 | 小助手 reported APMAS/ARMAS(见 §4.3 扩展) | 税号精确命中=绿;名称疑似=黄;查无=红→可自建 | `SUPPLIER_NOT_IN_APMAS`(小助手) |
| 6 | 科目可定 | doc 28 瀑布结果 | 记忆/规则命中=绿;只落默认=黄;不可解=红 | `no_*_account` / `account_not_in_chart` |
| 7 | 金额自洽 | common.amounts(±0.02) | 平=绿 | `amounts_not_consistent` / `entry_not_balanced` |
| 8 | 账期未关 | (落账时 Express 反馈) | — | Msg#70/71 → "账期问题"异常 |

> **实现要点**:preflight **不引入新判定**,而是把 `enqueue_express` 那条闸链**抽成可单独调用、返回逐项结果的函数**(`preflight_express(history, endpoint) -> [{item, status, reason, fixable}]`)。`enqueue_express` 自身改为消费 preflight 结果(同一套逻辑,单一真相源,不分叉)。这样体检清单 = 推送闸,永不打架。

### 2.3 体检结果落点
- 不单独建表。preflight 结果随推送写入 `erp_push_logs.response_body`(已是 jsonb/text),`{preflight: [...], blockers: [...]}`。
- 异常页/详情抽屉读 response_body 渲染清单(见 §5 UI)。

## 3. 核心新增 B:自建主数据闭环(重头戏 · 风险最高)

> "全自建"已批准。但**写客户真账套的主档/科目表,比记一笔交易风险高一个数量级**——主档/科目表是会计命根子,建错影响所有后续账。所以本节核心不是"怎么建",而是**"什么时候敢自动建、护栏怎么厚"**。

### 3.1 总原则:实体匹配先行,绝不重复建
任何"自建"前,先回答**"它是不是已经存在"**。重复建档(同一供应商两条记录、税号录错建成新户)比"暂时没建"更难收拾。

### 3.2 自建供应商(APMAS)— 全自动(高置信门)

**触发**:方向=采购,卖方税号 13 位合法 + 名称齐全,但 APMAS 查无此供应商。

**判定流水线(在小助手本地跑 —— 它读得到 APMAS):**
1. **税号精确匹配**(13 位):命中 → 用已有供应商码,**不建**。
2. **名称归一模糊匹配**(复用云端 `merchant.normalize` 同款归一 + token-set 相似度):
   - 高相似(≥阈值)但税号不同 → **疑似重复/税号录错** → **转人工**(黄,别自动建)。
   - 无相似冲突 → 进自建候选。
3. **置信门**(全过才自动建):税号 13 位校验通过 + 名称非空 + 无名称冲突 + 字段宽度合法(cp874)。
4. **自动建 APMAS**:供应商码生成(⚠️ 规则待勘探:Express `ISRUN` 下一号 or 派生规则)、TAXID、名称、地址、`ACCNUM`=账套默认应付(ISINFO ACCNUM07)。
5. 建完 → **自动重推该票**(走同一 idempotency_key)。

**护栏(小助手侧,复用现有 DBF 护栏 + 主档增量)**:
- 写前整目录备份(已有)→ 单事务 → 建后**读回 APMAS 确认存在**(扩展 `_verify`)→ 失败回滚。
- 建前**再查一次**税号(防并发/缓存重复)。
- 重建 APMAS `.CDX` 索引。
- 字段宽度预检(已有同款:REFNUM≤15/SUPCOD≤10/TAXID≤15/ACCNUM≤15)。
- 错误码新增:`APMAS_WRITE_FAILED` / `SUPPLIER_DUP_SUSPECTED`。

**客户主档(ARMAS)对称**:方向=销项时同理建**票上买方客户**,`ACCNUM`=应收 ISINFO ACCNUM02。⚠️ 此处"客户"= 一张销项票的买方,记进**已存在账套**的 ARMAS,轻、与供应商同级。**不是**给事务所新客户"建整套账套"(见 §3.6)。

### 3.6 ⚠️ 自建套账(建账套)≠ 自建买方客户 —— 两个东西,都要,但极重的那个先研究

> Owner 2026-06-23 点出"客户"歧义并定方向:**两个都要**,且**"不用能做,要能做做完做好做对"**(不是能跑就行,要么不做、做就做对)。

| | 票上买方客户(ARMAS) | 自建套账(新账套) |
|---|---|---|
| 是什么 | 一张销项票的买方,记进**已存在账套**的客户主档 | 给事务所一个**新客户开整套新账**(= 一家公司全套账) |
| 轻重 | 轻(一条主档,= S3 供应商) | **极重**:建目录 + ISINFO(公司名/税号/默认科目)+ GLACC 整套科目表种子 + SCCOMP 登录注册 + ISRUN 单号种子 + 会计期。**种错一处,这家客户整套账全废** |
| 归属 | S4 | **S7(研究先行)** |

**S7 原则:绝不猜着建。** Express 在 DBF 层"如何新建账套"目前**未知**(哪些表/字段要种、能否 DBF 直建还是必须走 Express"新建账套"界面)。这是全系统**风险最高的写**。

**S7 落法**:先做**逆向勘探 spike**——在真 Express 上新建一个账套,抓出它动了哪些表/字段/种子/成功判据 → **证明能 100% 正确复刻才建**;若证明 DBF 直建不可靠,**改走 Express 界面**(一次性 RPA 或引导事务所操作员一次性建),而非硬写。需真 Express 访问(companion 侧 `D:\_express_audit` / Owner 机)。研究产出前不写任何建账套代码。

### 3.3 自建科目(GLACC)— ⚠️ 工程建议:默认不新建,落已有最接近;新建走人确认

> Owner 批了"科目也全自建"。作为工程负责人,我有责任标注一个**反对全自动新建科目**的专业判断,请你看完再定(本页 §10 留你拍板):

**为什么科目自建该比供应商保守:**
1. **场景极少**:doc 28 瀑布已保证收入/采购科目**几乎总能落到一个已存在的科目**(ISINFO 账套默认)。真正"目标科目在 GLACC 里不存在、必须新建"的票非常罕见——只在规则/AI 想用一个该账套没设的细分科目时才出现。
2. **风险极高**:新增一个 GL 科目 = 改科目表结构,影响报表(ภ.พ.30 / 财务报表)、所有后续自动过账、科目层级。建错一个静默错账,比建错一个供应商难查得多。
3. **Express 自己也不新建**:它的自动过账(G/L Link Account)永远从**已有** ISINFO 默认科目里取,从不为一张票新建科目。

**推荐设计(默认)**:
- 目标科目不在 GLACC → **回落到该账套已有的最接近科目**(瀑布的"账套默认"层,doc 28 §3.3),标"默认·待核",**不新建**。
- 若用户确实要细分(例:想把油费单独记 `52-09` 但账套没设)→ 进**待补条件卡**,人确认建一次。
- "科目也全自动新建"列为 **v2 可选旋钮**(`autonomy=全托管` 时才考虑开),且开了也要求父级科目存在 + 科目类型明确 + 写前白名单。

**若 Owner 坚持科目也全自动**:那必须满足比供应商更严的门——目标 code 的父级科目在 GLACC 存在 + 科目类型(资产/负债/收入/费用)能从瀑布确定 + 写前白名单二次校验 + 建后读回。我会照做,但默认仍建议上面的保守版。

### 3.4 自建动作的留痕(审计 · 必做)
每次自建写 `erp_push_logs.response_body.provisioned`:`{kind: supplier|account, code, name, tax_id, source, confidence, matched_by, at}`。供应商建档同时考虑学进 `expense_learned`(税号→供应商码记忆),下次直接命中。

## 4. 架构归属(谁在哪一层做什么)

```
云端(判定 + 门控 + 留痕)
  · preflight_express():逐项体检(= 抽出来的闸链)
  · 置信门 + 自治级别策略:决定"自动建 / 转人工 / 自动推"
  · 下发"意图":这票需要 [建供应商X / 用科目Y / 推送]
  · response_body 记 preflight + provisioned + 来源 + 置信
        ↓ 意图(不下发"怎么建",下发"需要什么")
小助手(唯一能读写 APMAS/GLACC/DBF 的手)
  · 实体匹配(读 APMAS/ARMAS,税号精确 + 名称归一)— 因为它读得到主档
  · 自建主数据(护栏:备份/单事务/读回/回滚/CDX/幂等)
  · 写交易 DBF(已有)→ ack 回单号 / 回"建了供应商X码=..."
  · 心跳扩展:除 GLACC chart,增报 APMAS/ARMAS 摘要(见 §4.3)
        ↓ 心跳 + ack
网页(看 + 补 + 设)
  · 体检清单视图 / 待补条件卡 / 自治级别设置(只读镜像科目,沿用 doc 28)
```

**关键决策:实体匹配 + 建档都在小助手本地做**,不在云端。理由:只有小助手读得到客户内网的 APMAS/GLACC,云端做实体匹配要么数据不全要么得反复问小助手,脆且慢。云端只下发"需要供应商X(税号+名称)",小助手本地判"已有就用、没有按门建"。

### 4.1 置信门控放哪
- **业务/会计置信**(科目定不定得准、方向判不判得出)→ **云端**(已有 `confidence.grade` + 瀑布)。
- **建档物理置信**(税号合法、名称无冲突、字段不溢出)→ **小助手**(它才有主档全集做匹配)。
- 两层都过才自动建/自动推;任一不过 → 转人工,带人话原因回到异常页。

### 4.2 自治级别如何贯穿(见 §6)
云端按 endpoint.config.autonomy 决定:高置信结果是"自动放行"还是"先问"。小助手永远只执行云端已门控过的意图。

### 4.3 心跳扩展(小助手 → 云端)
现心跳报:account_sets / GLACC chart / 选中账套 / 6 默认科目。
**新增报**:APMAS/ARMAS 摘要(`{tax_id, code, name}` 列表,上限如 5000,供云端 preflight 第 5 项判"对手方在不在主档",避免每票现问)。隐私:只摘要不全字段。

## 5. 核心新增 C 的 UI(增量,不重做)

> 现有集成页(卡片/日志卡/异常统计卡+issue卡/详情抽屉/待补科目卡)是真站验过的,**不重做,增量改 + 一个新视图**。沿用 doc 28 §6 的五落点,在其上加/改:

| 动作 | 落点 | 说明 |
|---|---|---|
| **新增** | 「推送就绪度 / 体检」清单 | 详情抽屉概览 tab 顶部 + 异常卡内:8 项红绿灯,一眼看出缺什么。把裸英文码统一成人话(复用 `_expressFriendlyReason` / `push_log_friendly`) |
| **升级** | 待补科目卡 → **待补前置条件卡** | 同一张卡,从只补科目扩到:补供应商(一键"建档")/ 补主体绑定 / 补科目。每种缺失一个动作按钮 |
| **新增** | 自治级别设置 | Express 连接卡 / 设置区一个三档开关(全托管/标准/全人工),per 连接 |
| **新增** | 自建动作可见 | 日志卡/详情显示"Pearnly 已为你建供应商 X(码=...)",来源+置信可见(诚实状态) |
| **清理** | 误导 UI | 0/0/0 计数器与实际打架、税号非 13 位静默无反馈等(记忆已记的几处),一并收口 |

**保持不变**:网页**绝不**放可编辑科目映射(doc 28 红线);科目仍只读镜像 + 异常补录。

### 5.1 引导式 UX 原则(Owner 拍板 · 不准甩"一堆看不懂的开关")

核心逻辑全在后端,但**用户面前不能出现裸技术**。每个元素必须自己解释自己:

1. **用结果说话,不用术语**:不写 `direction_unknown`,写"这张票还不知道是你开给别人(销项)还是别人开给你(进项)——通常是因为没认出你公司的税号"。失败码一律过人话层(`push_log_friendly` 4 语)。
2. **体检清单 = 进度感,不是错误墙**:8 项做成"✓ 已就绪 / ⚠ 待补 / ✗ 缺"的清单,绿的让用户安心,红黄的**每条直接带"这是什么 + 为什么需要 + 一键去补"**。不是报错,是"还差这两步就能自动了"。
3. **自治级别不是三个干巴巴选项**:做成**带后果说明的卡片选择**——
   - 全人工:"每张票都让我先看一眼再入账"(适合刚开始、还不放心)
   - 标准(推荐):"有把握的自动入账,拿不准的才问我"(默认,卡片标"推荐")
   - 全托管:"尽量都帮我搞定,只有真出问题才打扰我"
   每张卡下面一行小字说清"会发生什么",而不是让用户猜 autonomy=standard 是啥。
4. **自建动作要"看得见、可后悔"**:Pearnly 替用户建了供应商,不能静默——卡片明说"我帮你在 Express 建了供应商【ABC公司】(税号 …,记到应付科目 21-02-01),已自动入账。不对的话点这里改",给一个**撤回/更正入口**。建立信任靠透明,不靠藏。
5. **首次引导(onboarding)**:第一次进 Express 推送,走一个**轻量分步引导**(复用现有 wizard 骨架):①连小助手 → ②选账套 → ③确认 6 个默认科目 → ④选自治级别(默认帮选标准)→ "好了,以后拍发票我来记账"。一句话收尾,不留用户对着空界面发懵。
6. **空/加载/出错/正常 四态齐全**(铁律四态 UI):没票时说"还没有要推送的发票",而不是空白;小助手离线时整块给"小助手没连上,所有票会先留着等你,连上自动继续"。

> 落地不新造组件:体检清单 = 升级现有 issue 卡;自治选择 = 现有设置区加一组 radio 卡;onboarding = 复用 `18-t8-wizard-consolidation` 的向导。**新增的是文案与引导密度,不是新控件堆叠。**

## 6. 自治级别(三档 · 出厂默认=标准)

per 连接 `config.autonomy`:

| 档位 | 高置信(科目+建档物理都过) | 低置信 | 适用 |
|---|---|---|---|
| **全人工** | 转人工确认 | 转人工 | 刚上线、建立信任期 |
| **标准(默认)** | 自动建+自动推 | 转人工问 | 对齐 STP,Owner 选定的出厂默认 |
| **全托管** | 自动建+自动推 | 尽量自动(科目落默认标待核、供应商自动建);仅金额不平/方向判不出等硬阻塞才转人工 | 最省心、信任 Pearnly 的客户 |

- 阈值不写死,走 config / 环境变量。
- 任何档位都**守死三道防错账闸**(doc 28 §5)+ 金额低置信永不自动入账(铁律)。
- 全托管也不绕过"每账套首次确认闸"(doc 28 §5.3)。

## 11. 技术选型结论(调研后,诚实)

- **没有可直接套用的开源项目**:所有 touchless AP 产品都接有 API 的 ERP(SAP/Xero/QuickBooks),没人接 Express FoxPro/DBF + 客户内网。我们这套"云队列+本地 Agent 写 DBF"本就是为绕开这点自建的,只能继续自建。
- **实体匹配不引 Splink/dedupe**:量级小(一家几百供应商)、要确定性可审计、引重库违反最小净增。**税号精确 + 名称归一(merchant.normalize)+ token-set 相似度**足够。Splink 是百万行级杀器,这里杀鸡用牛刀。将来量真大再说。
- **不引第三方规则引擎**:铁律"金额/科目走确定性引擎不用 LLM"。保持手写确定性规则表(doc 28 瀑布已是)。
- **行业范式背书**:touchless/STP(高置信自动、异常转人工、导出前校验)证明方向正确,我们已具备大部分地基。
