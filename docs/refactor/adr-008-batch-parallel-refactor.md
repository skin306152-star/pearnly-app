# ADR-008 · 整顿提速用并行 worktree agent(`/batch`)· copy-out 并行 / 串行接线

> **状态**:已采纳(2026-05-27 · Zihao 拍板"整顿恢复后走 `/batch` 加速")· 方案就绪 · 等 ERP 收尾后启动
> **关联 task**:REFACTOR-C1(`home.js`)· B2(`db.py`)· C2/C3(css/html)· D1/D2(测试网)· G(文档)
> **关联文档**:`docs/refactor/BATCH_STRATEGY.md`(详版权威 · 5 波次 + 承包清单 + agent 黄金模板)
> **关联铁律**:#23 八硬门槛(每模块带守门测试)· #16(高敏改动 Zihao 在场)

---

## 背景

整顿剩下的大头是重复性高的拆分活:`home.js`(~22k 行)拆 50-100 个模块、`home.css`(16k)、
`db.py` 剩余安全域、几十个测试网。单窗口串行做要几周。同时这些活有个共同的撞车风险:**多个改动
都在删同一个巨石文件的行**,并行做天然冲突。

`/batch` 是 Claude Code 内置命令:一条命令 = 研究+规划一个大改动 → 自动开 5-30 个隔离
git worktree,每个派一个 agent 并行干 → 每个开一个 PR → 人挑能用的合并。**只能 Zihao 本人敲**
(内置命令,窗口里的 Claude 替不了)。

## 决策

**整顿提速改用 `/batch` 并行 worktree agent,核心是"拆分=并行(copy-out)· 接线=串行(serial-integrate)"。
按 5 个波次推进,每波内嵌护栏。**

### 核心三原则(整套方案就靠这 3 条立住)

1. **拆分=并行,接线=串行(copy-out / serial-integrate)**
   - 并行阶段:每个 agent **只"复制出去"**——把一块内容**拷贝**成新文件(`src/home/<feature>.js` / `services/<域>/store.py`)+ 写好契约/渲染测试,**绝不删原巨石任何一行**。各 agent 只新增文件 → 零冲突。
   - 串行阶段:所有 PR 合完后,**单独开一个串行窗口**,统一去巨石里删旧代码 + 加 re-export 接线(复用 ADR-007 范式),一块/一域跑一次守门一个 commit。
   - 危险的"动巨石"被收拢成一个小心的串行尾巴,其余全部安全并行。

2. **一个 agent 只碰自己那块新文件,永不两人同改一个老文件**
   - 开 batch 前先出**「承包清单」(manifest)**:谁负责哪个功能 / 对应巨石哪段行号 / 产出哪个新文件,**互不重叠**。这是 `/batch` "research and plan" 那步必须产出的东西。

3. **测试先行:先用 batch 把安全网铺满,再用 batch 碰巨石**
   - 动屎山前必须有冒烟网。建网本身也能并行,几十个测试一波铺完。没 E2E 网就无法验证"home.js 拆完每个页面还能渲染"。**所以 Wave 0 永远第一个跑。**

### 5 个波次

| Wave | 大任务 | 开几个 agent | 模式 | 前置 |
|---|---|---|---|---|
| **0 安全网** | D1 十大 E2E + D2 集成测试 | 10–30 | 纯新增文件 | 随时 |
| **1 后端收尾** | B2 db.py 剩余安全域 | 3–6 | copy-out + 串行接线 | ME.ERP 合并后 |
| **2 主战场** | C1 home.js 拆 50–100 模块 | 12–30/波 | copy-out + 串行接线 | Wave 0 全绿 + ERP P2-C 之后 |
| **3 样式/结构** | C2 home.css + C3 home.html | 10–25 | 按组件拆(多为新增) | C1 主体完成 |
| **4 收官** | C6 i18n 拆 json + G 文档/ADR + I 抛光 | 5–15 | 纯新增文件 | 随时 |

> **节奏铁律**:Wave 0 / 4 是"纯新增文件"型,不碰巨石、不撞 ERP,任何时候都能跑。Wave 1/2/3 动巨石,必须等 ERP 收尾线(P2-B/C/P3)全部上线后再开(agent 从当时 HEAD 切 worktree,基线要新)。**尤其 Wave 2(home.js)必须在 P2-C 之后**——P2-C 也改 home.js,否则两边切同一文件 = 撞车 + 白拆。

## 护栏清单(按下按钮前对一遍)

- 开 batch 前 `git tag` 备份当前 master。
- 第一次只在 **Wave 0** 试水,看 agent 守不守门、PR 质量。
- 每个 agent 指令**内嵌 6 道守门 + 契约测试 + REFACTOR-id**(format:check / 全量 unittest / check_imports / check_i18n / node --check / build · E2E 按需)——否则批量产出违反 8 硬门槛 = 偷渡新债。
- **高敏域(登录/计费/OCR 热路径/auth/RLS)永远不进 batch**,Zihao 在场单独做。
- 动巨石的波次**必须等 ME.ERP 合并**。
- 删除巨石旧代码这步**串行**,不并行。
- 每个 PR Zihao 都过一眼(这步没人能替)。
- 每波结束跑 `python scripts/refactor_progress.py` 看数字真掉了。

## 分工(自动 vs Zihao)

| 环节 | 谁干 |
|---|---|
| 读文档、找下一波、出承包清单 + batch 指令 | Claude 窗口(自动) |
| 写代码、跑 6 道守门、开 PR | batch agent(自动) |
| **敲 `/batch <指令>`** | **Zihao**(内置命令只能本人敲) |
| **review PR、决定合不合** | **Zihao** |
| 串行接线删巨石 | Claude 窗口(自动,但一步步停下汇报) |
| 高敏(登录/计费/OCR/auth) | **Zihao 在场** Claude 才动 |

## 理由

1. **撞车风险被结构性消除**:把"删巨石"从并行里彻底剥离成串行尾巴,并行的全是"只新增文件",数学上零冲突——这是整套方案能成立的关键。
2. **承包清单防重叠**:plan 阶段强制产出 manifest,确保没有两个 agent 碰同一段巨石,合并时拷出型 PR 理论上可批量合。
3. **测试网先行**:Wave 0 先铺 E2E/集成,既是安全网,也是 batch 工作流的试水(在最安全的活上验证 agent 守不守门),再去碰巨石。
4. **守门内嵌进 agent 指令**:每个并行 agent 自带 6 道守门 + 契约测试 + REFACTOR-id,批量产出不破坏 8 硬门槛。
5. **copy-out 复用 ADR-007 范式**:后端 Wave 1 的 copy-out 就是 B2 的 `import db` + `db.get_cursor()` + 尾部 re-export;串行接线窗口统一删旧 def + 加 re-export。

## 取舍 / 边界(诚实的预期)

- **不可能一晚全完**:`home.js` 22k + `home.css` 16k 体量在那;每个 PR 要人过目;"删巨石接线"是真实串行尾巴;登录/计费/OCR 热路径规矩写死要 Zihao 在场。真实闭环 ≈ 并行 80% 飞快 + 高敏/串行尾巴 20% 是 gated 工作。
- **`/batch` 只能 Zihao 敲**:窗口里的 Claude 只能写承包清单 + 黄金指令,递给 Zihao,由他执行内置命令。
- **当前状态(2026-05-27)**:方案就绪 · 设计阶段 · 尚未开跑。启动前提是 ERP「开箱即用」收尾(P2-B 日志折叠 / P2-C 不裸透泰文 / P3 概念导航)修完——按"BUG > 整改",这三项是整顿启动的前置。
- **进度账本**:Wave 0 已在第三十五会话(2026-05-27)**未用 `/batch`、在单窗口直接做**了一批(17 个核心纯逻辑模块补行为契约 · unit 872→1095 · 挖修 3 真 bug);E2E 仍 1/10(需账号)。Wave 1/2/3/4 仍 ⚪ 待启动。

## 后果

- ✅ 整顿"如何加速"有了单一权威打法(详版在 `BATCH_STRATEGY.md`),接力窗口进来按 §10 进度账本找当前波次即可。
- ✅ copy-out + 串行接线把"动巨石"的高风险收拢可控。
- ⚠️ 依赖 Zihao 全程在环(敲 `/batch` + review PR + 高敏在场),不是无人值守的全自动。
- 📌 完成判定 = 解禁新功能:`home.js` < 200 / `app.py` < 500 / `db.py` < 500 / 模块 ≥ 100 / unit ≥ 500 / 集成 ≥ 20 / E2E ≥ 10 / 覆盖 ≥ 70% / 第三方体检"Google 级 90%+"。
