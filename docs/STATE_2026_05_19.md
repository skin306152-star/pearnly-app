## 3. Unfixed problems · current understanding

### P-1. `push_mrerp` is STILL a stub
- **Where**: `erp_push.py:207` · `push_mrerp(endpoint_config, payload)` returns `(False, 0, "mrerp push is not wired into push_to_endpoint yet; use MRERPAdapter.upload_invoice_batch directly")`
- **Why**: `push_to_endpoint` operates on single OCR records; `MRERPAdapter.upload_invoice_batch` operates on batches of histories with mappings. The wiring to feed single OCR rows through the batch adapter was scoped to "push-pipeline rewrite", never landed.
- **Impact**: Wizard saves the endpoint successfully (CHECK constraint fixed in v118.34.14), test-connection works (login fixed in v118.34.15), but clicking **「推送」** on a history row → still hits the stub → user gets `error_msg: "mrerp push is not wired..."` in the push log.
- **Guard test that locks this in**: `StubContractTests.test_push_mrerp_is_still_a_refusal_stub` deliberately asserts the stub stays a stub until someone wires it intentionally.
- **Next-window task**: implement single-history → batch adapter bridge. Probably new function `push_single_via_mrerp_adapter(endpoint, history) → (success, http_status, body)` that builds a 1-item batch, uses `enable_master_data_sync=True` + seed codes from endpoint.config.

### P-2. Login form lookup may still race on truly slow / weird MR.ERP responses
- **Where**: `services/erp/mrerp_adapter.py:_wait_for_login_inputs_with_retry`
- **Why**: 15s × 2 (initial + reload) is empirical; MR.ERP could legitimately take longer during their deploy windows. The 5-selector fallback list (`txtusers` / `txtuser` / `txtusers` id / `input[id*='user' i]` / `input[type='text']:visible`) covers known variants but not unknowns.
- **Impact**: User reported test-connection succeeded once then flapped. v118.34.15 should be MUCH more robust but isn't user-verified across 3 runs (window ended before that verification).
- **Next-window task**: have user click 3 times after deploy lands; if ANY fail with `screenshot=/tmp/mrerp_login_fail_*.png` in raw_error, ask operator to `scp` the screenshot to see what the form actually looks like in failure mode.

### P-3. Wizard re-edit of saved endpoint may double-encrypt
- **Where**: `app.py:erp_endpoints_create` v118.34.13 fix encrypts `username_enc`/`password_enc` if `not is_encrypted(v)`. But `update_erp_endpoint` (the PATCH path used when editing existing endpoint) has **no equivalent encryption logic**.
- **Why**: I added encryption to POST `/api/erp/endpoints` but did NOT mirror to PATCH `/api/erp/endpoints/:id`. When wizard's `_wizardFinish` edits an existing endpoint, it sends new plaintext password under `password_enc` field — PATCH stores it as-is — next test-connection tries to decrypt plaintext → InvalidToken → ERR_CRED_DECRYPT.
- **Impact**: First-time setup works (POST path). Editing credentials on an existing endpoint breaks.
- **Next-window task**: Mirror the `kms_helper.encrypt_str` + `is_encrypted()` guard from the POST route into the PATCH route handler in app.py.

### P-4. `erp-mrerp-connect.js` wizard JS doesn't surface screenshot path from raw_error
- **Where**: `static/erp-mrerp-connect.js:_wizardRunTest` displays `data.raw_error` text in the error bar, but doesn't pull out the `screenshot=/tmp/...png` substring as a separate hint.
- **Why**: When v118.34.15 saves a failure screenshot, the path lands in `raw_error` but the user sees it inline in the dense raw-error monospace block. Could be more useful as a "Save this path: /tmp/mrerp_login_fail_xxx.png — send to support" callout.
- **Impact**: Cosmetic — diagnostic is captured, just not foregrounded.
- **Next-window task**: small regex in `_wizardRunTest` to extract `screenshot=(\S+\.png)` and render it as a separate label below the friendly message. Optional.

### Decisions made this window
- **4-lang split**:
  - home.js dict: `zh / en / th / ja`
  - MR.ERP friendly catalog (`mrerp_business_friendly.py`): `zh / en / th / zh_TW` (Zihao swap; MR.ERP user demographic skews Traditional)
  - `erp-mrerp-connect.js` local `T` dict carries entries for ALL 5 keys (`zh/en/th/zh_TW/ja`) so either active-lang resolves
- **Adapter whitelist** (must be in sync across these 3 places):
  - Python: `erp_push.ADAPTER_REGISTRY` = `{webhook, flowaccount, mrerp}` (xero handled separately via OAuth, not in this dict)
  - DB: `erp_endpoints_adapter_chk` CHECK = `('webhook', 'xero', 'flowaccount', 'mrerp')` (v118.34.14)
  - Canonical Python tuple: `db.ensure_erp_endpoints_adapter_constraint`'s `canonical = ("webhook", "xero", "flowaccount", "mrerp")`
- **ERR_CODE catalog** (all 4-lang in `mrerp_business_friendly.py` + hardcoded fallback in `erp_push.py:test_mrerp_endpoint._FALLBACK_FRIENDLY`):
  - `ERR_NO_CREDS` — username/password missing in config
  - `ERR_CRED_DECRYPT` — Fernet decrypt failed (often plaintext stored in `_enc` field; see P-3)
  - `ERR_AUTH` — MR.ERP rejected credentials
  - `ERR_TECHNICAL` — network timeout / 5xx / missing selector after wait+reload
  - `ERR_BUSINESS` — MR.ERP accepted login but rejected business data
  - `ERR_UNEXPECTED` — catch-all
  - `ERR_PLAYWRIGHT_MISSING` — `import playwright.sync_api` ImportError
  - `ERR_KMS_MISSING` — `PEARNLY_KMS_KEY` env unset
  - `ERR_ADAPTER_NO_CUSTOMERS` / `ERR_ADAPTER_NO_PRODUCTS` — non-mrerp adapter hitting the customer/product listing route
  - `ERR_HTTP_<status>` — client-side fallback when fetch returned non-OK
  - `ERR_NETWORK` — client-side fetch exception

- **Async guard requirement** (v118.34.10/12 — IRON RULE going forward):
  - Any FastAPI `async def` handler that calls a sync `MRERPAdapter` helper (or any function transitively using Playwright sync_api) MUST wrap with `await asyncio.to_thread(...)`. Playwright `sync_api` explicitly checks `asyncio.get_running_loop()` and refuses to start when one exists.
  - `AsyncLoopOffloadTests` in the test file plants tripwire helpers that fire `RuntimeError` if called inside an event loop. Any new route touching MRERPAdapter MUST get a corresponding tripwire test there.
  - Lifespan itself is `async def` → lifespan calls to `_ensure_playwright_installed` must also use `to_thread` (v118.34.12 specifically fixed this).

- **Seed copy-from-seed pattern** (P1-B):
  - Customer master-data auto-create: clone from `seed_customer_code` (default `0006` — Skin Trading on test01). Inherits salesman / account code / unit / address refs.
  - Product master-data auto-create: clone from `seed_product_code` (default `P001` — Pepsi 500ml). Inherits unit / account / pricing refs.
  - Wizard Step 3 dropdowns populate from `/api/erp/endpoints/:id/{customers,products}` (60s cache).
  - Both seed codes live in `endpoint.config.seed_customer_code` / `endpoint.config.seed_product_code` (inside the JSONB blob).
  - Levenshtein thresholds: customer 0.82, product 0.90.

- **Chromium launch flags (Iron rule for all MRERPAdapter calls server-side)**:
  - `--no-sandbox` (mrpilot runs as root — sandbox=true crashes)
  - `--disable-dev-shm-usage` (VPS /dev/shm is ~64MB)
  - `--disable-gpu` (headless server, no display)
  - Lives in `services/erp/_browser.py:BrowserSession.__enter__`. Local dev w/ display unaffected.

- **Diagnostics on `/api/version`** (use these for next-window debugging without SSH):
  - `version` — frontend cache-bust number (`PEARNLY_FRONTEND_VERSION` from home.html `?v=`)
  - `playwright.playwright_installed` — pip package importable
  - `playwright.chromium_installed` — chromium binary on disk
  - `playwright.chromium_can_launch` — actual launch+version probe succeeded
  - `playwright.chromium_launch_error` — exact failure if launch broke
  - `playwright.python_bin` — `sys.executable` (confirms which interpreter was probed)
  - `playwright.deploy_log_tail` — last 40 lines of `/var/log/mrpilot-deploy.log`
  - `playwright.sentinels` — `/tmp/pearnly-*` files currently on disk
  - `last_500` — `{ts, path, method, exc_type, detail, traceback}` for most recent 500 (any route)

- **Webhook deploy pipeline**:
  - GitHub push → `/internal/deploy` (HMAC-verified) → detached bash → `/opt/mrpilot/git-deploy.sh`
  - `git-deploy.sh` is written by `app.py` lifespan at startup (heredoc in app.py). Heredoc now includes pip install of `playwright` + `playwright install chromium` + `playwright install-deps chromium`, all `timeout`-wrapped, with `MAX_WAIT=180s` for the post-restart health check.
  - **Chicken-and-egg**: a buggy heredoc takes effect only AFTER the lifespan that writes it runs successfully. v118.34.7/8 both wrote bad heredocs that propagated to subsequent deploys.

- **Bash deploy.sh constraints** (lessons from rollback cycle):
  - Don't `grep -oP` for autodetect — PCRE support varies. Use bash `case` instead.
  - Don't `pip install -r requirements.txt` in deploy.sh — drags in heavy deps (torch/easyocr) on first install. Just install the specific dep that's new.
  - Every install step needs `timeout` wrapper (60s pip, 120s chromium binary, 180s `install-deps` apt step).
  - PEP-668 hosts (Debian/Ubuntu 23.04+): system pip requires `--break-system-packages`; venv pip doesn't. **Always prefer `/opt/mrpilot/venv/bin/python -m pip`** over system `pip3` — that was the v118.34.6→.7 root-cause hop.
