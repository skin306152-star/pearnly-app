## 一、速度数据表

### 1.1 4 种配置对照

| 配置 | 文件数 | 总页数 | 总耗时 | 吞吐 | 加速比 | 峰值内存 | 单页成本 | 错误 | rate-limited |
|---|---|---|---|---|---|---|---|---|---|
| **c=1**(串行 baseline) | 49 | 97 | **757.5 s**(12.6 min)| **7.7 页/分钟** | 1.0x | ~200 MB(估)| ฿0.0843 | 0 | 0 |
| **c=5**(并发 5) | 49 | 97 | **180.7 s**(3.0 min)| **32.2 页/分钟** | **4.2x** | 670 MB | ฿0.0844 | 0 | 0 |
| **c=10**(并发 10) | 49 | 97 | **137.5 s**(2.3 min)| **42.3 页/分钟** | **5.5x** | 937 MB | ฿0.0837 | 0 | 0 |
| **大文件 60p**(c=1)| 1 | 50¹ | **199.4 s**(3.3 min)| 15.0 页/分钟 | n/a | 249 MB | ฿0.0682² | 0 | 0 |

注:
1. 合成的 60 页 PDF 被 pipeline 自动截断到 `max_pages=50`(默认值)
2. 大文件用 scan_002.pdf 内容(BAKELAB,无 L3 触发),所以单页成本低于普通批次

### 1.2 单页耗时分位数

| 配置 | min | p50 | p90 | p99 | max | mean |
|---|---|---|---|---|---|---|
| c=1 baseline(估算)| ~2 s | ~5 s | ~25 s | ~75 s | ~75 s | ~7.8 s |
| c=5 | 1.9 s | **4.2 s** | 26.7 s | 74.4 s | 74.4 s | 10.9 s |
| c=10 | 2.0 s | **5.2 s** | 27.9 s | **104.4 s** | 104.4 s | 12.4 s |
| 大文件 60p | 4.0 s | 4.0 s | 4.0 s | 4.0 s | 4.0 s | 4.0 s |

**观察**:
- p50 在 c=5 vs c=10 几乎不变(4-5 秒)— 单页 happy path 时间稳定
- **p99 在 c=10 反而上升到 104s**(c=5 是 74s)— 因为 photo_011 这种"L3 走 retry 全长"的页变成在更繁忙的 thread pool 里排队等
- 大文件批次单页 4 秒非常稳定 — 没 L3 触发,纯 L1+L2

### 1.3 加速比效率

| 并发 | 实际加速 | 理论上限 | 效率 |
|---|---|---|---|
| 5 | 4.2x | 5x | **84%** |
| 10 | 5.5x | 10x | **55%** |

**结论**:**c=5 是 sweet spot**。c=10 边际收益骤减,因为长任务(L3 失败 retry 的 75-104 秒页)成了 stragglers,10 个 worker 闲等。

## 四、推荐生产配置

### 4.1 并发数选择(分场景)

| 入口 | 推荐并发 | 理由 |
|---|---|---|
| `/api/ocr/recognize` 单页 PDF 上传 | **1**(单文件无并发可言)| 用户上传 1 张就过 pipeline,p50=5s 已经达标 |
| LINE bot 单图 | **1**(单图单线程)| 用户体验 <10s 就可 |
| email_ingest 后台批量 | **5** | 偶发批量(几张到几十张/邮件),5 并发足够 |
| **recon_routes 批量对账** | **5**(从当前 20 调低)| 月底大批量场景;实测 c=5 效率 84% 远高于 c=20 的"理论值",且内存只 670 MB 不挤兑 |

**关键建议**:接入 app.py 时,把 [recon_routes.py:648](../../recon_routes.py) 的 `asyncio.Semaphore(20)` 调到 **5**。原 20 假设的是 OCR 是 IO bound,但实测加上 L3 视觉调用后单页 5-25 秒,5 并发已能跑满 API 配额且效率最高。

### 4.2 超时配置

| 层 | 当前 | 推荐 | 备注 |
|---|---|---|---|
| Layer 1(Vision)| 60 s | **30 s** | 实测正常情况 <2s,30s 足够覆盖 p99 |
| Layer 2(Flash-Lite)| 60 s | **30 s** | 同上 |
| Layer 3(Flash 视觉)| 90 s | **60 s** | 视觉调用慢,60s 覆盖大多数 L3(实测 16-25s 典型,极端 ~100s) |
| Pipeline 整体(per-page)| 不限 | **180 s** | 即 L1+L2+L3 三层都跑完最坏情况 |
| Pipeline 整体(per-file 多页)| 不限 | **600 s**(10 min)| 50-page PDF 实测 200s,留 3x buffer |

### 4.3 retry 策略

| 层 | 当前 | 推荐 | 备注 |
|---|---|---|---|
| L2 JSON parse 失败 | 1 次重试 | **保持**(实测足够)| B2 修后 L3 JSON 失败率已从 2% → 1% |
| L3 JSON parse 失败 | 1 次重试 + retry prompt 强化 | **保持** | B2 已加 retry hint,降一半失败率 |
| Pipeline 入口失败 | 无 | **不在 pipeline 层做** | retry 上推到 app.py 接入层,因为业务 retry 还要看用户配额、缓存等 |
| L3 Quota / Transient 错误 | 抛 Layer3*Error | **保持** | 调用方决定是 backoff 重试还是降级到 L2 输出 |

## 七、内存监控

### 7.1 峰值 RSS(server-side `resource.RUSAGE_SELF.ru_maxrss`)

| 配置 | 峰值 MB | 增长 vs c=5 |
|---|---|---|
| c=1 baseline(估算)| ~200 | -70% |
| c=5 | 670 | baseline |
| c=10 | 937 | +40% |
| 大文件 60p(c=1)| 248 | -63% |

**内存随并发数大致线性增长**(每个 worker ~70-100 MB)。

### 7.2 内存压力点

每个 worker 持有:
- Vision API client(gRPC 连接 + buffer)~50 MB
- 当前正在处理的 PDF 渲染图片(200 DPI A4 ~3 MB / 页)
- pydantic 模型(Layer1Result 大文件可数 MB)

**生产部署建议**:
- 给 `mrpilot` 服务至少 **2 GB RAM**(c=5 配置 + buffer)
- 单进程下 c=10 需要 1 GB
- **不建议 c=20+** — 内存上 1.5+ GB 容易 OOM

## 八、关键发现

### 发现 1:c=5 是甜点(diminishing returns at c=10)

84% 并行效率,远高于 c=10 的 55%。生产用 c=5,把延迟降低 4x 同时不浪费机器资源。

### 发现 2:L3 触发率在并发下基本不变(19% ± 1%)

19/97 (c=5) vs 18/97 (c=10) — InvoicePatternMemory 在不同线程顺序下学习的"已知 pattern"略不同,导致触发数 ±1,可忽略。

### 发现 3:成本几乎不受并发影响

฿0.0843 / ฿0.0844 / ฿0.0837(c=1/5/10)— **API 单页价是固定的,并发只压缩 wall-clock,不增减总成本**。

### 发现 4:rate limit 完全没问题

c=10 实测使用率 < 10% 各 API 配额。可以继续放量到 c=20,但收益(从效率 55% → 更低)和内存代价不划算。

### 发现 5:Pipeline 对 >50 页 PDF 自动截断

`max_pages=50` 默认值在 [pipeline.py:DEFAULT_MAX_PAGES](../../services/ocr/pipeline.py) 控制。大文件测试输入 60 页 → 实际处理 50 页 → 输出无错。生产里 GL 13 页等远未触发,但偶发的扫描合并文档可能撞到。**建议留这个上限**(防止单个超长 PDF 拖垮系统)。

### 发现 6:L3 JSON 失败仍偶发(2% 重新出现)

c=5 / c=10 都见过 `expected JSON object, got list` 错误 — Gemini Flash 偶尔输出 JSON 数组而不是对象。我的 `_parse_json` 已拒绝数组(B2 修过),触发 retry。retry 仍可能拿到数组 → L3_failed → manual review。**这是 Gemini Flash 输出格式问题**,需要在 prompt 里加"single object only"约束(B3 候选改进)。

## 九、上线建议(性能维度)

| 行动 | 优先级 | 理由 |
|---|---|---|
| 接入 app.py 时,所有 OCR 入口用 c=5 并发 | 🟢 必做 | 实测 sweet spot,4x 加速,内存 OK |
| recon_routes.py 把 Semaphore(20) → Semaphore(5) | 🟢 必做 | 反直觉但实测 c=5 比 c=20 效率高,且省内存 |
| 配置生产 mrpilot.service 至少 2GB RAM | 🟢 必做 | c=5 需要 700 MB peak,留 3x buffer |
| 给 pipeline.run 加 600s 文件级超时 | 🟡 建议 | 防止单个超长 PDF 拖垮 worker |
| L3 失败 retry prompt 再加"single object"约束(B3)| 🟡 建议 | 还能再压 L3 失败率 |
| L3 异步化(用户先看 L2 结果)| 🔵 后续 v2 优化 | 优化 p99 用户体验,工程量较大 |
