## 1. 一句话结论

这是一个已经承载真实业务的 FastAPI + 原生前端 SaaS 项目，功能量很大，也有文档、测试和线上事故复盘沉淀；但整体仍然是“能跑的产品型巨石系统”，还不是标准 SaaS 工程结构。

最大问题不是“代码写得丑”这么简单，而是:

1. `app.py`、`db.py`、`home.js`、`home.css` 过大，业务边界被揉在一起。
2. 数据库 schema / migration 主要靠启动时 `ensure_*` 函数和管理接口维护，缺正式迁移体系。
3. 多租户、计费、权限、ERP、OCR 都已经进入商业复杂度，但测试和 CI/CD 保险还没跟上。
4. 有几个安全点需要优先收紧，尤其是内部部署端点、公开诊断信息、CORS 和前端 token 存储。
5. 项目并非无法救，反而已有不少“救生绳”: 文档多、测试有一定基础、`services/ocr` 和 `services/erp` 已开始模块化、admin SPA 已从 `home.js` 独立出来。

建议路线: 不要推倒重来。先补安全闸和自动化检查，再按模块渐进拆分。

## 5. 主要风险清单

### P0 / 高优先级

#### 1. 内部部署端点应当 fail-closed

位置: `app.py` 的 `/internal/deploy`

当前逻辑是: 如果 `GITHUB_WEBHOOK_SECRET` 存在，就校验 GitHub HMAC；如果 secret 没设置，则不会校验签名，仍会进入部署流程。

风险:

- 生产环境如果因为配置丢失导致 secret 为空，部署入口会从“受保护”变成“无签名保护”。

建议:

- 如果 `GITHUB_WEBHOOK_SECRET` 缺失，直接返回 403/500，禁止部署。
- 更正核对: `/internal/deploy/manual`、`/internal/deploy/log`、`/internal/install-playwright` 当前已经有 `if not secret or token != secret`，属于 fail-closed；不要把它们当作同级 P0 漏洞处理，只需避免后续改坏。

#### 2. `/api/version` 公开暴露诊断信息和最近 500 traceback

位置: `app.py` `/api/version`

当前返回:

- `playwright` 状态
- `last_500`，包含 traceback 片段
- release notes

风险:

- 公开接口暴露内部路径、异常类型、调用栈、部署状态，会帮助攻击者理解系统结构。

建议:

- `/api/version` 只保留 `version`、`release_notes`、必要的公开状态。
- `last_500`、deploy log、playwright diagnostics 移到超管鉴权接口。

#### 3. 全局异常处理向客户端返回异常类型和异常文本

位置: `app.py` `@app.exception_handler(Exception)`

当前返回类似:

- `detail: "{ExceptionType}: {message}"`
- `diag_url`

风险:

- 生产用户/外部调用者可能看到内部异常信息。

建议:

- 客户端统一返回通用错误码，如 `server.internal_error`。
- 详细 traceback 只进服务端日志或超管诊断接口。

#### 4. 前端 token 存在 `localStorage`

位置: `home.js`, `login.html`, `static/admin/admin.js`

观察:

- `mrpilot_token` 存在 `localStorage`。
- 请求时手动加 `Authorization: Bearer ...`。

风险:

- 一旦发生 XSS，token 很容易被读取。

建议:

- 短期: 加 CSP、减少 `innerHTML` 动态拼接、严格 escape。
- 中期: 评估迁移到 HttpOnly Secure SameSite cookie + CSRF 设计，或至少缩短 token 生命周期并加强刷新策略。

#### 5. CORS 过宽

位置: `app.py`

当前:

```python
allow_origins=["*"]
allow_credentials=True
allow_methods=["*"]
allow_headers=["*"]
```

风险:

- 对 SaaS 生产环境来说过宽。虽然 Bearer token 不会像 cookie 那样自动跨站发送，但仍应收紧。

建议:

- 生产只允许 `https://pearnly.com` 和明确的管理/测试域名。
- 本地开发用环境变量单独打开 localhost。

### P1 / 架构风险

#### 6. 后端入口是巨石

`app.py` 10,008 行，包含:

- FastAPI app 初始化
- lifespan 启动逻辑
- schema ensure
- 部署脚本写入
- OAuth
- OCR
- ERP
- admin
- billing
- static HTML routes

风险:

- 改一个局部功能时容易误伤启动、权限、部署、前端版本等无关部分。
- 路由 ownership 不清晰，后续 agent 容易重复造轮子。

建议:

- 按 domain 拆 router: `auth`, `ocr`, `history`, `erp`, `recon`, `billing`, `admin`, `team`, `static_pages`。

#### 7. `db.py` 既是数据库层，也是迁移层，也是业务服务层

`db.py` 9,230 行，包含:

- 连接池
- schema 创建/ALTER
- CRUD
- 计费扣费
- admin 查询
- ERP mapping
- 多租户迁移

风险:

- 事务边界难统一。
- schema 变更靠启动时执行，缺版本化 migration。
- 难做测试替身和分层。

建议:

- 引入正式 migration 工具，比如 Alembic。
- 将 `db.py` 拆成 `db/session.py`, `repositories/*`, `migrations/*`。

#### 8. 多租户隔离逻辑存在，但路径混杂

观察:

- 代码大量使用 `tenant_id`。
- 部分查询通过 `tenant_id` 直接过滤。
- 部分查询通过 `user_id IN (SELECT id FROM users WHERE tenant_id = %s)`。
- 还有历史 fallback、孤立用户修复、lazy tenant 创建、membership 模型迁移、RLS 试点接口。

风险:

- SaaS 最怕数据串租户。当前实现有许多防线，但不是统一模型。

建议:

- 先画出“租户隔离矩阵”: 每张表 owner / member / super admin 谁可读写。
- 为关键表补自动化隔离测试。
- 逐步统一到单一租户 membership 模型。
- RLS 要么正式推进，要么从“试点接口”中明确标记为未启用，避免误判。

#### 9. 前端主体验仍然是巨石

`home.js` 33,768 行，`home.css` 16,124 行，`home.html` 6,562 行。

风险:

- UI 改动难定位。
- i18n、状态管理、API 调用、DOM 渲染、业务逻辑混在一起。
- 大量 `innerHTML` 和 localStorage 逻辑增加 XSS 和状态残留风险。

建议:

- 不再往 `home.js` 加新功能。
- 优先把稳定边界模块拆出去: dashboard、OCR upload、history、clients、billing、recon、ERP。
- 每拆一个模块配一个 Playwright smoke。

#### 10. 依赖和本地环境不标准

`requirements.txt` 存在，但:

- 多个依赖未锁版本: `fastapi`, `uvicorn`, `psycopg2-binary`, `bcrypt`, `passlib`, `PyJWT`, `xlrd`, `google-generativeai`, `playwright`。
- 没有 `pyproject.toml`、`pytest.ini`、`Dockerfile`、`.github/workflows`。
- 本机静态 import 检查结果:
  - 根目录 `.py` 文件 33 个
  - 29 个 import OK
  - 4 个 import 失败
  - 缺 `passlib`, `psycopg2`, `xlrd`

建议:

- 先补本地依赖，让项目能在本机最小启动。
- 增加锁定方式: `requirements.lock.txt` 或转 `uv/poetry`。
- 增加 CI: import check + i18n check + unit tests。

### P2 / 中长期治理

#### 11. 文档很多，但导航和时效性需要整理

项目文档价值很高，但存在:

- 多个状态文档、handover、cleanup report 并存。
- 一些历史版本号、旧部署方式和当前方式混在一起。

建议:

- 增加 `docs/README.md` 作为文档入口。
- 将“当前事实”和“历史事故”分开。

#### 12. 测试有数量，但缺统一运行命令和产品级 E2E

观察:

- MR.ERP、OCR、i18n 有不少测试。
- 许多 integration 依赖真实 MRERP 环境变量。
- 没看到 `.github` CI。
- `TECH_DEBT.md` 明确提到 UI 自动化测试不足。

建议:

- 建立三层测试:
  - `unit`: 不联网、不连 DB。
  - `integration`: 需要测试 DB 或外部沙箱，默认 skip。
  - `e2e`: Playwright 登录、上传、对账、充值、admin 核心流程。

## 8. 分阶段改造路线

### 第 0 阶段: 安全和基线，不重构

目标: 保证以后改动不会越修越乱。

建议任务:

1. `/internal/deploy` webhook 改成 secret 缺失即拒绝，并确认其它内部部署辅助端点保持现有 fail-closed。
2. `/api/version` 移除公开 traceback 和内部诊断。
3. 全局异常处理改成用户友好错误码，不回传内部异常文本。
4. 补完整 `.env.example`，列出 `DATABASE_URL`, `JWT_SECRET`, `GEMINI_API_KEY`, `PEARNLY_KMS_KEY`, `GITHUB_WEBHOOK_SECRET`, `LINE_*`, `SMTP_*`, `XERO_*` 等。
5. 让本地 `scripts/check_imports.py --quiet` 通过。
6. 建最小 CI: import check + i18n check +核心 unit tests。

### 第 1 阶段: 先拆后端路由边界

目标: 不改业务行为，只改变文件组织。

建议顺序:

1. 抽出 `admin_routes.py`。
2. 抽出 `billing_routes.py`。
3. 抽出 `erp_routes.py`。
4. 抽出 `history_routes.py`。
5. 保持 API path 不变，只移动代码。

每次只拆一个 domain，拆完跑 import check 和相关 tests。

### 第 2 阶段: 正式数据库迁移体系

目标: 停止无限增加启动时 `ensure_*`。

建议:

1. 盘点 `db.py` 所有 `ensure_*`。
2. 标记哪些已经在线上执行过，哪些仍需要。
3. 引入 Alembic 或至少自建 `migrations/` 版本表。
4. 新 schema 变更不再直接塞进 `lifespan`。

### 第 3 阶段: 多租户和权限验收

目标: 证明不会串数据。

建议:

1. 写 tenant access matrix。
2. 选 10 个核心表做隔离测试:
   - OCR history
   - clients
   - ERP endpoints
   - ERP logs
   - credit transactions
   - topup requests
   - vat recon tasks
   - bank recon tasks
   - employees
   - notifications
3. 对 owner / member / super admin 分别跑读写测试。

### 第 4 阶段: 前端绞杀式拆分

目标: `home.js` 不再增长，逐步变小。

建议顺序:

1. billing / topup 独立。
2. dashboard 独立。
3. history 独立。
4. clients 独立。
5. recon 独立。
6. OCR upload 独立。

每拆一个模块，配一个 Playwright smoke。

### 第 5 阶段: 产品级 E2E

目标: 上线前知道核心流程没断。

最低 E2E 路线:

1. 注册 / 登录 / 退出。
2. 新用户余额为 0 时上传被 402 拦截。
3. 管理员充值审批后余额变化。
4. 上传发票识别成功。
5. 历史记录查看 / 编辑 / 删除。
6. 收入对账上传、完成、导出。
7. 银行对账上传、完成、导出。
8. MR.ERP endpoint 创建、测试连接、推送日志展示。
9. 超管后台用户列表、充值列表、成本监控。

## 12. 下一步建议

建议下一轮不要直接重构，而是做“第 0 阶段安全和基线”:

1. 先修 3 个安全点: deploy fail-closed、version 诊断鉴权、异常返回脱敏。
2. 补 `.env.example`。
3. 建 CI 最小流水线。
4. 跑一次只读 + 最小测试基线。

等这四件事完成后，再开始拆路由和前端模块。这样每一步都有保险，项目会越来越像标准 SaaS，而不是换一种方式继续堆。
