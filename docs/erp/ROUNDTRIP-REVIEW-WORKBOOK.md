# ERP 复核工作簿与回导闭环

> 2026-07-23 上线。会计拿到一批推完 ERP 的料之后,靠这个闭环核对、纠错、重录。

## 会计的真实动作(设计的出发点)

1. 一批发票推进 Express
2. 导出复核工作簿,在 Excel 里逐行核对
3. **把核对无误的行删掉**,只留要重做的
4. 在表里改正错的行(最常见的是改票号、改金额、改分类)
5. 去 Express 把错单删掉
6. 把改过的表格丢回 Pearnly 收料口,重录一次

第 6 步以前是断的 —— 收料口读得进 xlsx,但把网格丢给为扫描件写的 L2 提示词去猜,
它不知道那是「多单据、多行、单据级字段跨行重复」的结构。

## 四张表

| Sheet | 装什么 | 粒度 | 列位合同 |
|---|---|---|---|
| `ขาย` | 销项 | 每行一个商品行 | 前 12 列 = 与 MR.ERP(Korn)约定的公式合同,**一格不能动** |
| `ซื้อ` | 进项 | 每行一张票 | 沿用 ภ.พ.30 进项税明细的法定列 |
| `รอจำแนก` | 方向判不出 / 被闸拦下 + 原因 | 每行一张票 | — |
| `สรุป` | 本批新建的商品/客户/供应商 + 码 | — | 只读,不回导 |

**粒度差异是本质的,别硬捏平**:销项要拆到商品行(Express 靠它动库存、算移动均价成本),
进项是整张费用票进一个科目。压进一张表必有一边要造假行,而且同一列会一词多义
(客户 vs 供应商、销项税 vs 进项税),会计扫一眼就会读反。

**空表也写表头**:少一张表,会计分不清「没有待判的」和「这版没这功能」。

## 核心机制:方向由「行在哪张 Sheet」决定

不从内容猜。会计把一行从 `ขาย` 剪到 `ซื้อ`,就是在说「你们分错了」。

回导时 Sheet 名 → `fields.direction` → `express_push/direction.explicit_direction()`,
它的优先级**高于**税号自动判定。会计不用学新概念、不用填方向字段,**挪一行 = 改一次分类**。

留在 `รอจำแนก` 的 = 还没裁决:带出来标 `needs_manual_review` + 原因落 `notes`,
**不推**(推了就是替会计做方向决定)、**不丢**(丢了就是漏票)。

## 回导列(追加在各自合同列之后,不插队)

| 列 | 为什么必须有 |
|---|---|
| `เลขประจำตัวผู้เสียภาษีคู่ค้า` | 销项 12 列合同里没有税号,而 ภ.พ.30 要报;**挪行时它是唯一还带着税号的地方** |
| `เลขที่เอกสาร ERP` | 会计要去 ERP 删单。票面号超 12 字节会退回自编号,那时票号列 ≠ ERP 单号 |
| `รหัสสินค้า ERP` | 清理误建的商品档要码 |
| `รหัสคู่ค้า ERP` | 同上 |
| `สถานะการส่ง` | 不给的话每行长得一样,会计可能去删一张压根没建成的单 |
| `รหัสอ้างอิงระบบ` | 回导键 `PRN:<history_id>:<行序>`。**不含票号** —— 改票号正是这个流程的主要目的 |

## 几个容易再踩的坑

**公式没有缓存值。** openpyxl 写 `=E2*F2` 时不带计算结果,它自己也不算。原样导出的文件
直接回读,`data_only=True` 拿到的是 `None` —— 金额列全空。解法是**读侧按合同重算**
(`G/H/I = 数量×单价`、`J = 税前×7%`),会计把公式改成死值时死值优先。
不要改成写死值:那会破坏 Korn 要的「改数量单价自动重算」。

**字段名两套。** OCR 侧(ThaiInvoice)是 `subtotal`/`vat`,回导解析侧是
`amount_before_vat`/`vat_amount`。两条来源都会喂进同一张表,**只认一套键就会漏一半**
(真跑时进项表税前/税额两列全空,就是这个)。

**防重单的钥匙是原 history_id,不是当前的。** 回导时收料口会为上传的工作簿**新建**一条
history 记录 —— 拿新 id 去查上一版必然查不到,闸正好在它唯一存在理由的场景下哑火。
行键里带回的原 id 才对。

**删单不删档。** 在 Express 里删掉单据**不会**删掉我们为它新建的商品/客户档。若某行之所以
要重做正是因为品名读错建错了档,删单后垃圾档还在,改名回导又建一个,主档只增不减。
`สรุป` 表就是那份清理账 —— 不自动删(Express 里被引用过的档删不掉,硬删会伤账)。

## 已知债(2026-07-23 /simplify 四路审查列出、本轮**未做**的)

按优先级,后来人接手时照这个顺序打:

1. ~~写读两套四舍五入~~ **已修**(2026-07-23):钱全走 `services/excel/erp_money.py`,
   Decimal + 与推送路同一个 `_q`(两位 · ROUND_HALF_EVEN)。票面税额原样带走
   (回导块的「单据税额」列),只有税基真变了才重算。
2. ~~公式列写死了列字母~~ **已加闸**:`ColumnLetterDriftGate` 取公式实际指向的列、
   断言它就是「数量」「单价」所在列,插列立刻红。公式本身仍写字母 —— 根治要做列规格表,
   但现在至少不会静默错位。
3. **会计手打的自定义公式会被静默抹掉**。`_num` 把任何 `=` 开头的串一律当"没有值"
   按合同重算。设计立场说"死值优先,那是他的裁决",但只尊重"改成字面量"、不尊重
   "改成别的公式"。**该做**:我们没写过的公式 → 落 `needs_manual_review` + 原因,
   与待判票的处理口径一致(不静默丢)。
4. **`_REVIEW_WORKBOOK_ADAPTERS` 硬编码白名单**在 routes 层,是全仓第四个 adapter 知识点,
   且唯一落在 routes 的。而且它**没守住它声称守的东西** —— `HEADERS_TH` 无条件带回导列、
   单表导出也已用 `SHEET_SALES` 表名,所以非 express 的默认导出同样能被回导解析器认出。
   它真正区分的是「四表 vs 单表」,不是「哪个 ERP」。**该做**:抽
   `build_export_workbook(adapter, buckets, created) -> (bytes, filename)`,注册表放
   `services/`,最好挂到既有 `ADAPTER_REGISTRY` 上,新增 ERP = 加一行。
5. **`prior_docnum` 是 N+1**。挂在两个 mapper 的 return 上,每张票一次 SELECT;
   而这条链已有批级预取座位 `preflight.build_batch_prefetch`(注释原文就是"去 N+1")。
   更彻底的做法:只有记录带回导键/`erp_docnum`(确实是回导来的)才查,首推批次零查询。
6. **`xlsx_style` 只接了一半**。`excel_template_th` 仍自留 12 个同值样式常量,
   导致 `fill_new/fill_reused/DATE_FMT/QTY_FMT/TEXT_FMT` 五个导出零调用 —— 模块建了,
   漂的风险没消。
7. **`_mirror_history_status` 的 except 兜不住它想兜的**。UPDATE 一旦在 PG 侧报错,
   整个事务进 aborted 态,后续同事务语句全挂。注释说的「镜像失败不该回滚 ack 本身」
   需要 `SAVEPOINT` 才成立。
8. **`prior_docnum` 用 `db.get_cursor()` 无 user scope**,WHERE 里也没有 `user_id` —— 跨租户可读。
9. **存量 N+1(非本次引入)**:`ocr_export_routes` 里 `db.get_ocr_history_detail` 在循环里,
   导出 50 张 = 50 次往返。要修是给 `services/ocr_history/queries.py` 加批量版。
10. `services/excel/excel_export.py`、`services/fileconv/xlsx_out.py` 是同色系样式的
    第 3、4 份历史副本,属批次外历史债。
