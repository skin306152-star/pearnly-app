# Pearnly 设计语言规范(2026-06-08 Zihao 钉死 · 未来改/新增 UI 一律照此)

> 出处:做账 Phase 2 设计期 Zihao 拍板。**这是全站 UI 的构图/视觉基准 —— 任何窗口改 UI、加新模块、画新屏,先读这页,照此做。不照 = 不合规,返工。**
> 关系:本页管【构图与视觉语言】(布局/层次/容器/调色/间距);既有 `check_ui_consistency`(按钮蓝 `#2563EB` / `.modal` 非抽屉 / D1·D2)管【组件级硬闸】。两者叠加。
> 参照实现:`桌面/做账_主屏_收拢版.html`(收拢+分层标杆)、`桌面/设计语言对比_进项主屏_after.html`。

## 一、六条构图原则(每屏都守)

1. **收拢,不悬空**:元素收进**有边界的面板/容器**,不让按钮/卡片各自飘在空白灰底上。一个屏 = 一个(或少数)完整面板,不是一堆碎片浮着。(Zihao 原话:"要么都在一个大框里,分主次,别单独长在一个地方"。)
2. **分层,定主次**:面板内自上而下分**三层** —— ① 主信息带(北极星数 + 最该处理的动作)② 控件带(筛选/月份/导出,浅灰底区分)③ 内容(列表/表单/凭证)。主次靠位置 + 字号 + 底色,不靠堆框。
3. **克制,渐进披露**:**不每屏甩一排等权 KPI 卡**(那是后台模板味)。先放**一个最要紧的数(北极星)+ 一个该做的动作**,细节点进去看。对标 Stripe/Linear:"初看很静,点开很深"。
4. **内容当主角**:列表/表单是主角,给足空间和留白;统计/控件是配角,克制。
5. **行动卡,不堆被动统计**:该处理的事做成**行动卡**(待审/未付 + "去处理 →"),不是一排只读数字。Insight-first。
6. **动作有归属**:按钮收在它所属的容器里、右对齐(凭证的动作在凭证底、列表的工具在工具条),**绝不让按钮孤零零浮在大空白里**。

## 二、面板结构(标准骨架 · 列表型屏照搬)

```
标题 + 副标(灰底上,面板外)
┌─ 面板(白 · 圆角16 · 柔阴影) ─────────────────────────────┐
│ ① 主信息带   北极星数(左,大)        行动卡(右:待审/待办→)  │  ← 主
│ ──────────────────────────────────────────────────────  │
│ ② 控件带     筛选标签(左)      月份 ▾ · 导出/主操作(右)    │  ← 次 · 浅灰底
│ ──────────────────────────────────────────────────────  │
│ ③ 内容       行 / 行内展开 / 表格 …                       │  ← 内容
└─────────────────────────────────────────────────────────┘
```
- 表单型屏(如录入):面板内分"票图/凭据 + 信息/明细/合计"分区,同样收拢分层。
- 弹窗:桌面居中 `.modal` / 手机底部抽屉(既有规范)。

## 四、按钮 / 卡片规矩

- **按钮**:主 = 蓝实心(`--blue`,白字,半粗);次 = 白底描边;危险 = 红字。高 32–40,圆角 9–11。
- **永不悬空(含页面主操作)**:按钮/小卡收进所属容器,动作右对齐。**连标题旁的主操作按钮(如「拍进项票」)也收进面板顶部带**(北极星左 · 按钮右),**标题外不留任何悬浮元素**(Zihao 2026-06-08 明确)。月份+导出这类要**编组贴在一起**(有归属),不单独飘。
- **禁原生弹窗**:`prompt()`/`alert()`/`confirm()` 一律禁(丑 + 违 `.modal` 铁律)。加科目/确认等用站内 `.modal`(桌面居中 / 手机底抽屉)。
- **行动卡**:浅琥珀底 + 边,内含"标签 + 数 + 去处理→",收在主信息带里,不浮。
- **去 AI 字样**:文案不露「AI/人工智能」,用智能/自动/系统(见 [[no-ai-wording-in-ui]])。

## 四-bis、交互完整性(★封板必查 · 防"边角设计不全"反复发生)

> 2026-06-08 Zihao 追问根因后加。痛点:屏画齐了,但**动作的反馈、字段的空态**没设计 → 作废没提示/LINE费用字段空白/待归类无入口…边角反复漏。根治=把"边角"做成机械清单,封板时逐项过,不靠记性。

1. **每个动作必有三态反馈**:任何按钮/操作(保存/作废/删除/付款/匹配/入账/生成…)都必须设计 **① 成功**(toast/inline + 状态变化文案,如"已作废 · 库存已回冲")**② 失败**(具体错误 + 下一步)**③ 进行中**(禁连点/loading)。**绝不允许"点了没反应"或"静默改变"。**
2. **每个字段必有空态兜底**:任何展示字段缺值时,要么有合理占位("—"且有语义)、要么"缺则不显"、要么标"待补"——**绝不静默空白让人以为坏了**。数据本应有却空(如 LINE 费用没供应商)= 上游数据缺口,要么补取、要么显式提示"未识别·去补全",不留谜之空白。
3. **每个入口必有落点**:设计的每条数据通路(如 LINE→待归类)必须有**用户看得到、点得到**的 UI 落点;后端落了库、前端没入口 = 不算完成。
4. **删除/作废/不可逆动作**:必有二次确认 + 成功反馈 + 连带影响说明(回冲/级联)。

**封板清单(每模块过一遍,缺一项=没设计完)**:六类(操作面/配置后台/接入/四态/权限/联动)× 本节四条(动作三态/字段空态/入口落点/不可逆确认)。**实测文案必含"逐个点每个动作看反馈、每个字段看空态"**,不只跑 happy path。

## 五、双端

桌面 + 手机**都做**,非只手机、非缩水。桌面用两栏/工具条铺开;手机竖向堆叠、弹窗转底抽屉。同一份设计两端各自最优。
