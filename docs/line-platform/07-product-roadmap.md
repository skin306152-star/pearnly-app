## 2. 分期(先做透核心,再层层加)

### Phase A · 锚定核心(进行中)——"精准命令"地基
Slice 2b(草稿软删+恢复)→ Slice 3(强锚定:引用即围绕·不落新单)→ Slice 4(无引用置信度猜+列候选问+pending继承)。
**做完=引用谁只动谁,没引用就带置信度猜、猜不准就列着问。** 一切高级玩法的前提。

### Phase B · 单用户"变聪明"——建在 A 之上
1. **学习按钮(本次/以后供应商/这个客户)**——把已有学习引擎做成三档按钮,沉淀习惯。低风险高回报,**B 首推**。
2. **卡片状态机**——统一票据生命周期(收到→识别中→需确认→已入账→已撤销→失败待修),每次操作发新态卡引用旧卡。
3. **异常=会计待办卡**——"识别失败"改成"需确认3件事"+按钮(补税号/标普通收据/打开详情/问会计)。
4. **单任务 LIFF 窄页**——补税号/改分类/拆项/改VAT/看原图,只开这张这一字段。

### Phase C · 会计价值 + 留存
5. **月底交接包**(Mandarin Foods·2026-06:94票/52VAT/7异常/87已确认 + [打开包][下载Excel][分享给会计] LIFF share target)——**会计定位的杀手锏**。
6. **月底自动催缺票**(25号扫缺票/缺税号/未确认 → 推待办卡)。
7. **费用问答富回执 + 续问**(油费18,420/12票/最大4,200 + 续问"为什么高"沿用上下文)。

### Phase D · 多人平台(gated · 单用户车道优秀后再开)
8. **群组收票 + @Pearnly 响应 + 私聊/LIFF 审批**(群里轻反馈,高风险动作进私聊)。
9. **动态 Rich Menu**(老板/会计/员工/viewer · 平时/月底/异常)。

## 3. 不做(= 既有红线 · 保持锁死)
聊天里:报税 / 银行对账确认 / 反过账 / 改公司税号 / AI 自动批量删票 / 完整 ERP 推送审批。
LINE 只做:收集 / 确认 / 纠错 / 追问 / 提醒 / 交接 / 轻审批 / 打开精确 LIFF。**不把整个 Web 塞进聊天。**

## 4. PM 一句话
**Phase A 是地基(在做),Phase B 让它"聪明"(学习按钮先行),Phase C 让会计离不开(交接包),Phase D 才开多人/群。**
不追炫技、不提前开车道;每加一片都受 06 锚定框架 + 账务红线约束。
