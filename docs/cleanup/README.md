## 核心原则:先看清,再保护,最后才动

屎山清理是高风险操作,做不好会比新增功能引入更多 bug。所以策略上必须非常克制。

整体流程:
1. **第 1 段** — 让它做"代码健康体检",不动任何代码
2. **第 2 段** — 建立"安全网"(冒烟测试)
3. **第 3 段** — 只清最安全的 🟢 项
4. **第 4 段(可选)** — 逐项讨论处理 🟡 项

## 📋 第 1 段:让它先做"屎山体检",不动任何代码

新开一个 Claude Code 窗口,发这段:

```
我接下来要做一次代码清理重构,但在动手之前,我要你先做一份完整的"代码健康体检报告"。这一轮严禁修改任何代码,只读不写。

## 背景
项目积累了一些"屎山"(死代码、重复逻辑、过时实现、TODO 没清等),我想在不影响功能的前提下系统性清理。

## 第一步:扫描全项目,识别问题代码
按以下分类找出可疑代码,每一项都给出具体文件路径和行号:

1. **死代码 (Dead Code)**
   - 定义但从未被调用的函数、类、方法
   - 永远不会被执行的分支(unreachable code)
   - 注释掉的大段代码
   - import 了但没使用的模块

2. **重复代码 (Duplication)**
   - 几乎一样的函数/方法存在多份
   - 复制粘贴的代码块(超过 10 行的雷同)
   - 同样功能用了多种实现方式

3. **过时代码 (Obsolete)**
   - 标注了 TODO/FIXME/HACK/XXX 的地方,列出所有并标注年龄
   - 标注了 @deprecated 或注释里写"旧版本""废弃"的代码
   - 引用了已经不存在的功能或老接口

4. **可疑代码 (Suspicious)**
   - 空的 try/except(吞掉所有错误)
   - 写死的测试数据/魔法数字
   - 没有任何注释的复杂函数(> 50 行且无文档)
   - 命名极其混乱的变量/函数(a, b, tmp, data2)

5. **未使用的依赖**
   - package.json / requirements.txt / pyproject.toml 里列出但代码中没用到的包

## 第二步:分级
对每一项标注风险等级:
- 🟢 **安全清理**:绝对没人在用,删了没影响(如未使用的 import)
- 🟡 **需要确认**:看起来没在用,但可能有隐藏依赖(如反射调用、动态导入)
- 🔴 **谨慎处理**:涉及核心业务逻辑,删之前必须我确认

## 第三步:输出清单
保存为 `docs/code-cleanup-audit.md`,包含:
- 总览(共发现多少处问题,按等级分布)
- 详细清单(每一项的位置、类型、风险等级、建议处理方式)
- **不要做任何排序或优先级建议,我自己决定**

## 严格规则
- ❌ 不要修改任何代码文件
- ❌ 不要删任何东西
- ❌ 不要"顺手"做任何重构
- ✅ 只读、分析、写文档
- ✅ 不确定的全部标 🟡 让我决定

完成后告诉我文档路径,等我审阅。
```

## 📋 第 2 段:建立"安全网",这是清理前必须做的

它给完体检报告之后,**不要急着开始删**,先发这段:

```
我看完了 `docs/code-cleanup-audit.md`。在动手清理之前,我需要先建立"安全网",确保清理过程不会破坏现有功能。

## 第一步:盘点现有的测试覆盖
检查项目现有的测试情况:
- 有哪些测试文件?用的什么框架(pytest/jest/etc.)?
- 怎么运行测试?给我准确的命令
- 当前测试覆盖率大概是多少?哪些模块没有测试?
- 把这个写到 `docs/test-coverage-current.md`

## 第二步:识别"核心路径"
我要你列出产品的**核心用户路径**(critical user paths)——也就是这些功能挂了用户立刻会发现的:
- 比如:用户注册登录、上传文件、OCR 识别、查看结果、导出数据等
- 每一条路径,标注涉及的代码模块/文件
- 写到 `docs/critical-paths.md`

## 第三步:为核心路径补"冒烟测试"
针对每条核心路径,**新增**一个最简单的端到端冒烟测试(smoke test):
- 不要写得很全面,只要能验证"这条路径没断"就行
- 放在 `tests/smoke/` 目录,和现有测试隔离
- 文件命名:`test_smoke_<path_name>.py`(按项目语言调整)
- 给我一个统一的运行命令,比如 `pytest tests/smoke/`,几分钟内能跑完所有冒烟测试

## 严格规则
- ❌ 不要修改任何业务代码
- ❌ 不要清理任何"屎山"(那是下一阶段的事)
- ✅ 只新增测试文件
- ✅ 测试要简单,关注"能跑通"而非"覆盖率"

## 为什么要做这一步
清理屎山有风险,我需要在动手前有一套快速验证"功能没坏"的机制。每次清理一批后,跑一次冒烟测试,绿了就继续,红了就回退。

完成后告诉我:
- 创建了哪些冒烟测试
- 怎么一键运行所有冒烟测试
- 当前所有冒烟测试是否都能通过(必须全绿才进入下一阶段)
```

---

## 📋 第 3 段:开始清理,但只清最安全的那一层

冒烟测试全绿之后,才发这段:

```
冒烟测试全绿,我们可以开始清理了。**但只清最安全的那一层**,绝对不碰 🟡 和 🔴 的内容。

## 本轮任务:只处理 🟢 安全清理项
从 `docs/code-cleanup-audit.md` 里挑出所有 🟢 标记的项,按以下顺序处理:

1. 删除未使用的 import
2. 删除注释掉的代码块
3. 删除从未被调用的私有函数(以 _ 开头且全局搜索零引用的)
4. 删除未使用的依赖(package.json / requirements.txt)

## 强制工作流
**每完成一个文件就停下来汇报**,等我确认后再继续下一个。不要批量改一堆。

每次汇报包含:
- 改了哪个文件
- 删了什么(具体内容,不是"删了一些 import")
- 跑冒烟测试结果(必须全绿才汇报"完成")

## 任何不确定的立刻停
如果你不能 100% 确定某段代码可以安全删除:
- ❌ 不要删
- ✅ 升级到 🟡,加进"待确认清单"
- ✅ 继续下一项

## 严格规则
- ❌ 不要"顺便重构"
- ❌ 不要"改进命名"
- ❌ 不要碰 🟡 或 🔴
- ❌ 不要修改任何函数的逻辑
- ✅ 只做删除操作
- ✅ 每改一个文件跑一次冒烟测试

完成 🟢 阶段后,生成 `docs/cleanup-round1-report.md`,汇总:
- 删了多少行代码
- 改了哪些文件
- 冒烟测试一直全绿吗
```

---

## 📋 第 4 段(可选):处理 🟡 项,但要逐个讨论

🟢 都清干净后,如果你还想继续清 🟡(中等风险),发这段:

```
🟢 全部清理完成,冒烟测试全绿。现在进入 🟡 项处理阶段。

## 工作方式改变
🟡 的项目我不再批量处理,而是**逐项讨论**:

1. 你从 `docs/code-cleanup-audit.md` 里挑一项 🟡
2. 详细给我看:
   - 这段代码做什么的
   - 你为什么觉得它可疑/可以删
   - 你做了哪些搜索来确认它真的没人用(具体的 grep / 搜索关键词)
   - 你不确定的地方是什么
3. **等我决定**:删、保留、还是改成别的
4. 我说"动手"你才动,动完跑冒烟测试,绿了再处理下一项

## 严格规则
- ❌ 一次只处理一项
- ❌ 不要自己做决定
- ❌ 不要批量处理
- ✅ 每一项都要"先讨论,后执行"

从第一项开始。
```

---

## 🔴 关于 🔴 项(高风险),强烈建议

**不要在这个清理窗口里处理**。

🔴 项往往是"屎山的核心"——它丑,但它在工作。这类代码的处理应该:
1. 等新架构(Vision + Gemini 混合)上线并稳定后
2. 业务上确认旧逻辑不再需要后
3. 才在功能迁移的过程中**顺便替换掉**,而不是单独清理

单独清理 🔴 项,风险高、收益低、容易出事。

---

## 💡 让两条线(清理 + OCR 迁移)不冲突的关键

如果你同时有两个 Claude Code 窗口在工作:
- **窗口 A**(OCR 迁移):见 `docs/pearnly-ocr-architecture-recommendation.md`
- **窗口 B**(屎山清理):本文档

### 1. 告诉两边对方在做什么

**在窗口 A(OCR 迁移)的第一条消息里加一句**:
> "注意:有另一个会话正在做代码清理工作,会修改一些 import、删除死代码。如果你看到代码状态和你之前记录的不一致,以最新代码为准,先重新扫描一遍。"

**在窗口 B(屎山清理)的第一条消息里加一句**:
> "注意:有另一个会话正在为 OCR 模块做新架构迁移,会新增 `services/ocr/vision_layer.py` 等文件。**这些新文件你完全不要碰**,只清理现有的旧代码。"

### 2. 用 Git 分支隔离

- 屎山清理用一个分支:`chore/cleanup`
- OCR 迁移用另一个分支:`feat/ocr-vision-migration`
- 清理完一轮就合并回主干,迁移那边再 pull

### 3. 每天结束时同步一次

每天工作结束,把两边的进度小结互相通报一次,避免相同代码被两边同时改。

## 使用流程总览

```
开始
  ↓
[第 1 段] 体检 → 生成 code-cleanup-audit.md
  ↓
人工审阅文档,确认分级合理
  ↓
[第 2 段] 建安全网 → 生成 critical-paths.md + 冒烟测试
  ↓
确认冒烟测试全绿
  ↓
[第 3 段] 清理 🟢 项 → 每个文件单独汇报
  ↓
冒烟测试全程保持绿色,完成后生成 cleanup-round1-report.md
  ↓
(可选)[第 4 段] 逐项讨论清理 🟡 项
  ↓
🔴 项暂不处理,等 OCR 新架构上线后随业务迁移自然消化
  ↓
结束
```

---

## 关键安全机制(让 Claude Code 始终遵守)

无论哪一段,以下原则**任何时候都不能违反**:

1. **每个操作都要有"先汇报后执行"的环节** — 不要让它一次改一堆
2. **冒烟测试必须始终保持全绿** — 红了立刻停,回滚,排查
3. **不确定的一律不动** — 升级到 🟡 队列等讨论
4. **Git 提交粒度要细** — 每个清理项一个 commit,出问题方便定位回滚
5. **绝不"顺便"做事** — 只做明确批准的任务,看到其他问题写进文档不要动手

如果 Claude Code 在执行中违反了上述任何一条,**立刻打断它**:

> "停下,你违反了我设定的规则。当前任务只允许 [具体允许的操作]。把你刚才超出范围的修改回退,重新开始。"
