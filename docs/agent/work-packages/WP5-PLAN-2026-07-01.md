新插座 M1 栈已落地且 14 单测全绿:

**两个真实集成缺口(WP5 必须收口,WP1/WP3 交界没对齐):**

**缺口 A · 渲染未接线。** `copy_map._render` 返回 `key|slots` 占位串;LINE 文案实际走 **Python 端 `line_i18n.py`**,而 WP3 把 `agent.*` 4 语落在**前端 `static/i18n-data.js`**(浏览器侧,LINE 运行时根本不读)。→ 现在把 `handle_turn` 的返回直接发给用户,会看到 `agent.ok.history|count=5` 这种乱码。WP5 要补一个 **Python 端 agent.* 渲染器**。

**缺口 B · 槽位名对不上。** copy_map 提供的槽位 ≠ i18n 模板声明的槽位。例:`agent.ok.history` 模板=`本月识别 {count} 张·合计 {total}฿{top_list}`,但 `history_receipt` 只给 `count=total(条数), shown=...`,缺 `total(金额)`/`top_list`,且 `shown` 模板里没有。直接 `str.format` 会 `KeyError` 崩。

## 2. 落点(★ 能力只增不减 · MASTER-PLAN 铁律 `a0fce733`)

**新 loop 是纯加法层,不接管「理解」步**。它只处理**有工具的意图**(M1 = 5 个只读),其余一律 `defer` →
落回旧 `understand()+_dispatch_agent()` 照常完成。旧的撤销/改错/记账一个不丢;只有新旧 LINE 都不做的
(改密码/POS)才由旧路引导去 App。

`loop.handle_turn` 返回 `Optional[str]`:
- 工具成功 → 渲染回执(OWN,新路回)。
- 反问(brain 主动 ask / 必填槽被编造)→ ask 文案(OWN,守槽纪律,验收 #3)。
- **其余(out_of_scope / chat / 未知工具 / 工具失败)→ `None`(defer)**。

闸放在 `handle_expense_text` 的**大脑步**(现 L124「# 2. 大脑」之前):
- L42–L122 确定性路径(闲聊/收入/多笔/记账/缺金额 pending/改错路由)对所有人原样保留。
- 灰度命中 + 余额够 → 先试 `loop.handle_turn`;返非 None → 渲染 + `_charge_line_l2` + 回,return True。
  返 None(defer)→ **一行不改**,跌落旧 `understand()+_dispatch_agent()` 照常跑(撤销/改错/记账全在这)。
- 关/未命中 / 余额不够 / Agent 抛错 → 逐字节走旧路(try/except 兜底:Agent 任何异常 = defer,不报错给用户)。

计费对齐:仅当新 loop **OWN(返非 None)**才 `_charge_line_l2` 扣 1;defer 时由旧路自己计费 → 不双扣。
(已知 M1 成本:defer 的轮次会多一次大脑探测调用 backend;M2 把 understand 并进 Agent 后消除。)
