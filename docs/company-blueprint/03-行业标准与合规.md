# 平台总纲 · 03 行业标准与合规

> 骨架#3/#6:合规即产品;行业版=同一产品+预置配置,绝不 fork。
> ⚠️关键裁剪:金蝶的合规获客靠金税四期**强制大迁移**(deadline 驱动);泰国 e-Tax 推进慢、非强制 → **在泰国,合规是留存与信任的护城河,不是获客钩子**。获客靠"省事"(2.0 的不用录),续费靠"合规跟着 RD 走不用操心"。

## 国家标准对齐清单(Pearnly 的"金税工程"=RD)

| 标准 | 内容 | 现状 | 标准定为 |
|---|---|---|---|
| **e-Tax Invoice & e-Receipt** | RD 电子税票直报 | Phase3 图纸已封板(docs/tax-filing,e-Tax 先确认 RD 开放度、兜底导出手报) | 直连是目标形态;RD 没开放的部分,出"可直接上传/提交的合规文件包"过渡 |
| **PP30 / PND 3·53** | VAT 月报/预扣税申报 | 图纸封板:从做账账本自动算表+报前体检拦错 | "不用编表"是底线,所有税表必须从账本派生,禁止让用户填表 |
| **税票合规要素** | 连号/正副本/佛历/总分公司/税号 | 已上线(连号按主体·RD 合规) | 凡涉单据出口的新功能,合规要素 checklist 过一遍才算设计完 |
| **TFRS for NPAEs** | 泰国中小企业会计准则 | 做账图纸的泰标科目表预置 | 科目表/报表口径以 NPAEs 为默认,大企业准则后置 |
| **PDPA** | 泰国个资法 | 零散(密码哈希/审计日志有;无数据处理清单、无删除权流程) | 补:个人数据清单、保留期策略、用户数据删除/导出流程(F 层事务所代持客户数据,PDPA 责任更重,伙伴协议要写清角色) |
| **本地化基线** | 佛历/泰文 PDF/泰语 OCR/四语 | 🟢 已扎实落地 | 新功能默认四语+佛历,已是家规 |

合规运营机制(学金蝶"政策更新=订阅价值"):RD 改规则(税率/表式/起征点)→ 我们改产品 → 用户无感继续合规。**把"合规更新"做成对外可见的更新日志**,让客户知道订阅费买的是"永远跟得上 RD"。

## 行业模板升格(业态预设从"开关"到"全套预置")

现状(🟡 做浅了):6 业态(firm/retail/pharmacy/restaurant/service/b2b)只控制模块开关(services/modules/presets.py),科目表、单据样式、税务默认值都不跟业态走。

升格标准 —— 选业态一键预置**五件套**(金蝶行业方案的同构缩小版):

| 预置维度 | 例:餐厅 | 例:药房 |
|---|---|---|
| 模块开关 | POS+库存+销售(已有) | 同左 |
| POS 业态流 | 桌台/KOT/服务费(已有) | 批次/近效期(已有) |
| 科目表子集 | 餐饮常用科目+服务费科目 | 药品库存科目 |
| 单据/小票默认 | 含服务费行/VAT 价内 | 标准 |
| 税务默认 | VAT 价内、服务费计税规则 | VAT 标准 |

- 实现原则:**预置=数据不=代码**。presets.py 从"模块列表"扩成"模板包"(模块+科目种子+默认设置),apply_preset 一次落库;改业态=重套模板包(已有数据不动)。
- **绝不 fork**:餐厅版/药房版永远是同一个 Pearlny + 不同模板包(金蝶餐饮云仍是星空产品线;我们 POS 业态分流已是同构,坚持)。
- 新业态扩张(美容/汽修/建筑…)=加一个模板包,不加代码分支 —— 这是"泰国金蝶"扩行业的标准动作。
