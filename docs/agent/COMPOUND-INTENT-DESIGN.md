# 复合意图(一句话记账+查询)· 方案(待拍板)

> 2026-07-03 真机坐实(163 语音「กาแฟ 50 บาท เดือนนี้กี่ใบ」):模型不调工具只口头应承、
> 回复还截断(turn log `tool_trace=[]`)。打字同踩,非语音引入。钱面无风险(没错记),
> 纯体验缺口:两个意图只能干成零或一件。

## 根因(两层)

1. **结构**:loop 的写工具落地即终轮(`spec.writes → write_sink → return card_sent`),
   记账卡就是本轮回复 —— 就算模型想"先记后答",查询部分也没有出口。
2. **行为**:模型面对复合句倾向绕开工具用散文应承(知道自己一步做不完),散文超长被
   截断/救援,体验最差的一种输出。

## 方案(结构+行为各修一半 · 闸 `agent_compound_turn` 默认关)

1. **写后续步(loop)**:写工具经 write_sink 出卡成功后**不再终轮**,把「已记账 ฿X」作为
   观测喂回,允许模型继续调只读工具回答剩余问题;最终文字走 **push**(reply_token 已被
   卡消费,LINE 支持后续 push——语音回显同款先例)。步数/预算上限沿用现有(_MAX_STEPS/
   _TURN_BUDGET),不加长。TurnResult 增载荷:`card_sent` 可附带 `text`(入口有则 push)。
2. **提示词一行**:decide 规则补「message 含记账+其他问题时:先调记账工具,出卡后继续
   回答其余问题;绝不用散文替代动作」。
3. **闸关=逐字节现状**:写落地即终轮的现行为完全保留;闸只在 write 成功后的分叉处判。

## 不改什么(风险边界)

- write_sink/确定性落地/金额接地/confirm-first 一行不动——写的"手"不变,只是写完
  不掐断轮次;模型仍无直接写库能力。
- 多笔预判(record_multi)直分发路不动;推 ERP(confirm 卡)后同样不续步(等确认是终态)。

## 验收

- 单测:闸关现状不变/记账+查询→卡+push 答案两条都到/纯记账不多话/预算见底只出卡。
- 真机(163):语音与打字各发「กาแฟ 50 บาท เดือนนี้กี่ใบ」→ 记账卡 ฿50 + 一条"本月 X 张"
  的跟进消息;金丝雀 → 放量。
