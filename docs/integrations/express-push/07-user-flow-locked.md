# Express Push · 用户使用流程(已锁 · 2026-06-20 Owner 拍板)

> 这是 Express Push 的**用户体验准绳**。与 `01-ui-design-brief.md` / `03` / `06` 中关于"用户怎么用"的描述冲突时,**以本文为准**。技术实现服从本流程。

## 0. 铁律(Owner 定)
**不管用户选哪种录入方式,用户【完全不碰 Express】。** 用户只用 Pearnly。Express 的一切(开、登录、导航、录入、保存)由小助手自动完成。

## 1. 小助手(Agent)定位
- **两种录入方式都必须有小助手**(因 Express 在本地,云端够不着;不是 RPA 才要)。
- **一个点位装一个、装一次**(由管理员/IT 装)。**员工零安装、零接触 Express。**
- 小助手必须**全自动**:RPA 时它**自己启动 ExpressI.exe → 用 BIT9 登录 → 选账套 → 翻到采购单 → 填 → 存 → 读回**,全程无人;DBF 时连开都不用开,后台直接写文件。

## 2. 零用户校准(关键简化)
- Express 采购单界面固定 → **我们提前在真 Express 上把字段模板做好、随小助手一起发(出厂预制)**。
- 用户那边**没有"教它认界面/校准"这一步**。某台机 DPI/版本不同 → 小助手用 SIFT **静默自适应**,用户无感。
- 校准从"用户任务"变成"我们一次性开发任务"。

## 3. 两个使用场景
**场景 1 · 事务所 + 局域网**
- 小助手**只装一个**,在局域网里**一台常开机器/服务器**上(能访问 Express 数据/能跑 Express 的那台,如 `accserver`),管理员装一次。
- 所有员工:自己电脑上正常用 Pearnly 拍票/录入,**啥都不装、不碰 Express**。
- 票识别→自动排队→中央小助手搬进 Express→员工在 Pearnly 看「已推送 + 单号 + 分录」。

**场景 2 · 个人用户(自己电脑有 Express)**
- 小助手装在他**自己这台电脑**(与 Express、Pearnly 同机),装一次。
- 他只用 Pearnly,小助手后台推,**不手动开/碰 Express**。

## 4. 一次性设置(管理员/个人 · 不是"用 Express")
向导里填:Express 程序路径 + 登录(BIT9)+ 目标账套 + 录入方式 + 粘贴连接密钥。这是**配置小助手**,不是操作 Express。凭据本地加密存。

## 5. 录入方式取舍(都对用户透明)
| | 模拟录入 RPA | 直写 DBF |
|---|---|---|
| 宿主机要求 | 需保持**开机登录态**(GUI 活屏才能驱动) | 无需活屏/无需开 Express,**适合无人值守** |
| 安全 | 高(Express 自己算账过账) | 需**备份+借贷校验+失败回滚**护栏 |
| 默认建议 | 求稳用户 | 无人值守的事务所 |

## 6. 本流程对技术的影响(待施工方向调整)
1. **小助手要新增"自动会话管理"**:自动启动 Express + 登录 + 选账套 + 导航到采购单(companion 现假设 Express 已开,需补)。
2. **模板预制**:我们一次性在真 Express 上做采购页模板随包发,**不暴露校准给用户**。
3. **DBF(Track B)地位上升**:无人值守事务所首选。
4. 前端向导第 2 步"装 Agent"文案调成"一个点位装一个、零校准"。

## 7. 不变的现实
- 小助手仍必须装在**能碰到 Express 的那台机**(局域网一台 / 个人本机)。这是 Express 本地、云端够不着的物理约束,绕不开;但**对终端用户已完全透明**(他只用 Pearnly)。
- "做自动会话+模板"这件开发活,仍需在**真 Express**(Owner 机器)上做一次——但这是**我们一次性开发**,不是每个用户的负担。
