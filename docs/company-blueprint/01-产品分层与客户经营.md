# 平台总纲 · 01 产品分层与客户经营

> 骨架#1:分层产品矩阵,按层经营留存。金蝶依据:精斗云(微)/星辰(小)/星空(中)/苍穹星瀚(大)四层,FY2025 按层披露收入与 NDR(110%/97%/94%)——层不是市场话术,是经营单位。

## Pearnly 三层(同一代码库 · 套餐组合不 fork)

| 层 | 名 | 客户 | 套餐内容(复用 2.0 四档表) | 对标 |
|---|---|---|---|---|
| **S** | 小微商户 | 路边店/个体/freelancer | POS + 简单账 + 自动 VAT,一部手机+LINE | 精斗云 / FlowAccount Free·Standard |
| **M** | 成长企业 | SME/连锁/B2B | + 进项采购 + 库存 + 应收应付 + 做账 + 报税 + 多用户权限 + 审批 | 星辰·星空 / FlowAccount Pro / PEAK PRO+ |
| **F** | 事务所 | 会计事务所/代账 | 代多家(多套账)+ 批量识别 + 对账复核 + 客户工作台 + 喂 ERP | 账无忧 / PEAK 事务所计划 |

- 与 2.0 的"商户栈 vs 事务所栈"完全重合:S/M = 商户栈按规模切两档,F = 事务所栈。层 = `业态(business_type)× 模块开关(tenant_modules)× 计费档(04 章)`,三个既有机制的组合,**不新建产品线**。
- **升级 = 模块开关内完成,零迁移**。这是我们比金蝶强的地方(金蝶跨层=换系统+迁数据,官方"平滑长大"未确认);同一库长大是真卖点,写进营销话术。

## 🟡 纠正第一刀:tenant_type_v2

现状(摸底确认):`tenants.tenant_type_v2 DEFAULT 'firm'`(services/membership/schema.py:81)且全代码**零分支引用** → 99% 租户在数据里都是"事务所",字段既错又无用。

纠正方案:
1. 语义重定义:`tenant_type_v2 ∈ {'s_micro','m_business','f_firm'}` = **经营层**(本章的 S/M/F),不再与业态混用(业态在 `__business_type__` 哨兵行,各管各的)。
2. 存量回填:有真实业态的按映射回填(firm→f_firm;retail/pharmacy/restaurant→s_micro;service/b2b→m_business);没选过业态的置 NULL。
3. 默认值改 NULL;onboarding 选业态时同步落层(业态→默认层,可后调)。
4. 回填走 ensure 迁移 + 回填前后计数核对;改完挂一条机械闸:禁止新代码再读旧语义。

## 🔴 F 层核心缺口:事务所工作台

现状:代多家只有数据结构(workspace_clients 无上限 + client_assignments),**没有界面**——没有多客户汇总视图、没有跨客户批量操作、没有"本月哪几家该报税了"的台账。事务所今天在 Pearnly 里是"一个一个套账点进去"。

要建的(后续专窗口出图纸,这里只定标准):
- **客户总览**:全部代账客户一屏(本月票量/做账进度/报税状态/异常数),按截止日排序。
- **批量动作**:批量识别、批量出报税包、批量喂 ERP。
- **员工分工视图**:谁管哪几家(吃 `docs/permissions/` 的 member_scopes)。
- 这是 02 章渠道战略的前提:事务所工具不好用,就没有"事务所带客户进来"。

## 其它纠正/补缺

- **竞品导入工具(🟡)**:数据能导出(Excel/VAT 表)不能导入。FlowAccount/PEAK/Express 的客户想换过来要手搬 = 获客武器缺失。标准:每打一个竞品,先做它的导入器(主数据+期初余额);importer 框架已有,逐竞品补适配。
- **租户级运营工具(🟡)**:超管后台缺代开通/代配置(摸底确认无此类路由)。客服/运营帮客户开模块、调层,现在要动数据库。标准:超管后台补租户运营页(改层/改模块/改业态,全落 operation_logs)。

## 经营仪表盘(按层看,学金蝶把 NDR 当方向盘)

- 现阶段(credits 制):按层统计 活跃租户数 / credits 消耗 / 月留存,超管后台成本收入面板已有地基,加"按层分组"维度。
- 订阅化后(04 章):**MRR/ARR/NDR 按层披露**,每层单独看续费与扩张。S 层看量,M 层看 NDR,F 层看"每家事务所带来的下游客户数"(账无忧的北极星:1 家所 ≈ 百余家小微)。
