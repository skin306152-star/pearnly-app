# 平台总纲 · 06 研发与开发标准

> 骨架#5/#8:统一底座是收敛出来的,不是第一天建的;AI 做平台层下沉。这一章大部分是**确认已有制度有效并升格为公司标准**,新增的只有"底座清单保护"和"按金蝶教训防跑偏"两件。

## 第一原则:应用先行,平台从复用中长出来

金蝶教训(调研核实):苍穹 PaaS 发布于 2018 年 —— 创业第 25 年,先用 K/3、星空把应用做赢,平台是从成熟应用里**收敛**出来的;星空自己也是后来才逐步迁底座。
落地纪律:
- **不建 PaaS、不做低代码、不做"为了将来"的抽象**。共享能力只在"第二个使用者出现时"才抽(rule of two),抽完进底座清单。
- 防跑偏红线:任何窗口提出"先搭个平台/引擎/框架以后好扩展"= 违反本章,打回。

## 已是制度、升格为公司标准(继续执行,不重述细节)

- **质量门禁**:7 道守门(size/ratchet/SQL 闸/UI 闸/AI 味闸/视觉照搬闸/pre-push 硬拦)+ CI fail + 2900+ 单测 —— 对标金蝶"AI 生成 41% 代码"时代的答案:我们代码近 100% AI 生成,**机械闸就是研发管理**。能机械化的质量要求必须做成闸,不靠自觉(家规已立)。
- **设计先行**:UI 稿→拍板→图纸(00-05)→封板→一口气施工拎包入住;六类完整性闸(操作面/配置后台/接入流/四态/角色/联动)缺一类=没设计完。
- **工程硬约束**:单文件<500 / schema 走迁移 / SQL 参数化 / 钱用 decimal / UTC / 新文件≥1 测试 / 四态 / Conventional Commits / 去 AI 味(全局工程标准文件)。
- **验收诚实**:真账号+真浏览器+真库 E2E;headless≠真机;没验到的不说"已验证"。

## AI 战略(苍穹 GPT 路数 · 我们已同构)

金蝶做法:不自研通用大模型,接入多家(文心/OpenAI/火山/华为),自建企业级工程层(任务编排/嵌入助手/知识引擎),AI 收入单独披露,目标 2030 占 30%。
Pearnly 对应(方向已对,确认为标准):
- **不自研模型,多供应商可换**(现 Gemini 等;供应商是实现细节,接口层隔离)。
- **工程层是护城河**:置信门槛(高自动/低人审)、review_learned 越用越省、用户看得见的规则(不是黑箱)—— 与 Dext/Xero JAX 国际金标同路数,坚持。
- **AI 嵌进流程,不做单点 AI 功能**:智能在"录入即经营/自动做账/报前体检"里,不做独立"AI 聊天"噱头。
- **UI 不露 AI 字眼**(家规,金蝶 UI 同样用"智能助手"叙事):卖省事的结果,不卖 AI 的手段。
- 计费:AI 消耗走 credits 增值(04 章),成本可控可见(超管成本面板已有)。

## 研发节奏(裁剪金蝶,不照搬)

金蝶研发费率 20% 营收、转型亏五年 —— 那是上市公司的耐心。我们的版本:
- **闭环优先**:进项→做账→报税→e-Tax 走完才谈横向扩张(2.0 路线不动摇)。
- **每窗口可上线**:分批施工、批批独立可 push(权限图纸的 5 批就是模板),不积大坝。
- **维护预算**:守门/重构/补课(如 PDPA)占常态窗口的固定比例,不让功能窗口挤光(整顿期的教训:屎山是没维护预算欠出来的)。
