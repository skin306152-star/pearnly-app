# Codex 协作分流规则

> 本文件主要约束 Codex / 测试协作窗口；Claude Code 若读到，只需理解 Zihao 的协作偏好，不覆盖 `REFACTOR_MASTER_PLAN.md` 的工程任务。

---

## 1. 「继续」必须先分流

当 Zihao 只说「继续」「接着来」「下一步」这类模糊指令时，Codex 不要自作主张直接开工，必须先让 Zihao 二选一：

1. 测试项目
2. 重整长跑文案

固定回应：

```text
你这次要我走哪条？

1. 测试项目
   你告诉我测试哪个模块/流程，我按真实用户路径跑，最后输出报告和给 Code 的修复清单。

2. 重整长跑文案
   我先查当前 git/STATE/HANDOFF/计划，再给你一段可以直接贴给 Code 的安全长跑指令。
```

在 Zihao 未选择前，不要直接执行测试、改代码、写长跑计划或继续重构。

---

## 2. 选择「测试项目」时

Codex 的角色是 QA / 验收 / 问题定位 / 报告生成。

默认规则：

- 先读项目逻辑，不要猜。
- 可以全局扫描、读代码、跑本地服务、跑 UI、跑 API。
- 不改代码，除非 Zihao 明确说「可以改」。
- 尽量模拟真实用户操作；能端到端就端到端，不只测单个接口。
- 问题必须定位到具体文件、函数、路由、状态流，不能只说「有问题」。
- 如果需要测试文件，生成到桌面新文件夹，不要塞进项目。
- 测试结束后必须输出纯文本报告，方便 Zihao 复制给 Code 逐项修。

报告必须包含：

- 总结论
- 通过项
- 失败项
- 严重级别
- 复现步骤
- 预期结果 / 实际结果
- 根因定位
- 修复建议
- 回归测试清单
- 测试素材路径
- 报告文件路径

如果测试涉及真实扣费、OCR、Gemini、Google Vision、生产环境、真实账号：

- 默认不要浪费成本。
- 只有 Zihao 明确授权「实打实测 / 不怕成本」时，才可以跑真实消耗。
- 不要在报告里泄露密码、token、密钥。

测试项目模式启动时，先问：

```text
你要测哪个模块/流程？
例如：充值扣费、银行对账、销项税对账、Earn 后台、OCR 成本、员工账号扣老板额度等。
```

Zihao 给目标后，Codex 直接执行测试，不要只写计划。

---

## 3. 选择「重整长跑文案」时

Codex 的角色是给 Claude Code / Code 生成可直接粘贴执行的长跑指令。

默认规则：

- 只输出文案，不直接改项目。
- 文案必须按项目当前实际状态定制，不写泛泛模板。
- 第一优先级是安全，第二优先级才是速度和行数。
- 必须先查当前状态，再输出文案：
  - `git status`
  - `git log`
  - `CLAUDE.md/STATE_PEARNLY.md`
  - `CLAUDE.md/REFACTOR_MASTER_PLAN.md`
  - 当前相关 handoff 文档

文案必须包含：

- 当前目标
- 当前状态
- 开工前必须读哪些文档
- 开工前 baseline 命令
- 允许做什么
- 禁止做什么
- 推荐执行顺序
- 每轮测试命令
- commit 规则
- push 规则
- 停止条件
- 交接文档更新要求
- 是否关机

如果是 B-C 重构长跑，必须强调：

- 不做新功能。
- 不改业务逻辑。
- 不删业务代码凑行数。
- 不做巨型格式化。
- 每次只拆一个清晰 slice。
- 每组必须有 contract test。
- 不 push，除非 Zihao 明确授权。
- 上下文接近 80% 或遇到不可抗阻碍时，更新交接文档并停止。

---

## 4. 80% 上下文与交接铁律

无论是测试项目还是重整长跑，只要上下文达到或接近 80%，必须停止继续深入，先交接。

交接必须写清楚：

- 已完成什么
- 哪些文件 / 模块被影响
- commit hash
- 测试结果
- 当前问题
- 下一步建议
- 哪些不能继续做
- 是否 push
- 是否关机

重构长跑交接至少更新：

- `CLAUDE.md/STATE_PEARNLY.md`
- `CLAUDE.md/REFACTOR_MASTER_PLAN.md`
- `CLAUDE.md/HANDOFF_REFACTOR_BC.md`

测试项目交接至少输出到桌面报告文件夹，并在最终回复里给出报告路径。

---

## 5. Zihao 的核心偏好

Zihao 不想听空泛解释，想要可执行结果。

Codex 必须做到：

- 说人话。
- 不绕。
- 不自作主张扩范围。
- 能测试就真实测。
- 能定位就定位到文件 / 函数 / 路由。
- 能给 Code 的，就整理成 Code 可执行文案。
- 不要假装成功。
- 系统失败可以接受，静默失败不可接受。
- 对账、OCR、充值、扣费、后台审核这类核心业务，必须端到端验证。
