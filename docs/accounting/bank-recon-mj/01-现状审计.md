## 3. 重复建设 / 冲突 / 技术债风险点

1. **旧 bank_recon 表不能直接装新功能**:session/user 模型 vs 新功能需要的"按套账持续流水池"(tenant+workspace_client 隔离、跨期累积、状态机)。**决策:解析复用、存储新表**(acct_bank_accounts + acct_bank_lines,见 04 技术方案);绝不往 session 表加 ws 列硬改(动既有对账中心读路径,blast radius 大)。
2. **"对账中心"边界必须钉死**:既有对账中心 = 事务所工具(VAT 报表对发票、GL 参考工具,一次性分析,不写账);新银行对账 = 做账模块内的记账动作(写凭证、动账本)。两者数据源/产物/入口都不同——**导航上对账中心在「事务所工具」组,银行对账在「做账」组**,文案避免互相蹭词。风险:用户两边都看到"对账"二字 → PRD 文案区分(对账中心=核对报表,银行对账=对银行)。
3. **学习记忆只学 purchase**:`review.py` L95-96 对 sale/pos/payment 返回空——扩 bank_line 时照 purchase 的 desc_hash 模式加,别发明第二套记忆表。
4. **凭证生成防重**:uq_jv_source(source_type,source_id)排除 void——bank_line 生成凭证用 source_type='bank_line'+line id 天然防重,**不要**再加自己的去重表。
5. **bank_recon v1/v2 双版本债**:services/recon 里 v1 store 与 v2 并存、recon 路由分散 4 文件——**本功能不碰不修**(只 import 纯函数),留给体检排队制。
6. **POS 日结匹配粒度**:pos_sales 是逐单,银行入账常是日汇总——匹配按日聚合 POS 现金/转账小计,04 技术方案里定。
7. **手工凭证现状直接 posted**:create_manual_voucher 无草稿态。加 draft 参数走 status='pending_review',不新建状态。

## 4. 审计结论

- 不需要新权限码、不需要动 authz registry。
- 新增面收敛为:2 张新表 + 1 个新 router + 2 个前端屏 + sources/review 各一小段扩展 + 模板 1 张表。
