# 超越方案 · 进项采集与自动做账(2026-06-10 调研定稿)

> Zihao 问题:进项刚收尾、做账未建,别人已经成熟,怎么超越?
> 答案先行:**不在别人的赛道追(OCR 精度军备/通道大全/bank feed 聚合),把成熟基线的工程活补齐,然后在五个结构性空档全力打穿 —— 其中三个我们已经站着半只脚。**
> 依据:两份深挖调研(进项采集:Dext/Paypers/FlowAccount AutoKey/PEAK×ZTRUS/Hubdoc;做账:Xero JAX/QBO/Vic.ai/Digits/Botkeeper/Puzzle/AccRevo/泰国银行条件),官方文档+真实差评核实,本页只留结论。

## 一、先认清:他们"成熟"的真实成色

| 玩家 | 成熟在哪 | 差评里挖出的真实弱点 |
|---|---|---|
| Dext(采集金标) | 通道全(拍照/email-in/**门户 fetch 周轮询**)+ 供应商规则 7 字段 + Auto-Publish + 重复检测双模式 | 手写/低质票照样人工;行级提取要加购 credits;自动续约陷阱口碑稀烂;**零泰国本地化** |
| Paypers(泰) | LINE 入口正确(拍照+打字+群组)、电商截图、替代收据自动化 | **终点是 Google Sheet 不是账** —— 客户长大就流失;无 API 无对接;2 人团队无引擎沉淀 |
| FlowAccount | 单据驱动分录标配、AutoKey 扫票 | AutoKey=预填不是入账;不吃手写;无行级无邮件通道;**无 bank feed 无月结自动化** |
| PEAK | 泰国做账最深;**Krungsri 流水 API 日拉**(全泰唯一真 feed) | 只此一家银行;对账=建议+人确认,无置信自动确认无学习;重票量要绕 ZTRUS 外部工具 |
| AccRevo(泰·事务所) | 5 本日记账自动分流方向对、事务所网络心智 | 实质=OCR+模板初稿;无自动化率/置信机制/银行对账披露;话术先于深度 |
| Xero JAX/QBO/Vic.ai(国际做账金标) | 置信分流+学习、>80% 自动对账目标、90%+ 直通率(成熟期) | **燃料是 bank feed,泰国 2027 年前没有这个基建** —— 金标核心体验在泰国失效 |

两个行业真相(全部翻车案例的共性):
1. **自动化率是营销值**(成熟期+干净数据源),冷启动 4 周才爬到 ~78%(Vic.ai 实测),没人敢说 100%。
2. **用户暴怒从来不是因为分类错,是"错得安静"**:QBO 静默批量改账、Puzzle 改不回来。**安全带(看得见+撤得掉+不再犯)比准确率值钱。**

## 二、三类动作

### A. 别追的(明确不做,省下力气)

- OCR 通用精度军备赛:Dext 喂了十年票还是 99.9% 营销口径+手写翻车,追不上也不必追。
- 欧美门户 fetch 清单、Plaid/Yodlee 式聚合:泰国没这个基建,抄不动。
- Digits 式"95% 全自动账本"叙事:专业圈正在反噬黑箱,我们不碰这种承诺。

### B. 必须补的(成熟基线=工程活,不补谈不上同台)

进项侧四件,一个小窗口能收掉:
1. **批量上传**(Dext 数百张/次;我们逐张)。
2. **email-in 专属转发地址**(票据转邮箱即入待归类;Gmail 自动抓票后置)。
3. **供应商规则 + 免人手直发**:按供应商记默认科目/税率/方向,高置信熟面孔直接入账不过收件箱(= Dext Supplier Rules + Auto-Publish;我们 review_learned 的进项侧落地,机制同构提前到采集层)。
4. **重复检测双模式**(供应商+日期+金额三要素,自动并/标黄人审)。

### C. 全力打穿的(五个结构性空档 · 超越的本体)

**C1 · 第一方数据闭环(最大的结构优势,调研证实)**
Dext/Paypers 解析票据是因为不拥有交易源头;Puzzle 证明了结构化源直连才是高自动化率的真路径(它靠 Stripe/Brex 直连,不靠 OCR)。**我们的 POS 卖货、销项开票、采购下单在 Pearnly 里发生 —— 这部分零 OCR 零猜测,凭证置信度天然 100%。** 打法:做账引擎按数据源分级(第一方交易→直通自动过账;OCR 票据→置信分流;银行流水→建议匹配),把"自动化率"的大头建在自家地基上,OCR 只覆盖外部来票那一截。

**C2 · 泰国合规三件套闭环(全球零覆盖+本地没人接通,我们已站半只脚)**
- WHT:OCR 认出扣缴凭证 → 自动生成 ภ.ง.ด.3/53 行(全球玩家零覆盖;我们进项已入账,差报税侧接通)。
- 替代收据:生成(已有)→ **自动入账 → 进税表**(Paypers 只归档不入账,闭环是空的)。
- LINE→复式账→税表:Paypers 证明 LINE 是对的入口,但它终点是 Sheet;**这条线通到账和税,泰国没人做** —— 就是 Phase2+3 本身。补一个 Paypers 已验证的轻功能:LINE 群组多人提交(店员都能拍票)。

**C3 · 错账安全带 = 做账的信任护城河(执行级修订,并入做账图纸)**
我们置信门槛+人审例外+review_learned 的设计 = 国际金标同路数,且**泰国三家都没有**。但调研的翻车案例给出三件必补的安全带(JAX 范式):
1. **Method 标注**:每笔凭证标"自动/建议/人工",列表可按此筛(JAX Method 列)。
2. **一键撤销重做**:自动过账的凭证可整笔 undo+重判,绝不"改不回来"(Puzzle 头号差评)。
3. **粒度 opt-in**:自动过账按规则/科目粒度开关,新租户默认"建议模式"跑两周再开自动(QBO 静默批改的反面)。
加上已有的"人话翻译每笔为什么这样记"= 可解释,四件齐 = 2026 金标交互水平。**宣传口径也定死:不吹自动化率,吹"错了立刻看见、立刻撤销、下次不再犯"。**

**C4 · 银行对账的泰国现实路径(JAX 的燃料层,自己造)**
泰国无 open banking;BOT "Your Data" 法规已发布(2025-10),个人账户 2026 年底先行、企业相关 2027-2028 分期 —— 这是确定的时间表。打法三段:
- 短期(做账上线即要):**对账单文件解析做到极致** —— 泰国主流银行(KBank/SCB/BBL/Krungsri/KTB/TTB)的 PDF/Excel 对账单逐行解析,复用现成 recon/importer(流水导入/AI 映射/模板学习已有,这是别人没有的存量资产)。
- 中期:学 PEAK 逐家谈 statement API(Krungsri 已证明可谈成)。
- 预埋:做账的银行接口层按 Your Data 标准留位,**2027 开闸做第一批接入者** —— 到那天,JAX 的燃料在泰国点着,而交互范式我们已经跑了一年。

**C5 · 泰国版 Invoice Fetch(品类内含金量最高的未开垦通道,P2)**
Dext 每 7 天自动从供应商门户拉票,清单全是欧美;泰国的电力(MEA/PEA)/电信(AIS/True)/Shopee·Lazada 卖家后台,**没有任何人做自动拉票**。固定月费票(水电网话)是小店最规律的进项,自动拉=每月零动作。工程量大且涉门户稳定性,**等真实用户验证需求后再上**,但方向占住:这是我们而不是 Dext 能做的(本地门户+泰文)。

## 三、落地顺序(并入现有路线,不另开战场)

| 步 | 动作 | 归属 |
|---|---|---|
| 1 | C3 安全带三件套修订进做账图纸(02-posting-rules+屏稿),随做账施工 | 做账窗口(下一个大窗口) |
| 2 | C1 数据源分级写进做账引擎(第一方直通/OCR 分流/银行建议) | 同上 |
| 3 | C4 短期:六大行对账单解析,复用 recon/importer,= 做账屏 6 银行对账的施工内容 | 同上/紧随 |
| 4 | B 基线四件(批量/email-in/供应商规则直发/重复检测) | 进项收尾小窗口(做账后) |
| 5 | C2 报税侧接通(WHT→PND 行/替代收据→税表) | Phase3 报税施工(图纸已封) |
| 6 | C5 泰国 fetch + 电商 API 直连 + LINE 群组 | P2,设计伙伴用户反馈后排期 |

## 四、一句话

**别人的"成熟"是在"采集"或"记账"单段内卷出来的;泰国市场的结构性事实(无 bank feed、LINE 即入口、WHT/替代收据无人闭环)+ 我们拥有交易源头(POS/销项/采购第一方数据),决定了超越的打法是把"经营动作→账→税"整条线第一个走通,并用安全带赢下信任 —— 而不是在任何单段做到世界第一。**
