## 0. 使用规则

1. **一次只派一个任务**  
   尤其是 P0 / P1，避免多个 Code 同时改 `app.py`、`db.py`、`home.js` 造成冲突。

2. **每个任务都要先看 `git status`**  
   不允许覆盖已有未提交改动。当前已知工作区有:
   - `.claude/settings.local.json` modified
   - `_deploy_nginx/`, `_test_reports/`, `probes/`, `recon-i18n-fixes-2026-05-19.md`, `usage_report.py` untracked

3. **先修安全和基线，再做拆分**  
   推荐顺序: P0 → P1 → P2 → P3。

4. **任务完成必须给验收证据**  
   至少包含:
   - 改了哪些文件
   - 跑了哪些检查
   - 哪些检查没跑，原因是什么

5. **默认不部署生产**  
   除非你明确说“部署上线”，否则只改本地项目。

### 0.1 每项任务是否可以部署生产

结论: **不是每一项都需要部署生产，但每一项都必须做“部署判断”。**

你可以按下面规则打勾:

- 改到生产运行代码的任务: 完成本地验证后，通常可以单独部署生产。
- 只改文档、测试、CI、检查脚本的任务: 完成后提交即可，通常不需要部署生产。
- 架构拆分、前端模块拆分、依赖锁定类任务: 可以部署，但要先做冒烟测试，范围越大越不适合当天强行上线。

## 2. P0 任务卡

### P0-01 · `/internal/deploy` webhook fail-closed

**目标**  
如果 `GITHUB_WEBHOOK_SECRET` 没有配置，`POST /internal/deploy` 必须拒绝请求，不能降级成无鉴权并触发部署。

**背景**  
体检发现 `/internal/deploy` 当前逻辑是: 有 secret 才校验 HMAC；如果 secret 为空，可能直接进入部署流程。这是生产安全高风险。

补充核对: `/internal/deploy/manual`、`/internal/deploy/log`、`/internal/install-playwright` 当前已经使用 `if not secret or token != secret`，属于 fail-closed。这个任务不要无意义重写那 3 个端点，只需要确认不破坏它们。

**建议涉及文件**

- `app.py`
- 可选: `tests/unit/test_internal_deploy_security.py`

**具体要求**

- `/internal/deploy`: `GITHUB_WEBHOOK_SECRET` 缺失时直接返回 503 或 500，不执行部署。
- `/internal/deploy`: secret 存在时必须始终校验 `X-Hub-Signature-256`。
- `/internal/deploy`: 签名错误时返回 403，不执行部署。
- 可选: 在启动流程中增加生产环境缺少 `GITHUB_WEBHOOK_SECRET` 的 critical/error 日志。
- 确认 `/internal/deploy/manual`、`/internal/deploy/log`、`/internal/install-playwright` 仍然保持 fail-closed，不要改坏。
- 不改变成功路径的行为。

**验收标准**

- `/internal/deploy` 在 secret 缺失时不会执行 `subprocess.Popen`。
- `/internal/deploy` 在 secret 存在但签名错误时返回 403。
- `/internal/deploy` 在 secret 存在且签名正确时保持原逻辑。
- 新增或更新 unittest 覆盖 secret 缺失场景。项目已有 `tests/unit` 和 unittest 风格测试，可以沿用。

**建议验证命令**

```powershell
python scripts\check_imports.py --quiet
python -m unittest tests.unit.test_internal_deploy_security -v
```

**可直接发给 Code 的指令**

### P0-02 · `/api/version` 去除公开内部诊断

**目标**  
公开 `/api/version` 只返回公开信息，不再暴露 `playwright`、`last_500`、traceback、部署内部状态。

**背景**  
当前 `/api/version` 返回内部诊断信息，包含 `last_500` 和 Playwright 状态。公开暴露这些信息会帮助攻击者理解系统。

**建议涉及文件**

- `app.py`
- `static/version-banner.js` 如依赖字段，需要确认
- 可选: `tests/unit/test_version_public_contract.py`

**具体要求**

- `/api/version` 保留:
  - `version`
  - `ts`
  - `release_notes`
- 从公开返回中移除:
  - `playwright`
  - `last_500`
  - traceback / deploy log / sentinels / python_bin 等内部字段
- 新增超管鉴权接口，例如:
  - `/api/admin/diagnostics/runtime`
  - 返回原先内部诊断信息
  - 必须 `_require_super_admin(request)`
- 前端版本弹窗仍能工作。

**验收标准**

- 未登录访问 `/api/version` 看不到 `last_500` 和 `playwright`。
- 超管访问新 diagnostics 接口能看到内部诊断。
- 普通用户访问 diagnostics 返回 403。
- `static/version-banner.js` 不报错。

**建议验证命令**

python -m unittest tests.unit.test_version_public_contract -v

### P0-03 · 全局异常返回脱敏

**目标**  
生产 API 遇到未捕获异常时，客户端只收到通用错误码，不收到异常类型、内部 message、traceback 或诊断 URL。

**背景**  
当前 `@app.exception_handler(Exception)` 返回 `"{ExceptionType}: {message}"`，会泄露内部信息。

**建议涉及文件**

- `app.py`
- 可选: `tests/unit/test_error_response_sanitized.py`

**具体要求**

- 保留 `_record_500` 记录能力。
- 客户端响应改为:
  - HTTP 500
  - `{"detail": "server.internal_error"}` 或类似稳定错误码
- 不返回:
  - 异常类名
  - 原始异常文本
  - traceback
  - `/api/version` 诊断提示
- `HTTPException` 原有业务错误不受影响。

**验收标准**

- 人为触发未捕获异常时，响应不包含 Python exception 名称或原始 message。
- 服务端日志仍记录异常。
- `_last_500_event` 仍可被超管诊断接口读取。

**建议验证命令**

```powershell
python scripts\check_imports.py --quiet
python -m unittest tests.unit.test_error_response_sanitized -v
```

**可直接发给 Code 的指令**

不要把异常类名、异常文本、traceback、diag_url 返回给客户端。

### P0-04 · 生产 CORS 收紧

**目标**

**背景**

allow_methods=["*"]

**建议涉及文件**

- `.env.example`
- 可选: `tests/unit/test_cors_config.py`

**具体要求**

- 默认生产允许:
  - `https://pearnly.com`
- 本地开发可通过 env 加:
  - `http://localhost:7860`
  - `http://127.0.0.1:7860`
