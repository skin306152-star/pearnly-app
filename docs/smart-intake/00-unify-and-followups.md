## P0 · 涉钱 / 数据正确性(最高优先 · Codex 严重)

### F1 服务 WHT 保存后丢失 · 付款基数算错(涉钱)
- 现象:表单切 บริการ 即时合计对(税前1000+VAT70−WHT30=实付1040);**保存后详情/付款卡按 ฿1070 应付**;部分付款 ฿500 后未付显 ฿570(应是 ฿540)。截图 06_service_wht / 10_partial_paid_result。
- 已查:后端 `services/purchase/docs.py` **有**存 wht_amount/net_payable、汇总未付用 `net_payable-paid_amount`(对)。所以根因二选一,先查保存后的单子:
  - ① 前端保存时**没把行的 wht_rate/item_type=service 传给后端** → 后端重算 wht=0、net_payable=grand_total。查 `purchase-form.ts` 提交 payload 的 lines 是否含 wht_rate/item_type。
  - ② 详情/付款卡**显示/算未付用了 grand_total 不是 net_payable**。查 `purchase-detail.ts`/`purchase-modals.ts` 付款卡。
- 修:保证 行 wht_rate/item_type 入库 + 详情/付款一律按 net_payable。E2E 断言:服务行单子 net_payable=1040、部分付500后未付=540。

## P1 · 功能走不通(Codex 高)

### F3 无发票费用 / 替代收据 路径走不通(双缺)
- (a) 首屏没有明显"手动新建·没发票费用"入口(只有 LINE/拍票)。真实老板不会知道要先进拍票表单再找替代收据。**修:首屏加「手动记一笔」入口** → 直接进录入屏(费用模式)。
- (b) 表单内点 `+ สร้างใบรับเงินแทน`(生成替代收据)**按钮可点但 no-op**(无弹窗/无PDF/无新页)。截图 05b。后端 `/docs/{id}/substitute-receipt` 已存在 → 前端按钮没接/没处理返回。**修:接通 → 生成 PDF 挂附件 + 打开/下载 + 成功反馈。**

### F4 采购设置页空白
- 现象:从详情侧栏进 ตั้งค่าจัดซื้อ,左侧高亮切了,**主区空白**(无设置项/loading/空态/错)。截图 13。
- 根因:`acct`...→ 实为 `purchase-settings` 路由/loader 没渲染(core-boot 路由映射或加载失败)。修:查 `purchase-settings.ts` 挂载 + core-boot 路由;补 loading/错误态(交互完整性)。

### F5 糊图无失败态 · 反生成 ฿0 表单(危险)
- 现象:糊图上传后**无"不清晰/识别失败/请重拍"**,直接进表单:税号识成"13"、金额 ฿0、行空。零会计老板可能误存垃圾单。截图 16。
- 修:OCR 低置信/抽取过空 → **坏图四态**「看不清·重拍/手填」,不直接进可保存表单;或进表单但**禁保存 + 顶部红条**提示需补全。(交互完整性·四态)

## P0/架构 · 统一智能入口(★根治归类冲突 · Zihao 真用发现)
> 商户上传吃饭费用票走「上传识别」→ 进了事务所的「识别记录」,没进费用/采购。根因:识别记录是老事务所工具、对所有业态显示、不分流;§三-bis 智能分流只 LINE 做了一半。

- **F12 统一智能入口**:一个上传/拍照口,AI 判 是什么+哪家公司(套账)+谁是我(方向)→ 决策表分流(我开的销项票→销项·进项票→采购·费用小票→费用·PO→在途·银行单→对账)。
- **F13 待归类收件箱(★长尾安全网·全局唯一)**:拿不准/低置信/unknown/没想到的 → 落收件箱(复用 intake_items·**做出 UI 入口**),AI猜测+票图+一点选(进项/费用/销项/对账/不是票)→ 建单 + 系统记住(下次同类自动)。**绝不静默丢错地方。**
- **F14 识别记录降级事务所专用**:按 business_type 门控,只事务所显示;商户上传走 F12 分流。事务所路径一字不破坏。
