## 一、后端真实现状(落地前必须知道)

### 1.1 三个对账各自的真实输入与端点

| UI tab | 后端 job_type | 真实输入(两份) | 提交端点 |
|---|---|---|---|
| 银行对账 | `bank` | 银行账单(stmt) + 总账 GL | `POST /api/recon/bank-v2/submit` |
| 收入对账 | `glvat` | **总账 GL + 税表(VAT 报告)** | `POST /api/recon/gl-vat/submit` |
| 销项税报告核查 | `salesvat` | 税表(VAT 报告) + 销项发票明细 | (salesvat submit 路由需先确认是否已接,handler `run_salesvat` 已存在) |

任务状态/结果:`GET /api/recon/jobs/{job_id}`;银行侧逐行确认 `POST /api/recon/bank-v2/confirm-rows/{job_id}`。
解析/对账引擎在 `services/recon/*` 与 `services/recon_jobs/*`,**已就绪且银行侧刚修过**(GL 堆叠版式解析、对账单日期/摘要、期初行序无关)。

### 1.2 模板以"我们后端对齐的"为准 —— UI 稿里的模板标签作废(Zihao 2026-06-13 定)

**UI 稿里的模板定义是错的,不要参考。** 模板一律用本仓库后端对齐的那套(§1.3 + 桌面已生成的
测试文件为样)。具体:

- 收入对账(glvat)的输入是 **GL + 税表 VAT**,不是 UI 稿写的「收款流水 / 销售收入」——按 GL + 税表实现。
- 模板中心里每个 tab 列的模板,以 §1.3 表为准;UI 稿里的「收款流水标准模板 / 销售收入标准模板」
  等名称**不采用**。
- 列结构以解析器实际认得的为准(已逐项验证),不照搬 UI 稿里的列。

### 1.3 四个模板与解析器现状(复用,勿重造)

| 模板 | 用于 | 解析器 | 列(语义) |
|---|---|---|---|
| 银行账单 | 银行对账 | `parse_bank_stmt_xlsx_direct` | 日期·摘要·提款·存款·余额 |
| 总账 GL | 银行对账 + 收入对账 | `bank_gl_excel.parse_gl` / `gl_vat_parse_excel.parse_gl` | 日期·凭证号·科目代码·摘要·借方·贷方 |
| 税表 VAT | 收入对账 + 销项税核查 | `vat_report_parser.parse_vat_report` | 日期·发票号·买方名·买方税号·分支·税前·VAT·合计 |
| 销项发票明细 | 销项税核查 | 同税表(行项结构一致) | 同税表 |

**GL 模板一套通吃银行对账与收入对账两侧(已验证)。税表与发票明细列结构相同。**

## 二、对草稿的逐条修正

1. **§开头「模板文件已经准备好」不成立。**
   仓库当前**没有**模板文件、没有 `static/templates/`、没有下载端点。模板要**新建**,
   且按下面 §三的 i18n 逻辑**由后端按当前语言动态生成**,不是静态文件。
   草稿里"找到并复用真实模板文件"改为"调用模板生成/下载端点"。

2. **§六 模板中心 · 收入对账** 的「收款流水 + 销售收入」改为「总账 GL + 税表 VAT」(见 §1.2)。

3. **§四.1 标准模板识别**:当前**没有**模板 ID/签名机制,只有"列头关键词匹配"(能用)。
   草稿已优雅兜底("若无可靠识别机制保留兼容逻辑")——保留;**建议新增一个轻签名**
   (固定 sheet 名如 `Pearnly-GL` + A1 隐藏标记 `PEARNLY_TPL_<TYPE>_V1`),让"标准模板读取"
   能确定判定,而不是靠文件名。

4. **新增 i18n 列头逻辑(草稿缺,Zihao 要求)**:见 §三。

5. **新增两条后端前置缺口(草稿缺,不补则模板导入跑不通)**:见 §四。

## 三、模板 i18n 逻辑(Zihao 拍板 · 必须实现)

**模板的列头与说明文案,跟随用户当前系统语言(zh / en / th / ja)。**

- 中文界面 → 下到中文列头模板;泰文 → 泰文列头;英文 → 英文;日文 → 日文。
- 因此"下载模板"是**后端按 `lang` 动态生成 xlsx**(例:`GET /api/recon/template?type=gl&lang=zh`),
  而非固定文件。生成时写入 §一.3 的列(对应语言)+ 1~2 行示例 + 一个 `说明/Instructions` sheet。
- **前提**:导入解析器必须认全 4 种语言的列头(否则用户下到自己语言的模板、填完导入却不认)。
  现状:银行账单/GL 解析器已认 中/英/泰;**VAT 解析器只认 泰/英**(见 §四.1)。

英文列头注意:税前金额列英文用 **`Net`**(解析器认 net,不认 `Amount`)。

## 四、两条后端前置缺口(模板能真正导入的前提)

### 4.1 VAT 解析器补 zh / ja 列头关键词
`services/vat/vat_parser_common.py` 的 `_DATE_H / _NO_H / _NAME_H / _TAX_H / _BRANCH_H /
_NET_H / _VAT_H / _TOTAL_H` 目前只含 泰/英;补中文(日期/发票号/买方名/买方税号/分支/
税前/税额/合计)与日文,否则中/日界面的税表+发票模板解析失败。

### 4.2 销项发票侧补 Excel 结构化路 ★实测已证阻塞
销项税核查的发票侧现在是 OCR-only(`extract_invoices_parallel`,不吃 Excel)。
**2026-06-13 实测(vat_recon)**:税表 Excel 解析 602 行正常,但发票 Excel 没被读 → 发票 0 行 →
对账 600 笔全部「报告有·发票无」、匹配 0。**这条不补,销项税核查的模板路完全不可用。**
修法:`run_salesvat` 发票侧按扩展名分流,xlsx/csv 走行项解析(复用 `parse_vat_report`/`parse_excel`),
产出与 OCR 发票相同 shape 的发票记录喂给对账;图片/PDF 仍走 OCR。

> 这两条是后端小改。若本次 UI 窗口不做后端,**必须在交付报告里明确标注为未完成的后端依赖**,
> 否则税表/发票模板的"标准模板读取"在中/日界面或发票侧会静默失败。
