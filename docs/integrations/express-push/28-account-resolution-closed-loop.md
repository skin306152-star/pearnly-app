# Express 科目解析闭环 · 施工总纲(2026-06-23)

> 承接 2026-06-23 与 Owner 的整轮讨论(科目机制 → UI 该怎么设计 → 自动推送场景 → 对抗设想 → 小助手冲突)。
> 本页 = 真相 + 设计 + 任务 + 阻碍兜底,冲突时以本页为准。前序背景见 `27-make-push-actually-land-fix-plan.md`。
> 入口认知:科目**不是开店前填一次的设置**,而是**每张发票被识别后、推送时现场产生的结果**(自动匹配到 / 匹配不到)。

---

## 0. 核心洞察(Owner 拍板)

旧心智模型(错):一个"科目映射设置页",用户开店前把 6 个科目填好。
新心智模型(对):**科目是每张票的运行时结果**。UI 长在发票生命周期里,不是设置页。

- 6 个科目里**只有「收入 / 采购」会随票变**;应收/应付随客户/供应商主档,进/销项税账套固定。
- 自动推送本身 = **按科目能不能定准的闸**:定得准才自动放行,不准当场降级转人工。钱的事低置信永不自动入账。

## 1. 已用真数据锁死的事实(不靠推断)

### 1.1 Express 的科目来源模型(官方帮助 `10linkac.htm` 权威确认)
- 销售收入 / 销项税 / 进项税 / 采购 → 取 **G/L Link Account**(= 账套 `ISINFO.ACCNUMxx` 默认向量)。
- **应收** → 优先取客户主档 `ARMAS.ACCNUM`,缺则回落 ISINFO 默认。
- **应付** → 优先取供应商主档 `APMAS.ACCNUM`,缺则回落 ISINFO 默认。
- 存货销售另有 COGS + 成品出库(STMAS,存货账套);非存货无此两行。

### 1.2 ISINFO.ACCNUMxx 索引 → 角色(跨 4 账套实证一致 · 见 §1.4)
| 角色 | ISINFO 槽位 | DATAT 实值 | 泰文名 |
|---|---|---|---|
| 收入 revenue | `ACCNUM11` | 41-01-00-00 | รายได้จากการขาย |
| 应收 AR | `ACCNUM02` | 11-02-01-00 | ลูกหนี้การค้า |
| 销项税 output VAT | `ACCNUM10` | 11-05-04-02 | ภาษีขาย |
| 采购 purchase(默认) | `ACCNUM04` | 11-04-02-00 | สินค้าสำเร็จรูป |
| 应付 AP | `ACCNUM07` | 21-02-01-00 | เจ้าหนี้การค้า |
| 进项税 input VAT | `ACCNUM06` | 11-05-04-01 | ภาษีซื้อ |

### 1.3 真单地面真相(读账套真会计录的单 · 校验上表)
- 销项:借 现金/应收 → 贷 `41-01` 收入 + 贷 `11-05-04-02` 销项税 ✓
- 进项:借 `11-05-04-01` 进项税 + 借 **按性质的费用科目**(`52-09` 油费 / `53-02` 手续费…)→ 贷 现金/应付
- **教训**:① 我曾把销项税推成 `21-03-03-00`,真值是 `11-05-04-02` → **绝不信单点推断**;② **采购费用科目按性质变**,分类是刚需不是锦上添花。

### 1.4 跨账套一致性(锁死索引语义的证据)
4 账套(57ASCRD / dat / korn / test)ISINFO 同索引指向同角色泰文名一致;但**代码差异巨大**(57ASCRD `1130-01` vs test `11-02-01-00`)。
→ 结论:**① 按索引读 ISINFO 稳;② 绝不硬编码代码,必须按账套读。**

## 2. 架构与职责(消除小助手/网页冲突)

```
小助手(配对窗)= 账套级 6 个默认科目【唯一写入处】
   · 从该账套 ISINFO.ACCNUMxx 自动识别预填(不再全落现金 11-01-01-00)
   · 会 Express 的人「确认一次」(首次确认闸)→ 配对时随 config 发云端
   · 已能读 GLACC(probe_chart_of_accounts)+ 心跳上报(chart_accounts_payload)
        ↓ config(revenue_acc/ar_acc/vat_output_acc/fallback_acc/ap_acc/vat_input_acc)+ defaults + chart
云端 = 每张票【运行时科目解析】(分级瀑布)+ 写前 GLACC 白名单 + 不可解转人工
        ↓ 解析结果 + 来源 + 置信
网页 = 只读镜像(看小助手设了啥)+ 每张票科目(详情块 / 异常补录 / 日志列)
        · 网页【绝不】再放可编辑科目映射(否则两处可改 = 冲突)
```

**层级**:小助手设"账套默认/兜底"(地板);云端按供应商/品类逐张精修(地板之上);网页看+补+学。两层叠加,不重叠。

## 3. 科目解析分级瀑布(云端 · 每张票)

按置信从高到低,命中即用:
1. **供应商/客户记忆**(per 租户学过):复用 `expense_learned`(`tax:`/`seller:` 键)。最高、确定性、可审计。
2. **确定性规则**:复用 `category_ai.classify_rules`(品名/商户规则)→ pearnly_category → `erp_account_mappings` → ACCNUM。
3. **账套默认**:ISINFO 默认(收入 ACCNUM11 / 采购 ACCNUM04)+ 客户/供应商主档(AR/AP)+ 账套固定(进/销项税)。
4. **AI 兜底**:LLM 仅"从真实 GLACC 选",低于阈值不采信。
5. **不可解** → 落异常「待补科目」,**绝不回落占位科目瞎记**。

**固定项**(永远能定,不进瀑布):应收=ARMAS/ISINFO·ACCNUM02;应付=APMAS/ISINFO·ACCNUM07;进项税=ACCNUM06;销项税=ACCNUM10。
**会变项**(走瀑布):收入科目、采购/费用科目。

## 4. 自动推送的科目闸(场景 → 措施)

| 场景(收入/采购科目怎么来) | 自动推? | 措施 |
|---|---|---|
| 供应商记忆命中 | ✅ | 用历史科目 |
| 确定性规则命中 | ✅ | 用规则科目·记来源 |
| 只能落账套默认 | ⚠️ 看策略 | 稳健=转人工;省事=推但标"默认·待核" |
| AI 兜底低于阈值 | ❌ | 转人工·AI 建议预填异常卡 |
| 完全匹配不到 / 无默认 | ❌ | 落异常「待补科目」·人补一次+记住 |
| 方向判不出(主体没绑/税号不齐) | ❌(前置) | 进异常·不进自动推送 |

**策略旋钮**(per 连接):稳健(仅记忆+规则自动)/ 省事(默认也自动·标待核)。阈值不写死。
**留痕回流**:每张自动推的票记 科目+来源+置信;改科目→学进记忆;已过账改错→冲销-重过账(v2)。

## 5. 三道防静默错账闸(对抗设想逼出 · 必做)

1. **不信推断**:科目从该账套 ISINFO 默认 + 真单/名校验得来,绝不跨账套硬编码代码。
2. **写前白名单**:每个 acc 必须 ∈ 当前账套 GLACC(可记账科目),否则拒写→转人工(挡死科目/AI 乱码/缓存过期)。
3. **每账套首次确认闸**:某账套第一次自动推前,把读出的 6 默认科目带名字给会 Express 的人确认一次,确认后才开自动。一次人工,挡静默错账灾难。

## 8. 补充项(对抗设想新发现 · 别漏)

- **预扣税 WHT(หัก ณ ที่จ่าย)**:泰国服务类采购常有预扣税(ภ.ง.ด.3/53),多一个科目+申报。v1 检出有 WHT→转人工不自动;v2 建模。
- **存货 vs 非存货**:真销项含 COGS(51-01)+成品出库(11-04-04)。现 writer 只非存货。客户卖存货→分录不同。v1 范围限非存货,存货→转人工或标注。
- **现金 vs 赊账对手方**:HL/HP 用现金 11-01-01;RR/IV 用 AP/AR。解析器按付款状态(RECTYP)选对手方科目(已有方向判定,确认覆盖)。
- **舍入**:VAT 自洽容差已 ±0.02(`common.amounts`);不平→留人工不强配。
- **幂等**:异常补录后重推须走同一 `idempotency_key`,不重复入账。
- **账期关闭**:超期 Express 拒(Msg#70/71)→ 识别为"账期问题"异常,非静默失败。
- **schema 迁移**:push 记录加 科目/来源/置信 列走迁移不 ad-hoc;decimal;UTC。
