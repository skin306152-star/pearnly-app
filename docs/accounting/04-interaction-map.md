## 一、导航 IA(做账组)
设计语言钉死期已建占位:「做账」组(`data-collapsible` 新建) → `自动凭证`(原"凭证中心"改名·`data-route="vouchers"`)。本期落地后该组:
```
做账 ▼ (门控 key=accounting · 业态套餐开)
  ├ 做账 / 凭证      data-route="vouchers"        → 屏1 主屏(凭证列表)  主入口
  ├ 逐笔审          data-route="acct-review"     → 屏2(待审·有数才显红点)
  ├ 科目表          data-route="acct-accounts"   → 屏3(平铺页·配置后台)
  ├ 做账设置        data-route="acct-settings"   → 屏5(平铺页·配置后台)
  └ 出账本/报税包    data-route="acct-books"      → 屏4(月末出口)
```
> `自动凭证` 占位 → 落地即换成上述真子项(凭证=主屏)。门控 `assert_module_enabled(accounting)`。

## 三、状态流
```
业务(进项post/销项issue/POS埋单/收付款) ──hook enqueue──► 引擎套模板 + 定 source_tier
   confidence≥门槛 且 auto_post 放行(全局/规则粒度) → auto_posted·method=auto → 主屏「自动·已过账」绿标
   confidence<门槛 或 建议模式(auto_post=false·新租户默认) → pending_review·method=suggested → 「待审」黄标 + 行动卡 +1
逐笔审定夺 ──review(remember)──► posted·method=manual + 写 review_learned(下次同类自动过)
撤销重做 ──unpost──► 原凭证 void + 同 source 重判(吃最新映射/记忆)
月末 close-period ──► 跑 R9 VAT 结转 + 锁期 → 出账本可导
```
> 单一事实源:主屏/待审/出账本都从 journal_vouchers 派生。

## 五、四态 + 边界(每屏必覆盖)
| 屏 | 空 | 加载 | 错 | 边界 |
|---|---|---|---|---|
| 1 主屏 | 「本月还没业务,做账等业务来 —— 去开张发票/拍进项票」 | 骨架行 | [重试] | 全自动无待审=不显待审卡 |
| 2 逐笔审 | 「没有待审,都自动记好了 ✓」 | 骨架 | [重试] | 最后一笔审完→回主屏 |
| 3 科目表 | (预置已 seed·不空) | 骨架 | — | 预置不可删·可停 |
| 4 出账本 | 「本月还没结账」/未结提示先结账 | 骨架 | 导出失败重试 | 借贷不平=红条「有不平凭证,先去逐笔审」禁结账 |
| 5 设置 | (默认已 seed) | 骨架 | 保存失败提示 | 改存货法/启用期二次确认;**自动过账区=全局开关+按规则(R1-R9)粒度开关·新租户默认建议模式**(安全带③);开自动二次确认「确认让 X 类业务免审自动过账?」 |
> 全局边界:**借贷不平的凭证绝不落库/过账**;红冲方向不明→待审;缺科目映射→待审+提示去配。

## 六、角色权限
- owner / 会计:看 + 审 + 改科目 + 过账 + 改设置/映射 + 结账 + 导出。
- 员工:**只看**凭证/账本,不可审/改/过账/结账。
- 写接口 `require_account_owner` 或会计;机械闸 `test_accounting_permission_gate`。

## 八、交互完整性(过 DESIGN_LANGUAGE 四-bis · 封板必查 · 每动作三态 + 不可逆确认 + 落点 + 字段空态)
> 补:做账图纸先于「交互完整性」闸成稿,此节按四-bis 逐动作钉死反馈,防进项那类"作废没提示"边角漏。

| 动作 | 成功 | 失败 | 进行中 / 边界 |
|---|---|---|---|
| 逐笔审「确认·下一笔」 | toast「已过账·借贷已平」+ 推进下一笔;最后一笔→空态「都记好了 ✓」回主屏 | inline「借贷不平/异常·未过账」 | 禁连点 |
| 逐笔审「改科目」 | `.modal` 选完即时重算预览 + 确认 | — | 非原生 prompt |
| 逐笔审「跳过」 | 留 pending +「已跳过·稍后再审」 | — | — |
| 凭证「作废」 | **二次确认**「作废将生成反向凭证·已过账影响 X」→ 成功 toast「已作废·已生成反向凭证 #」+ 详情灰态 | 「该期已结账·不可作废」 | 不可逆→必确认 |
| 凭证「撤销重做」(安全带②·auto/posted 可用) | 确认「撤销这笔自动凭证并按最新规则重判?」→ toast「已撤销·重判结果:自动过账 #N / 已进待审」 | 「该期已结账·不可撤」`acct.period_closed` | 原凭证置 void·新凭证引用同 source·绝不"改不回来" |
| 凭证「编辑/改科目」 | pending 可改→重断言平+toast;posted→提示「已过账·走调整分录」 | 不平拦 | — |
| 手工凭证 | 成功 toast | inline「借贷差 ฿X·补平再存」**(不平绝不落)** | 实时显借贷差 |
| 月末「结账」 | **二次确认**「结账后本期锁定·不可再生成/改」→ 成功「6月已结·VAT 结转凭证 #·应交 VAT ฿X」 | 「有不平凭证·先去逐笔审」禁结 | 不可逆→必确认 |
| 出账本「下载/生成」 | 下载/打开 PDF | 「生成失败·重试」 | 生成中 loading;未结账→引导先结账 |
| 科目「加/改」 | `.modal` + toast | 编号重复提示 | 预置删→禁+提示;停用→确认 |
| 设置「保存」 | toast「已保存」 | 保存失败说明 | — |
| 缺科目映射(自动过账撞缺) | — | 落待审 + **行动卡「去做账设置配这个科目 →」直达**(入口落点·非干瞪眼) | — |

**字段空态**:human_note/source_ref 缺→合理占位不空白;account 名必有(预置);科目映射未配→显式标「未配·去配」非静默空白。confidence/金额缺→不显假 0。
> 凡"点了没反应/静默改变/谜之空白"= 不合规,封板打回。

## 九、银行对账 + 可见规则(竞品调研后补 · 见 `docs/product-vision/02`)
> 补做账的最大缺口:**账与银行对上=账对不对的最终校验**(Xero/QBO/PEAK 核心)。不补 → 账可能跟银行对不上,老板不敢信。设计稿 `Pearnly_做账_UI预览/06-银行对账`。

- **导航**:做账组加 `data-route="acct-bankrecon"`(银行对账)。
- **复用(不从零造)**:`services/importer`(流水导入/解析 stmt_review + AI 映射 ai_mapping + 模板学习 template_learning)+ `services/recon`(匹配引擎)。商户做账 = 在其上做一个**简化的"流水↔凭证"对账面**。
- **C4 泰国现实路径(超越方案定稿)**:泰国 2027 前无 open banking → 短期把**六大行(KBank/SCB/BBL/Krungsri/KTB/TTB)PDF/Excel 对账单逐行解析做到极致**(importer 的解析/映射/模板学习是别人没有的存量资产);中期学 PEAK 逐家谈 statement API;银行接口层按 BOT "Your Data" 标准留位,2027 开闸做第一批接入者。
- **银行流水 = source_tier:bank**:只做建议匹配,人确认才落凭证,绝不自动过账(见 02 数据源分级)。
- **流程**:导入流水/连银行 → 引擎把每条流水匹配账本收付款凭证 → **高置信自动对**(显「已对✓」)· **低置信留确认**(建议 + [确认]/[选别的]/[新建凭证]) → 没匹配的给建议(如"手续费")+「以后这类自动记」**学规则**。
- **数据**:`bank_statements` + `bank_lines`(导入·套账隔离)+ line↔voucher 匹配关系;复用既有 recon 结构优先,不够再扩。
- **交互完整性**:确认匹配→toast + 行变「已对」;新建凭证→走手工凭证(借贷平);学规则→落 review_learned 并在「可见规则」列出。
- **可见规则**:做账设置加「自动记账规则」列表 —— 把 review_learned 学到的(供应商→科目/描述→科目/WHT)**列出 + 可编辑/删**(像 Xero bank rules)。智能不黑箱。

> 月末「结账」前提补一条:**银行对账完成度**(未对账流水多 → 提示"还有 N 笔没对账,建议先对完再结")。
