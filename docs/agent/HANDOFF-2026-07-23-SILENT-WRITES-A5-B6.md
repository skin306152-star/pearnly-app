## 这场在解一个什么问题

Pearnly AI 月结线上有一类共同形状的缺陷:**「读不出来 / 认不准 / 机器改过」的东西被当成
「没有 / 是同一张 / 已核对」悄悄放行**,结果就是申报数字错了而没有任何人知道。本场按严重度
逐条根治,七条全部上线 CI 绿。

判据统一成一句话:**机器不确定或机器改过钱,就必须有人看一眼**——绝不自动吞、也绝不自动剔除
(自动剔除同样是背着人改钱,只是方向反了)。

## 每条的落点(改哪儿、为什么这么改)

### A-5 · 读不出的钱字段
`reconcile_gates.unreadable_money_fields` 把「印了但读不出」与「没印」分开:`None`/空串交
`_effective` 派生兜底,其余解不出的(含 `N/A`、横杠,以及**合法但会把整单污染成 NaN 的
`NaN`/`Infinity`**)一律停机点名,报到票、到字段、并带上机器读到的原文。

取数入口收成**唯一一处** `_face_value`,四条按票面取数的路径共用(自动 ok / flagged 按票面裁决 /
方向裁进项 / 销项票改判进项)。判据分散正是 A-2 那种「唯独漏了一处」的病根。

### A-4 · 银行改写不留人
`_repair_amount_from_balance` 改写金额时把 `balance_ok` 由 False 翻 True,而 `needs_review`
正是由 `balance_warn_count` 驱动 → 整份账单被判「不用人看」。
- **上半**:改写行单独计数 `amount_fixed_count`(不并进 `balance_warn_count`,并进去会污染断链数、
  误触换眼重读),由它触发 S8 逐行核对关(主站对账 job 路)
- **下半**:银行件标 `flagged` + `flag_reason=bank_amount_rewritten:N` → 进 /ai 审核队列并挡住
  签批冻结(月结工单路)

★ **订正盘点里的两处夸大**:Excel 明细行早就按 `stmt_autocorrected` 渲染黄色「已修正」,
Excel 汇总页与 /ai 机器改动清单也早就按件计数。真正缺的只有一条:**没有任何闸「要求」人去看**。

### B-5 · 防重指纹撞车
不放宽判重(那只是把「少抵」换成「多抵」),而是按**指纹强度**分流:
- 强(票号在场 + 金额读得出)= 身份确凿 → 自动排除的既有行为一字不变
- 弱 → 留在进项堆标 `duplicate_suspect:{原件}`,进人审、挡签批

判据 `dedupe_key_is_strong` 与指纹算法同放 `purchase/totals.py`(同一处知识不分家)。

### B-2/B-3 · VAT 读花被当非税件排掉
VAT 读不出时**两个静默出口都不走**(判银行流水页 / 判无税务要素的 non_tax),改归
`kind=unknown` + `vat_unreadable` 走既有方向裁决通道。读不出税额本就判不了进/销,交人定向是对的
语义;**裁成进项后票面税额还要再过 A-5 那道读花闸**——两条修复自然咬合。

判据复用 A-5 落的 `unreadable_money_fields`,不另写一套「什么叫读不出」。

### B-1 · 认不出的表当销项汇总
此前根本没有「这是不是销项汇总」的**正面判据**,只有 GL 不像、银行不像之后的盲猜。补
`_xlsx_header_is_sales`(与 `_GL_NAME_KW`/`_BANK_HEADER_KW` 对称):
- 文件名已明说 / 表头有销售语义列名 → 照常,零摩擦
- 三家都不像 → 仍归销项汇总(下游口径不变)但标 `table_kind_unclear` 要人确认一次

★ **订正盘点**:原描述说「损坏 xlsx 被静默收进来且不进任何清单」不成立——`_classify_summary`
读不了会标 flagged。真正没被拦住的是「能正常打开、有数据行、但其实不是销项汇总」的那一类。

### C-3 · 批量覆盖已裁票
`runGroupBatch` 只滤 `isArchived` 不滤 `isDecided`。判据不是新写的:`AI.reviewQueue.isDecided`
早就在,`ai-review-bulk.js` 那条路一直在用,**只有收件箱这条漏了**——同一判据两处各写一份、
漏同步一处。

### B-6 · 跨期票照抵
规则收成一处 `tax.aggregate.claim_window_expired`(账本线改调它,行为不变),工单线接上。
只识别 + 点名,绝不自动剔除。

2. **回滚验证抓到"测了个寂寞"**:A-5 第一版测试把含税额写死,退回修复后红的是 R4 试算闸而不是
   本闸——被另一道闸顺手拦下,等于没测到。改成含税额缺省(派生后票面自洽,试算闸拦不住)才隔离住。
   **凡新加一道闸,回滚验证时要确认红的是它,不是别人。**

3. **断言全绿 ≠ 界面对**:A-4 首版把银行件直接塞进为票据设计的审核卡,四个断言全过,截图一看:
   工单 chip 甩裸机器名、读值表整列「—」、动作是「采纳票面/改数/剔除」。**截图必须真看,不能只看
   断言。**

4. **佛历期间不转换 = 判据永远不触发**:工单 period 是佛历(2569-05),票面日期落库是公历,
   直接比差 543 年。B-6 第一版就这么写的,测试没触发才发现。换算收进
   `core.thai_date.gregorian_period`,并专门写了一条测试拿佛历/公历两种期间跑同一张票。

5. **`ai-i18n-*.js` 是直接 serve 带 `?v` 的,不进 bundle**,同一批文件每改一轮都得重新 bump
   (本场 bump 了四轮)。

6. **用 python 字符串替换改源码时 CRLF/LF 不匹配会静默不改**,脚本还照打 "ok"——跟「闸后接管道
   吞退出码」是同一类病。后面都改用 Edit 工具或带 assert。

7. **本地全量单测别 `export PYTHONIOENCODING=utf-8`**:子进程写 UTF-8 而父进程仍按 cp874 解码,
   `subprocess` 类测试的 stderr 会解码炸成 None → 假红。但**直接往控制台打中文的脚本
   (`check_i18n`/`check_ai_smell`/`check_file_size`)反而必须设它**,两者场景相反。
   `test_agent_capability_audit` 的本地红也是同一族(cp874),Linux CI 不会碰。
