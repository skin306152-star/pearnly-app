## 0. 一句话目标

新用户充值后,**连一次 MR.ERP(只填登录 + 选账套)→ 传发票 → 自动推成功**,中间不需要懂"工作空间/买方客户/种子模板/client_ids"这些概念,不需要逐个商品手动绑,不需要等几分钟,不需要在两个页面之间核对矛盾的状态。

**当前现实**:推送内核能推绿(实测 SI690301-677725 成功),但**一张 6 商品的发票要 130 秒**、会**建一堆重复商品**、**重试出重复日志行**、**推送日志说成功而异常栏说失败**、连接向导第一步就**卡死新用户**。离"开箱即用"还很远。

## 2. 根因诊断(为什么繁琐/慢/绕)

1. **把 MR.ERP 当成"完美镜像"在维护**:试图为每张发票的每个买方 + 每个商品行都在 MR.ERP 里建一条主数据并逐条复核。对泰式烘焙这种"每行是带规格的长描述"的业务,等于把发票行当商品建,产生海量一次性垃圾商品 + 每行都要建+复核 → 慢 + 脏。
2. **无 API 的硬约束**:MR.ERP 无开放 API,一切走 Playwright 浏览器步,每个表单动作都是秒级。建 6 个商品 = 6×(建+复核)= 分钟级。**这是物理下限,不靠"逐个建"能突破。**
3. **概念是按"内部数据模型"暴露给用户的**:workspace_client / client_ids / seed / 归属,全是工程概念,没翻译成会计能懂的"我的公司 / 买家 / 默认科目"。
4. **状态用"时间序列多行"表达,但 UI 没收敛**:推送/重试每次插一行,两个视图(日志、异常队列)各自聚合,口径不一 → 自相矛盾。

## 3. 开箱即用目标设计(核心)

### 3.1 🥇 核心决策:商品「匹配优先 + 通用兜底 · 不再自动建商品」(✅ Zihao 2026-05-26 已拍板)

**这是本方案最大的提速 + 简化 + 去错点,且兼顾"精准"。**

- **现状(弊)**:每个发票商品行 → 在 MR.ERP **现场新建一个商品** → 逐行复核。慢(逐行建几十秒)+ 垃圾(几百个一次性商品)+ 撞码 + 截断 + 不符。**今晚所有推送痛苦的根。**
- **改为(智能三段 · 永不自动建)**:
  1. **匹配**:行描述能对上 ERP 里**已有的真实商品** → 用那个真商品码(**精准**,有 SKU 报表价值的就精准记)。
  2. **通用兜底**:对不上(是定制长描述)→ 挂**一个"销售收入"通用商品**,真实描述放进行名称/备注(**快、无错**)。
  3. **砍掉"逐行自动建商品"**(慢+脏+错的根源),买方仍按税号自动建(买方是真实主体)。
- 推送一张发票 ≈ 1 次 upload(匹配是轻量搜索 + 通用商品只验一次)→ **秒级**;不建垃圾、不撞码、不截断失败,`商品不符` 异常基本消失。
- **为什么仍"精准"**:金额/买方/日期/税额/每行描述**两种做法都一字不差保留**;区别只在"每行要不要各自一个商品码"。定制描述类业务(烘焙)单品码本无意义;ERP 里真有的商品照样精准匹配。**精准其所当精准,通用其所该通用。**
- ERP 里真实商品越多 → 匹配命中越多 → 越精准;这给用户一个自然正反馈(想更精准就维护自己的商品库),而不是被迫手动绑。

> 会计合理性:对服务/多 SKU 中小商户,"销售收入"通用 + 明细在行描述是 ERP 常见且合规用法;要单品核算的,ERP 里建好真实商品即被自动匹配。**✅ Zihao 2026-05-26 已拍板采纳此方向。**

### 3.2 买方:税号自动建(保留 · 已能用)
- 买方有合法 13 位税号 → 自动建 MR.ERP 客户(`resolve_or_create_buyer_client` + sync 已现成)· 一张发票一个买方,成本可接受。
- 无税号/多页冲突 → 进异常待用户确认(现有 qa gate)。
- 复核:税号优先 + 名称兜底(已实现)+ 1 次重试(C5)。

### 3.3 卖方 = 工作空间 = 账套:连接即绑定,上传不选
- **概念锁定**:工作空间 = **你的公司(发票卖方/开票方)= MR.ERP 的一本账套**。
- **连接 MR.ERP 时**:把"这个 MR.ERP 账套 ↔ 哪个工作空间(卖方)"一次绑好(连接向导里选/或单工作空间时自动绑)。
- **上传时**:smart 模式按发票卖方税号自动归账套 → 自动路由到对应端点。**不弹工作空间选择**(右上角切换器仅作"查看过滤")。单账套用户**完全无感**。
- 多公司用户:卖方税号没绑过 → 进异常"待确认卖方账套"一次,绑定后记住(`seller_workspace_routes` 已现成)。

### 3.4 连接向导精简(砍掉 client_ids 那步)
新向导只剩**对会计有意义**的步:
1. **填 MR.ERP 登录账号 + 密码** → 测试连接。
2. **选年度账套**(TEST2019 等)。
3. **(可选/智能默认)选"销售收入用的通用商品"**(§3.1)+ 确认买方自动创建模板。默认给合理预选,能一键完成。
4. 完成 → 默认"识别后自动推送"开。

→ 删掉"用于哪些买方客户"整步(§6 client_ids 废逻辑)。新用户不再卡死。

### 3.5 推送反馈 + 重试 + 异常,三处状态单一真相
- **识别后立刻日志显示「推送中」**(已两条路径都做),推完原地变 ✓/✗。
- **重试不再新插行**:改为**更新原行**(status/attempt/时间),或日志列表/异常队列**按(发票×端点)折叠只显最新一条**。彻底消除"重复行" + "两处打架"(A3/A4)。
  - 推荐:① 重试 UPDATE 原行(保留 attempt 计数);② 异常队列与日志同一派生口径(都取"该发票×端点最新一条",成功即从异常消失)。
- **异常队列**:只留"真要用户决策"的(买方多页冲突/卖方账套未绑/通用商品没配)。`商品不符`在通用商品模式下基本不出现。

## 4. 实施要点(技术)

### 4.1 商品「匹配优先 + 通用兜底 · 不自动建」(§3.1)落地
- endpoint.config 增 `generic_product_code`(通用销售商品码 · 连接时选/默认预选一个 ERP 已有收入类商品)。
- 商品解析改为三段(替换现 L4 auto-create 默认路径):
  1. L1/L2/L3 匹配(DB 映射 / 名称精确 / 模糊)→ 命中真实商品码(轻量 `_search_listing`,不翻全表)。
  2. 未命中 → `generic_product_code`(行名/备注 = OCR item.name,超长截断 + WARN)。
  3. **不再调 `_layer4_auto_create` 建商品**(默认)。
- `_verify_resolved_master_data`:匹配命中的按码复核(已有截断容忍);通用码只验一次存在。
- `generic_product_code` 未配 → 明确提示去连接里选(不静默)。
- (可选)保留 auto-create 代码但默认不走;真要"每行建商品"的极少数用户单独开关。
- 守门:契约测试(命中行用真码 + 未命中行用通用码 + 不触发 auto-create + N 行只验少数几个码)。

### 4.2 重试 UPDATE 原行(§3.5)
- `erp_routes.py` `/api/erp/logs/{id}/retry` + `/batch-retry`:重试结果 `update_log_status_after_retry(原 log_id, ...)`(已有该 DAL),`attempt += 1`,**不 INSERT**。
- 异常队列 `list_push_exceptions` 与日志 `list_push_logs` 统一"按(history×endpoint)取最新一条"口径。
- 守门:重试后同(发票×端点)只剩一条 + 成功后异常队列为空。

### 4.3 连接向导精简 + client_ids 退役(§3.4 + §6)
- 前端 `erp-mrerp-connect.js`:删 step1 客户 picker;步数 3→ 重排。
- 后端 `erp_routes.py`:`client_ids` 不再必填(删 138-139 / 255-256 校验)、新建时默认 `[]`;`config` 保留字段不报错(兼容老数据)。
- 改/删守门测试 `EndpointClientIdsRequiredTests`(4 条 · 反过来锁"允许空")。

### 4.4 概念 + 导航(§5)
- 工作空间弹窗:名称 `title` + ellipsis;加"管理工作空间"(改名/绑账套);文案"选择你的公司(发票卖方)"。
- "客户"导航 → 见 §5.3 决策。
