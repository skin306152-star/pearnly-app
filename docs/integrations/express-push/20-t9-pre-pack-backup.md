# T9 施工单 · PACK 前自动备份(保真客户账套·安全网)

> 来源:Owner 2026-06-22。PM 查实——写发票前有自动备份(`dbf_writer._backup`),但**自动 PACK 前没有备份**。PACK 是动整个账套的较重操作,万一中途被打断(断电/崩溃)老 FoxPro 文件可能不一致 → 接真客户前补一道 PACK 前备份。
> 施工窗口:companion `D:\pearnly-companion`。**与 T6/T7/图标同一个 companion 窗口顺手做**(一次重打、一次重验)。

---

## 1. 要解决什么

`pack_scheduler` 夜间触发 → `pack_runner` 驱动 Express 跑 PACK,**跑的是客户选定账套的全套文件重整**。当前**无 PACK 前备份**(测试时是我们手动 robocopy 备份的脚手架,不在正式流程)。风险:PACK 中途异常 → 账套文件不一致、无快照可还原。这是接真客户前唯一未堵的"对真实套账的影响"。

## 2. 改法

### 2.1 PACK 前必备份(无备份不 PACK)
- `pack_scheduler.run_pack` 在启动 `pack_runner` **之前**:把**选定账套整目录**(`config.account_dir`)备份到本地时间戳目录(可复用 `dbf_writer._backup` 的 copytree 思路·或 robocopy 等价)。
- **fail-safe 铁律**:**备份不成功 → 本次不 PACK**(跳过·下个窗口重试)。宁可发票晚一晚显示,也绝不在没有快照的情况下动客户整套账。

### 2.2 磁盘空间守卫
- 备份前查可用空间 ≥ 账套体积(留余量)。**空间不足 → 跳过 PACK + 记日志/托盘提示**(不硬塞把盘撑爆)。

### 2.3 保留与清理(夜夜全套备份会堆满盘)
- 备份成功 PACK 完成后:**保留最近 K 份**(默认 2-3·可配),清理更早的。
- **PACK 失败**(runner exit 非 0/2):**保留该次备份不清**,记日志 + 托盘告警(留作还原)。

### 2.4 还原(v1 稳妥)
- v1 = 出事时**备份在、可还原**:PACK 失败 → 保留快照 + 明确日志/托盘提示,**人工/指引还原**(自动覆盖还原有风险·暂不自动)。
- 备份位置/保留份数进 config(方法无关·并入 ExpressSettings seam,守可扩展性契约)。

## 3. 边界 / 红线
- 只加"PACK 前备份 + 守卫 + 保留",**不碰 PACK 本身、不碰写盘、不碰三重闸**。
- 备份是**只读复制**客户账套 → 不改其任何数据。
- 账套白名单/选定账套仍是写盘与 PACK 的唯一目标(T2)·备份只备那一个。

## 4. 测试 / 验收
- PACK 前确有整套备份生成(时间戳目录·文件数==源)。
- 备份失败(模拟权限/路径错)→ **PACK 被跳过**(fail-safe·不裸跑 PACK)。
- 磁盘空间不足(模拟)→ PACK 跳过 + 告警。
- 保留:连跑多次 → 只留最近 K 份、旧的清掉;PACK 失败那次的备份**不被清**。
- 既有 PACK 链(占用检测/账套硬闸/还原)不退化。
- DATAT 真机:配置开 PACK → 跑前看到备份生成 → PACK 完成 → 备份按保留策略留存;写前/PACK 前两类备份不打架。
