# 平台总纲 · 02 生态选择

> 骨架#2/#7:事务所=渠道放大器;生态分层、应用市场后置。关键决策:**渠道机制学 Intuit/Xero 的"会计师即渠道",不学金蝶的保证金代理商制** —— 金蝶 SSP 模式(保证金+交付人员数门槛)是 ERP 项目制基因+中国市场密度的产物,泰国体量小一个数量级,前提不成立。

## 双边飞轮(Pearnly 生态的核心结构)

```
事务所(F 层)用 Pearnly 做账 ──给客户发采集端──→ 商户(S 层)拍票/POS/LINE
        ↑                                            │
        └────── 商户长大要自己管账,升 M 层 ←──────────┘
                事务所继续做它的税务顾问(留在环里)
```

- 金蝶依据:账无忧 4.2 万代账公司 → 间接覆盖 560 万中小企业(官方口径)。事务所**既是付费用户又是获客渠道**,1 家所 ≈ 几十上百家小微,是财税 SaaS 触达小微的最高杠杆。
- 我们的对应优势:商户端采集(拍票/LINE/POS)本来就是给非会计的人用的 —— 事务所发给客户的"采集器"我们已经有,缺的是 F 层工作台(01 章)把两边接起来。
- 与 Paypers 的差异在此闭环:Paypers 止于"整理好交给会计",我们让"老板的采集"直接落进"事务所的做账"。

## 开放能力三步(应用市场是第 1500 步,不画饼)

**第 0 步 · 出向(已有,🟢 资产)**:erp_push 六适配器(FlowAccount/Peak/QuickBooks/Xero/MR.ERP/通用 webhook,services/erp/erp_push.py,带三次重试+日志)。"不锁死客户"(2.0 原则#5)已经成立,继续维护。

**第 1 步 · 入向 API(🔴 P1 范围)**:
- API key 体系(租户级签发/吊销,挂 `docs/permissions/` 的权限点体系,key 绑权限子集)+ 出向 webhook 订阅(单据事件)。
- 首批场景找真需求:电商订单落账(Shopee/Lazada/TikTok,竞品调研确认是泰国小店刚需)、银行流水 feed、事务所自有工具对接。
- 对标:PEAK 把 API 放 PRO Plus(฿12,000/年)档 —— API 在泰国市场是**高档付费能力**,进 04 章付费墙。

**第 2 步 · 应用市场(远期,有前提)**:金蝶 1500+ 应用建立在 3000 伙伴+万级客户之上;客户数没到量级前不做市场、不做低代码。写在这里是为了**防止提前做**。

## LINE 的生态位(泰国特有,金蝶没有的牌)

LINE 是泰国的"微信"。现状已是采集入口(OCR/一句话记账/@pearnly 官方号);生态意义上它是我们的**零安装客户端**:事务所给客户发的"采集器"可以就是一个 LINE 加好友动作。伙伴计划的获客动线优先走 LINE,不强求装 App。红线不变:LINE 主路径一字不动,扩展只加分流。
