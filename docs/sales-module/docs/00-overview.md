# 00 · 概览:统一智能录入引擎 + 销项第一出口

> **🧭 先分清「销项 vs 进项」(防绕晕 · 2026-06-06 Zihao 拍板点明)**
> - **进项** = 记录你**花出去的钱**(交水费、买东西、打车)。客户给的 **Paypers LINE 截图**「`ค่าน้ำ 50`(水费50)→ 自动记一笔支出 + 自动归类 + 出凭证」**就是进项**——属本引擎的「**进项方向 + LINE 一句话入口**」,spec 见 `docs/14-line-quick-entry-spec.md`。
> - **销项** = 你**开发票给客户、收客户的钱**(`ใบกำกับภาษี` / `ใบเสร็จ`)。
> - **本项目 = 这一套引擎(销项 + 进项共用同一骨架)。落地顺序:先做「销项开票」=第一出口;Paypers 那个「进项 · LINE 一句话记账」是同引擎的进项入口、靠后做。**
> - ⚠️ 别把「下一步=销项」误当成 Paypers 那个 LINE 记账:那是**进项**,同引擎、不同方向、排在销项开票之后。

## 一句话愿景

把 Pearnly 从"只能识别进项票 + 对账"扩成一个**统一的「智能单据录入」能力**:
一套引擎、两个方向(销项开票 / 进项记账)、多种轻量入口、全程 AI 加持,**比 Paypers 更智能**。
销项发票 / POS 开票是这套引擎落地的第一个出口。

## 为什么不是"只做一张销售发票页"

如果只单做销项页,以后接"进项轻量录入"会返工——两者骨架几乎一样:
> 选对方 → 加几行明细 → 自动算 VAT/WHT → 出 PDF → 发 LINE/邮件。
所以抬一层,先定义引擎的三段(入口 → AI 处理层 → 出口),销项作为第一个出口落地。

## 引擎三段

```
[入口 · 多模态]            [AI 处理层 · 我们更智能的地方]        [出口 · 单据]
手填表单                   ├ Gemini 提取(图/语音/一句话)        销项:ใบกำกับภาษี / ใบเสร็จ
拍照 OCR        ───────►   ├ RD 直连查税号真伪 + 带 17 字段  ───► 进项:费用记录(已有 OCR 复用)
扫条码 / 二维码            ├ 知识库规则引擎校错(算术/完整/查重)    共用:PDF 生成 + LINE/邮件 + 推 ERP
LINE 一句话                ├ AI 推荐分类 + 借贷科目分录
菜单点选(POS)            └ 银行流水自动对账核销
```

## 分阶段(Phase)· 客户已答锁定(2026-06-05 · 见 `docs/09`)

- **Phase 1 · 合规开票闭环**:商品主数据(系统内建 + Excel 导入 + **带图**)→ 选客户 →
  **菜单式图卡点选** / 手输码加行 → 自动算 7% VAT → 生成合规 ใบกำกับภาษี/ใบเสร็จ PDF
  (**连号锁 + 开出不可改 + 红冲/补开 ใบลดหนี้/ใบเพิ่มหนี้**)→ 发送(**邮件 / LINE / 打印纸**全做)→ 汇入 ภ.พ.30。
- **Phase 2 · 库存 + LINE 开单 + 应收**:库存系统(คลังสินค้า · `inventory_*` 扣减/查询)·
  LINE bot 开单(复用 line_webhook)· 应收追踪(账龄 + 催收,导航已留位)。
- **Phase 3 · e-Tax 直报**:对接 RD e-Tax Invoice & e-Receipt 直报税局(需电子证书 ใบรับรองอิเล็กทรอนิกส์ · "可行才做")。
- **下一个独立项目 · POS**:收银 / 扫条码 / 扫 QR / 现场卖货(**不在本模块** · 客户明确要拆成下一个项目)。

> 用户两类都要支持:**会计事务所代客开票** + **客户公司自开**(复用多租户/clients,加角色区分)。

## 与既有 PRD 的边界(关键 · 防重造/脱轨)

| 既有 PRD | 覆盖 | 本沙盒 |

intake(LINE/OCR)/PDF/对账底座,不另起炉灶。迁回前与这两份 PRD 属主对一次口径。
