# POS 项目 · 00 概念与边界

> 这是 POS+库存项目的北极星。任何窗口动 POS 前先读这页。
> 状态:图纸阶段(封锁整顿期 · 纯文档/设计 · 未开建生产代码 · 待 Zihao 破例授权才施工)。

## 一句话概念

**Pearnly POS = 一个能离线用的网页收银台:收银员在柜台卖货、收钱、出小票,而每一笔卖出的货自动变成会计端的销售账、自动扣库存、需要时一键升级成正式税票(ใบกำกับภาษี)。**

灵魂 = **收银即记账**。市面上 POS 与记账是两套系统、店员卖货会计月底再录一遍;Pearnly 因为底座(商品/账套/税票/对账)已具备,柜台一刷,账和税同时长出来。

## 这是什么、不是什么

- **是**:Pearnly 平台的一个模块(按租户开关),不是独立产品;复用现有商品主数据、账套主体、税票合规、PromptPay、热敏小票、对账。
- **不是**:又一个孤立收银机。不靠"功能比别人全"赢,靠"卖货即自动记账报税 + 泰国本地 + 离线可用"。

## 三层归位(对齐 `docs/PRODUCT_VISION_MODULAR.md`)

| 层 | POS 相关 |
|---|---|
| ① 底座(共享) | 商品主数据 · 账套主体 · 账号/权限 · 多语言 · AI 录入 |
| ② 模块(按租户开关) | **库存** · **POS 收银** · 开票(销项) · 进项 · 对账 · 知识库 |
| ③ 出口 | 小票(热敏) · 税票(ใบกำกับภาษี) · 销售报表 · 报税(ภ.พ.30) |

## 范围边界(防跑偏)

**本项目做**:库存模块(后台) + POS 收银前台 + 离线能力 + 收银员角色 + 模块开关机制(tenant_modules)。

**本项目不做 / 留后续**:硬件驱动深度集成(只走浏览器标准:扫码枪=键盘输入、小票机=浏览器打印/ESC-POS 后续)、多门店连锁总部合并报表(Phase 2)、电商/线上订单打通(后续)。

## 与现有模块的咬合(详见 02-architecture)

- **商品**:复用 `products`(已含 barcode/qr/unit/图片),禁止重建。
- **卖方**:复用账套主体 `workspace_clients`,POS 单据指定以哪个主体开出。
- **小票/税票**:复用 `sales_documents`(doc_type=receipt / tax_invoice_receipt)+ 连号 + 冻结快照 + 热敏 PDF。
- **收款**:复用 `promptpay.py`(扫码)+ 现有 payment_status/method/amount 字段。
- **库存**:新建(零现存),被 POS 出库扣减、被进项入库喂,事件松耦合(关了不崩)。

## 业态策略(通用兼容 · 用户自选 · Zihao 2026-06-07 定调)

一个可配置内核 + 业态预设开关。用户开通时选业态(零售/餐厅/服务),系统一键开好对应功能块;其余隐藏不出现。详见 01-requirements。

## 阶段顺序(Zihao 2026-06-07 定调)

**库存模块先做 → POS 收银后做。** POS 的实时扣库存接在库存模块之上。两个模块一起设计(图纸),分先后施工。
