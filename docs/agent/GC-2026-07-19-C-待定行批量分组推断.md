## 0. 背景（已勘察实锤，不用再查）

- 「让 AI 预判待定行」现状：`routes/workorder_bank_sales_routes.py:49-58` 的 `POST /api/workorder/orders/{id}/bank-sales/run`，在 `db.get_cursor(commit=True)` 的**单 HTTP 请求单事务**里调 `bank_sales_brain.run`；`services/workorder/steps/bank_sales_brain.py:176-211` `_run` 对 `engine.pending_rows(events)` **逐行串行** `ask_model`（每行一次模型调用，60s 超时）。705 行=705 次串行调用压在一个请求里，不可用。
- 大脑单行契约：`build_question`（brain.py:74-82，`{row_fingerprint,date,deposit,description}`）→ `build_prompt`（:85-90）→ `transport.text_to_json`（services/ai_gateway/transport.py:79，单 prompt 无批量参数）→ `parse_suggestion`（:93-135，校验 `suggestion∈{sales,non_sales,cannot_judge}`、`confidence∈[0,1]`、`reason_zh`、`cited_row_fingerprints⊆题面`）→ 逐行落 `bank_sales_suggested` 事件，`dedupe_key=f"{engine.SUGGEST_DEDUPE_PREFIX}:{fp}"`。
- 读侧裁决优先级 `_effective`（bank_sales_suggest.py:429）：人裁 > 大脑 > 规则；`cannot_judge` 仍算 pending。суggested 过的行不再出现在 `pending_rows`，**天然断点**：重触发只补没判过的行。
- 行指纹 = `f"{date}|{signed}|{seq}"`（bank_sales_suggest.py:145）。人工确认端点 `POST .../bank-sales/decide`（routes/workorder_bank_sales_routes.py:61-80）单行 `{fingerprint, verdict}` → `bank_sales_review.record_bank_sales_decision`（services/workorder/bank_sales_review.py:22-45，落 `human_decision` 事件，写前 `_row_exists` 防野指纹）。
- 前端：编排 `static/ai/ai-intake-bank-sales.js`（`run()` :65-85、`decideRow` :43、`apply` :90）；渲染 `static/ai/ai-bank-sales-render.js`（`canApply()` :80-86 = `applicable && reliable && pending_count===0`，已有 `groupRows` 三区清单）；API 客户端 `static/ai/ai-api.js:193-205`。
- 并发/断点先例：`services/workorder/steps/classify.py:229-253` `_ocr_in_order`（ThreadPoolExecutor+原序回收）；prod 为 systemd 单进程 uvicorn（内存级运行标志可用）。
- 成本归因：单行流用 `task="taxops.verdict"`（brain.py `_default_ask`）。**保持不变**，别改 task 字符串（成本看板依赖）。

## 1. 目标

705 行待定 → ① 确定性规则先消化一批；② 剩余按批（40 行/次）送大脑，后台跑、可断点续、有真实进度；③ 人按组批量确认，不点 705 次；④ 全程不重读银行页、不碰金标。

## 2. 改动点

### 2.1 规则层扩充（`services/workorder/steps/bank_sales_suggest.py`）

`classify_row`（:287-302）现状：zero/withdrawal/fee/cancel → non_sales；`(date,deposit)∈strong_sales_keys` → sales；其余入账 → pending。

新增**保守**关键词规则（常量放文件顶部，全大写，泰文原文注释一句 why）：

1. description 含 `รับเงินจากการขายด้วย Thai QR` 或含 `K SHOP/MYQR`、`EDC` 渠道字样 → **sales**（票面自己写着"销售收款"，确定性判据）。
2. description 含 `โอนไป`、`ถอนเงิน` 已是取出行（withdrawal>0），维持 non_sales，不动。
3. description 含撤销/退回类关键词（`ยกเลิก`、`คืนเงิน`、`Reversal`）→ **non_sales**。
4. 其余入账行（含普通 `รับโอนเงิน` 个人转入）**保持 pending**——不许把"个人转账=销售"写成规则（老板转入/借款同形，误判即虚增销项税）。

新增 `group_key(row) -> str` 纯函数（同文件）：按 description/渠道归组，返回组标识：`qr_edc`（刷卡/QR 结算）、`transfer_in`（个人转账入账）、`cash_deposit`（现金存入，关键词 `ฝากเงินสด`/`CDM`）、`other_in`（其他入账）。供 suggest() 输出与前端分组共用（单一事实源，别在 js 里再写一份关键词）。

`suggest()` 输出增加每个 pending 行的 `group` 字段与按组汇总 `pending_groups: [{key, count, sum}]`（:358 起的输出契约处）。

### 2.2 大脑批量化（`services/workorder/steps/bank_sales_brain.py`）

1. 新增 `BATCH_SIZE = 40`（模块常量）。
2. 新增 `build_batch_question(rows) -> dict`：`{"rows": [build_question(r) 的形状 × ≤40]}`；新增批量 prompt 模板（沿用单行模板的判据文案，改为"对下列每一行独立判断，返回 JSON 数组，每个元素含 row_fingerprint/suggestion/confidence/reason_zh；不确定就 cannot_judge，禁止编造题面外指纹"）。
3. 新增 `parse_batch_suggestions(data, allowed_fps) -> list[dict]`：复用 `parse_suggestion` 的单元素校验逐个过（列表不是 dict 时按全无效处理）；响应里缺失的行视同 cannot_judge**但不落事件**（留在 pending 等下批/人工），题面外指纹丢弃。
4. `_run` 改造（:176-211）：
   - `rows = engine.pending_rows(events)` 后按 `BATCH_SIZE` 切片；每批一次 `ask_model`（`timeout_s=120`、`max_tokens` 调到够 40 行响应、`temperature=0.0`、task 不变、`trace_id=f"bank_sales_brain:batch:{work_order_id}:{批序}"`）。
   - 每批的合法建议在**该批自己的短事务**里逐行落 `bank_sales_suggested` 事件（dedupe 键不变），批间提交——中途死掉已判的批不丢。
   - 硬上限：单次触发最多 `MAX_BATCHES = 30` 批（1200 行），到顶即停并在进度里注明，防失控烧钱。
   - 单批模型异常：记 log、该批行留 pending、继续下一批；连续 3 批失败即终止本次运行（进度标 failed）。
5. 新增模块级进度注册表（单进程 uvicorn，内存即可）：`_PROGRESS: dict[str, dict]` + 锁，形如 `{"running": bool, "total": n, "done": n, "failed_batches": n, "started_at": iso, "finished_at": iso|None}`；提供 `progress(work_order_id) -> dict|None` 与运行互斥（已 running 再触发 → 拒绝）。进程重启注册表清空=允许重跑，幂等安全（dedupe 兜底），docstring 写明这一句。

### 2.3 路由后台化（`routes/workorder_bank_sales_routes.py`）

1. `POST .../bank-sales/run`（:49-58）改为：鉴权与双闸判定**不变**（`_authorize_bank_sales` :40-46）；改为起守护线程跑 `bank_sales_brain.run_async(work_order_id, tenant_id, ...)`（线程内部自取 `db.get_cursor`，不再占请求事务），立即返回 `{"started": true, "total_pending": N}`；已在跑 → HTTP 409 `{"running": true}`。
2. 新增 `GET .../bank-sales/progress` → `bank_sales_brain.progress()` 结果 + 当前 `pending_count`（重放事件现算）。无进度且无运行 → `{"running": false}`。
3. 新增 `POST .../bank-sales/decide-batch`：请求 `{"decisions": [{"fingerprint": str, "verdict": "sales"|"non_sales"}] }`，上限 800 条；鉴权同 decide；循环调 `record_bank_sales_decision`（同一事务），任一野指纹 → 整批 400 报出指纹值。响应 `{"applied": n}`。
4. 三个端点都要契约测试（见 §4）。新路由文件已存在，只是加函数——`lint-routes` 契约要求照 `tests/unit/test_workorder_bank_sales_routes_contract.py` 现有样式补。

### 2.4 前端（`static/ai/ai-intake-bank-sales.js`、`static/ai/ai-bank-sales-render.js`、`static/ai/ai-api.js`）

1. `ai-api.js`：新增 `bankSalesProgress(orderId)`、`decideBankSalesBatch(orderId, decisions)`（照 :193-205 现有两个的写法）。
2. `ai-intake-bank-sales.js` `run()`（:65-85）：改为触发后进入轮询（每 3s 调 progress），`running=false` 或连续 2 次 `done` 不变即停止轮询并 `refreshOrder()`；运行中按钮置 disabled 并显示「已判 X / 共 Y」。409 → 提示"正在后台预判，别重复点"。
3. `ai-bank-sales-render.js`：
   - 待定区按 `row.group`（后端给的组键）分组渲染：组头 = 组名 + 笔数 + 合计；组内金额 ≥ ฿10,000 的行单列（必须逐行看），其余折叠为"共 n 笔小额"；每组两个按钮「本组全部=销售」「本组全部=非销售」→ `decideBankSalesBatch`（弹站内确认弹窗，蓝按钮，写明组名/笔数/合计——全站弹窗硬闸）。
   - `pending_count===0` 时「采用建议值」逻辑不变（canApply :80-86 不动）。
   - 倒推数为 0 时不许再渲染「差异 100.0%·超阈值」徽章：`applicable && reliable` 但 sales 合计为 0 → 显示「尚未形成倒推数」中性态（找到现有徽章渲染分支改判定，别加新样式类）。
   - 文案走 static/ai 现有 i18n 惯例，语种齐全，禁硬编码单语。
4. 若这些 js 属于打包产物链（查 `scripts/check_asset_bundling.py` 的口径），按总则跑 build 并带产物；若 nginx 直 serve 则 bump 引用页 `?v=`。
