## 4. 今天发现的问题

### P0-1：AI 与收银台共用登录令牌，跨标签页会把权限串掉

现象：

- 手工销项表里输入 `858780.16` 和 `60114.61`，前端却说数字无效。
- 生产日志在 19:02:31、19:02:45 两次记录 `/sales-summary` 返回 403。
- 授权拒绝项为 `tax.filing.create`，原因 `entrance_scope`。
- 当时同一个 Chrome 同时开着 Beauty Store Cashier 和 Pearnly AI。

根因：

- `/pos` 登录与 `/ai` 登录都使用 `localStorage['mrpilot_token']`。
- 收银台登录会覆盖 AI 令牌。
- 已打开的 AI 页面不会发现令牌入口已变化；提交时读取到收银台令牌，后端按入口权限正确拒绝。
- 前端把所有 API 异常统一映射成“请填有效数字”，因此把权限问题伪装成输入问题。

这不是账号“没有会计权限”，也不是数字格式错误，而是登录隔离与错误映射缺陷。

明天必须一起修：

1. `/ai`、`/cashier` 使用入口隔离的令牌或入口感知会话，不能互相覆盖。
2. 页面发现令牌入口不匹配时立即失效并引导重新登录。
3. 无权限时隐藏或禁用人工申报动作。
4. 401/403、校验失败、网络失败分别显示真实原因。
5. 用同一 Chrome 同时登录 AI 和 Cashier 做真浏览器回归。

临时绕行仅用于诊断：重新登录 AI 后，不在同一浏览器配置中登录 Cashier。它不能替代正式修复。

### P0-2：银行倒推没有形成可用建议，705 笔待定不能交给人逐笔点

现象：

- 18 份银行资料成功读出 734 笔，但只有 29 笔确定为非销售，705 笔待定。
- 页面出现“让 AI 预判待定行”，同时提示还有 705 行待确认。
- 当前实现按行串行调用模型，单行可等待到 60 秒；在真实月度数据上不可用。
- 用户无法知道会跑多久、处理到哪一笔、依据是什么，也不可能手工判 705 次。

裁决：

- “银行资料能读”不等于“已经能倒推出销项”。
- 银行流水可以作为销售完整性的佐证，但不能无依据替代 POS 月结或已申报销项权威数。
- 当前系统需要诚实停在缺销项，不能为了出包自动填金标值。

明天需要把逐行模型改成批量、后台、可恢复的推断任务：

1. 先用确定性规则归并同日同渠道、转账、现金存款、调拨、退款和非营业入账。
2. 只把剩余小集合批量送模型，并返回分类依据、置信度和可追溯行号。
3. 展示真实总数、已处理、待处理、失败数和最后活动时间。
4. 支持断点续跑，不重新 OCR 已成功的 18 份银行资料。
5. 低置信结果进入分组确认，不允许 705 个单行按钮。
6. 无法达到可信阈值时明确要求 POS Z 报、POS 月报或已申报 P.P.30 作为凭据，不生成建议值。

### P1-4：折扣采购票可能被错误判成数字不自洽

样本：

- 票号 `57016198`。
- 票面商品合计 ฿12,810.00，折扣 ฿3,162.00，折后未税 ฿9,016.82，VAT ฿631.18，含税 ฿9,648.00。
- 页面把它标成“勾稽红灯·数字不自洽”。

可能根因：

- 校验器混用了折扣前商品合计、折后未税金额和含税金额。

待办：

- 按“折前金额 - 折扣 = 含税或未税基数”的票面语义分别验算。
- 增加该真实票的回归 fixture，不能仅靠 OCR 字段相加。

## 5. “人工销项汇总”应该从哪里来

真实会计不会凭空知道销项额。产品必须给出来源层级，而不是只展示三个空输入框：

1. 首选：POS Z 报或月度销售汇总导出。
2. 次选：已申报 P.P.30 第 1 行销售额、第 5 行销项税，用作申报凭据和复核来源。
3. 辅助：已开销售发票汇总，只能覆盖开票部分；本轮约 58 张、去重后 57 张，金额远低于整月官方销售，不能代表全部营业收入。
4. 佐证：银行入账、PromptPay、EDC、现金存款等，用来检查漏报和时间差。
5. 上述都没有时：系统只能标记“缺权威销项来源”，不能让会计猜数，也不能静默生成交付包。

人工录入必须同时保存：

- 来源类型
- 来源文件或申报表附件
- 账期
- 销售额与销项税
- 填写人、时间和备注
- 后续与银行、发票汇总的差异说明
